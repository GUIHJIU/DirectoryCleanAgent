# C5-模拟运行与报告导出

> 阶段性关键决策与细节控制文档 | 2026-06-24

---

## 一、 关键架构/设计决策记录

### 决策 1：流式两遍处理 vs. 单遍全量缓存

**最终决策**：采用流式两遍处理。
- 第一遍（`SimulateAsync`）：流式遍历 B1→B2→B3 管道，仅产出聚合统计 + 每组 ≤10 条样本路径。
- 第二遍（`StreamAnalysisAsync`）：仅在用户请求导出时触发，重新流式遍历管道，逐条 yield return 给导出器。

**核心权衡（Trade-off）**：导出时需要重新遍历整个 B1→B2→B3 管道（对于 200 万文件的磁盘约 10-30 秒额外延迟），但换取了：
- 第一遍内存峰值 < 50MB（无论文件总量多大）
- 第一遍速度极快（只做分组统计，不保留全量数据）

**被否决的替代方案**：
1. **全量缓存方案**：第一遍将所有 `SimulationFileEntry` 存入内存 `List<T>`，导出时直接使用。**致命缺陷**：200 万文件场景下内存占用 > 500MB，违反项目 < 200MB 内存硬约束。
2. **单遍聚合+导出共用**：第一遍同时做聚合和写文件。**致命缺陷**：用户在导出前可能需要先看模拟结果再决定是否导出，单遍方案丧失了交互灵活性。

### 决策 2：SimulationService 实现位置 — 主 App 项目 vs. 独立项目

**最终决策**：将 `SimulationService` 和 `ReportExporter` 放在主 `DirectoryCleanAgent/Services/` 下，接口和 DTO 放在 `DirectoryCleanAgent.Core`。

**核心权衡（Trade-off）**：放弃了独立项目带来的模块边界清晰性，换取了：
- 不增加 .csproj 数量和解决方案复杂度
- 遵循现有 C1/C2/C4 模块的放置模式（UI 层协调服务放在主项目）
- 2 天工期不需要额外的项目结构开销

**被否决的替代方案**：
1. **新建 `DirectoryCleanAgent.Simulation` 项目**：遵循 B 模块模式（每个模块独立项目）。**致命缺陷**：2 天工期下增加项目创建/引用配置/CI 调整的边际成本过高，且 C 阶段模块天然是协调层而非独立能力层。
2. **全部放在 Core 项目**：接口、DTO、实现全部在 Core。**致命缺陷**：Core 项目不应依赖 Microsoft.Extensions.Logging 的具体实现细节（虽然当前已依赖），且实现类需要访问 WPF 相关类型（如 `SaveFileDialog` 在导出流程中可能涉及）。

### 决策 3：跳过 SHA-256 哈希 vs. 完整快照生成

**最终决策**：模拟运行调用 `IDecisionEngine.Arbitrate()`（单文件快速仲裁）而非 `DecideAndSnapshotAsync()`（批量哈希+快照生成）。

**核心权衡（Trade-off）**：模拟运行的分析结果可能略低于真实删除时快照的精度（缺少哈希用于事后验证），但换取了：
- 模拟速度提升 5-10 倍（SHA-256 是管道中最耗时的操作）
- 模拟运行不产生墓碑、不写审计日志，语义上更纯粹

**被否决的替代方案**：
1. **完整快照方案**：模拟运行也调用 `DecideAndSnapshotAsync` 生成完整快照。**致命缺陷**：模拟运行本质是"预览"，生成 SHA-256 和不可变快照完全浪费 CPU，且 200 万文件场景下哈希计算耗时数分钟，用户体验不可接受。

## 二、 隐含前提与关键假设清单（风险预警）

1. **[假设 1]**：Everything SDK 在目标机器上已正确安装且版本 ≥ 1.4.1.1000。若不成立，`IFileListProvider.EnumerateFilesAsync` 无法提供文件流，仿真运行将直接失败。
2. **[假设 2]**：模拟运行期间用户配置（`UserConfig`）不会由其他线程并发修改。`SimulationService` 在方法入口读取一次 `IConfigService.Current` 的快照，贯穿整个流式处理，不做中途刷新。
3. **[假设 3]**：`IRuleEngine.Evaluate()` 和 `IDecisionEngine.Arbitrate()` 是线程安全且无副作用的。当前它们在单线程的 `await foreach` 中串行调用，若将来改为并发处理需要重新评估。
4. **[假设 4]**：HTML 报告的嵌入式 CSS 在主流浏览器（Edge ≥ 90, Chrome ≥ 90, Firefox ≥ 90）上渲染结果一致。未测试 IE 或旧版浏览器。
5. **[假设 5]**：CSV 导出的目标消费者是 Excel/WPS，这些应用能正确识别 UTF-8 BOM 并用正确的编码打开文件。若用户使用纯文本编辑器（如 Notepad），BOM 可能导致首行出现 "\uFEFF" 标记。

## 三、 边界与范围界定（明确"不做什么"）

1. **[明确不做]**：C5 不做 PDF 报告导出。设计文档仅要求 HTML/CSV，PDF 导出若需加入属于范围蔓延。
2. **[明确不做]**：C5 不做模拟运行与真实删除的差异对比报告。模拟运行结果和实际删除结果可能因文件系统变化而不同（例如文件在模拟后被其他进程修改），此差异对比属于高级功能，不在本阶段范围。
3. **[明确不做]**：C5 不做定时自动模拟运行。自动化调度属于 C8/C9 阶段的功能。
4. **[明确不做]**：C5 不做增量模拟运行（仅分析新文件/变更文件）。每次模拟运行都是全量重新遍历管道。
5. **[明确不做]**：HTML 报告不做 JavaScript 交互（如排序、分页、搜索）。报告为静态 HTML 文件。

## 四、 待确认事项（需要人工介入的模糊点）

1. **[待确认项 A]**：`EverythingQueryParams.Volumes` 默认值。当前实现中若用户未配置 `IncludedVolumes`，默认扫描 `["C:"]`。**待确认**：是否应扫描所有可用 NTFS 卷，还是仅 C 盘？
2. **[待确认项 B]**：`SimulationProgress` 的进度报告节流间隔。当前设定为每 100 个文件报告一次进度（`ProgressReportInterval = 100`）。**待确认**：此间隔是否合适？太小可能 UI 卡顿，太大可能进度条跳跃感明显。
3. **[待确认项 C]**：`SimulationGroup.SampleFiles` 的最大样本数。当前硬编码为 10 条/组。**待确认**：是否需要将此值暴露为用户可配置项（如设置页面或导出选项对话框）？
4. **[待确认项 D]**：HTML 报告的颜色方案和排版。当前使用内嵌 CSS 的固定配色（绿/蓝/橙/灰），**待确认**：是否需要与应用的深色/浅色主题联动？
5. **[待确认项 E]**：非管理员只读模式下，"导出报告"按钮逻辑已确认保持可用。但"模拟运行"按钮在只读模式下也应保持可用（当前 `CanSimulate` 已实现无管理员检查）。**待确认**：是否有其他 UI 状态要求（如只读模式下隐藏删除相关 UI 元素）？

## 五、 关键外部依赖锁定

*   **依赖项**：`IDecisionEngine.Arbitrate()` — 来自 `DirectoryCleanAgent.Decision` 模块（B3 阶段）。若 B3 的 `Arbitrate` 方法签名或决策表逻辑变更，`SimulationService.ProcessSingleFile()` 需要同步更新。
*   **依赖项**：`IFileListProvider.EnumerateFilesAsync()` — 来自 `DirectoryCleanAgent.Everything` 模块（B1 阶段）。当前 DI 容器中注册为 `MockFileListProvider`（返回约 500 条 Mock 数据），替换为 `EverythingFileListProvider` 后 C5 自动获得真实管道能力。**若 Everything SDK IPC 通信不稳定，模拟运行将直接失败**。
*   **依赖项**：`IRuleEngine.Evaluate()` — 来自 `DirectoryCleanAgent.Rules` 模块（B2 阶段）。规则热加载（`RulesChanged` 事件）在模拟运行期间可能触发，但当前实现中模拟运行使用入口时读取的 `UserConfig` 快照，不受热加载影响。
*   **依赖项**：`IOperationExecutor.QueryRecycleBinCapacityAsync()` — 来自 `DirectoryCleanAgent.Operations` 模块（B4 阶段）。若 B4 的 `SHQueryRecycleBin` P/Invoke 调用在特定 Windows 版本上失败，C5 记录警告日志后继续，不阻断模拟运行。
