# 阶段性关键决策与细节控制文档 — B5 回滚管理器

---

#### 一、 关键架构/设计决策记录（必须包含"被否决的方案"）

##### 1. 回滚恢复路径设计

**最终决策**：采用"隔离区优先 → 回收站其次"的双路径恢复策略。隔离区作为主路径（文件命名 `{SHA256前16位}_{原始文件名}_{Unix秒}.quarantine` 完全可控），回收站作为次要路径（按 `$Recycle.Bin\{SID}\$R*` 枚举 + SHA-256 匹配）。

**核心权衡（Trade-off）**：
- 隔离区恢复通过 sha256 前缀 + 文件名构建搜索模式 `{sha256Prefix}_{originalFileName}_*.quarantine`，再逐个计算哈希精确匹配。此方案在隔离区文件数量大时线性搜索较慢，但避开了需要存储精确隔离区路径的 schema 变更。
- 回收站恢复需枚举所有 `$R*` 文件并计算 SHA-256，在大回收站（>1 万个文件）中性能较差（O(n*m)），但以缓存字典缓解同批次中的重复计算。

**被否决的替代方案**：
1. **在 DeletionRecord 表中增加 quarantine_file_path 字段** —— 放弃原因：需修改 B4 阶段已完成的 schema 和 `DeletionRecord` DTO，引入跨阶段重构风险。且隔离区文件可能因过期清理而路径失效，增加该字段会产生"路径存在但文件不存在"的模糊状态。
2. **使用 IFileOperation COM 接口的 RestoreItems 方法从回收站恢复** —— 放弃原因：项目已选择 `SHFileOperationW` 为主路径，引入 `IFileOperation` COM 互操作需定义大量接口（`IShellItem`、`IFileOperationProgressSink` 等），代码量大且与现有模式不一致。

##### 2. 路径冲突解决策略

**最终决策**：恢复时若原路径已存在文件，使用 `_restored_N` 后缀（1~99），而非直接覆盖。若 99 次均冲突则报告失败。

**核心权衡（Trade-off）**：
- 不覆盖已存在文件保证了数据安全，但可能导致恢复后的路径非用户预期。
- 99 次上限是为了防止无限循环（极端情况下目录中已有 `file_restored_1` ~ `file_restored_99`）。

**被否决的替代方案**：
1. **弹出对话框询问用户是否覆盖** —— 放弃原因：`IBackupManager` 是后台服务接口，不应依赖 UI 交互。覆盖/跳过的选择应由 UI 层在调用前完成。

##### 3. 回收站哈希缓存设计

**最终决策**：在 `RollbackAsync` 和 `VerifyHashesAsync` 的回收站恢复路径中使用 `Dictionary<string, string>` 缓存已计算的回收站文件哈希 → 路径映射，避免同一批次中多次计算同一文件的哈希值。

**核心权衡（Trade-off）**：
- 缓存仅在方法调用生命周期内有效（不跨调用），内存占用可控。
- 缓存键是 SHA-256 哈希字符串（64 字符），字典查找极快。

**被否决的替代方案**：
1. **在回收站目录预建索引** —— 放弃原因：回收站状态随时变化（系统后台清理、用户手动清空），预建索引容易过时且增加复杂度。

#### 二、 隐含前提与关键假设清单（风险预警）

1. **[假设 1]**：假设隔离区文件在恢复前未被系统或手动清理（设计文档定义 30 天保留期，默认 `QuarantineRetentionDays=30`）。若清理先于回滚发生，该文件将报告 `QUARANTINE_FILE_NOT_FOUND`。
2. **[假设 2]**：假设隔离区文件命名格式 `{SHA256前16位}_{原始文件名}_{Unix秒}.quarantine` 中的原始文件名不含特殊字符导致搜索模式不匹配（如文件名含通配符 `*` 或 `?`）。当前实现使用 `Directory.EnumerateFiles` 进行匹配，若文件名含通配符可能产生误匹配，但 NTFS 文件名不允许这些字符（设计文档规定了路径校验），此风险极低。
3. **[假设 3]**：假设回滚时当前用户与执行删除操作的用户相同（回收站按用户 SID 隔离）。若不同用户执行回滚，回收站路径不匹配，回退到隔离区恢复或报告失败。
4. **[假设 4]**：假设 `SHFileOperationW` 的 `FO_MOVE` 操作在跨卷场景下能正确执行（复制+删除）。若此行为在特定 Windows 版本中变化，有 `File.Move` 作为回退。
5. **[假设 5]**：假设 `DeleteSnapshotEntry.Sha256Hash` 与 `DeletionRecord.FileHash` 一致（均由 B3 决策引擎同一哈希计算流程产出）。

---

#### 三、 边界与范围界定（明确"不做什么"）

1. **[明确不做]**：本期不做"定期清理机制"中的启动扫描（检查墓碑记录 file_path 是否已存在新文件）。此项属于 B5 的附属功能，但实现依赖于 Everything SDK 的 FRN 探针能力（需启动时加载），与回滚核心逻辑耦合度低，应在独立组件中实现（如 `TombstoneStartupScanner` 或在 `TombstoneCache.LoadAsync` 中集成）。
2. **[明确不做]**：本期不做隔离区空间配额监控和自动清理。此功能属于 B6（隔离区管理）的范围。
3. **[明确不做]**：本期不做跨用户的回滚。回滚仅支持同一 Windows 用户（回收站按 SID 隔离）。
4. **[明确不做]**：本期不做回滚操作的审计记录写入（不写 AuditLog）。回滚结果通过日志记录，不在 AuditLog 表中追加条目（现有 `IAuditLogRepository` 仅有 Insert 接口，回滚记录格式尚未定义）。
5. **[明确不做]**：本期不做网络驱动器/UNC 路径文件的回收站恢复（`$Recycle.Bin` 仅存在于本地 NTFS 卷）。

#### 四、 待确认事项（需要我人工介入的模糊点）

1. **[待确认项 A]**：`IBackupManager.GetRollbackHistoryAsync` 返回的 `OperationBatchInfo.IsRollbackable` 字段当前始终返回 `true`（由 `DeletionRecordRepository.GetBatchSummariesAsync` 设定）。实际应根据隔离区文件是否存在、是否超过保留期等因素动态计算。**需业务方确认是否需要在 B5 阶段实现动态计算。**
2. **[待确认项 B]**：回滚后的隔离区文件处理策略：当前实现将隔离区文件移动（Move）回原路径，隔离区文件消失。是否需要保留隔离区副本（Copy 而非 Move）？**需确认安全策略。**
3. **[待确认项 C]**：回收站恢复中，`Directory.EnumerateFiles(path, "$R*")` 匹配所有回收站数据文件。某些 Windows 版本可能有 `$I` 元数据文件前缀。**当前仅匹配 `$R*`，需确认是否涵盖所有回收站实现。**
4. **[待确认项 D]**：`UserConfig.DataStoragePath` 为空时，默认路径为 `%LocalAppData%\SpaceClear`。此行为与 `OperationExecutor` 一致。**需确认是否需要在配置界面中显式展示此路径。**

#### 五、 关键外部依赖锁定

*   **依赖项**：`ShellFileOperation.MoveFile`（底层 `SHFileOperationW` FO_MOVE）—— 若 Windows Shell API 行为在特定版本中与预期不一致（如跨卷 Move 失败），我们的隔离区恢复主路径将受影响。已有 `File.Move` 回退机制。
*   **依赖项**：`System.Security.Principal.WindowsIdentity.GetCurrent()` —— 若应用程序在非 Windows 平台或受限沙箱中运行，获取用户 SID 将失败，回收站恢复路径将不可用（隔离区主路径不受影响）。
*   **依赖项**：`ILocalTombstoneRepository.DeleteByOperationIdAsync` —— 若 SQLite 数据库被锁定或损坏，墓碑清除失败。当前实现中墓碑清除失败不阻断回滚结果（文件已恢复，墓碑将随过期机制自动清理）。
*   **依赖项**：B4 阶段 `OperationExecutor` 的隔离区文件命名格式 —— 若 B4 的命名格式 `{SHA256前16位}_{原始文件名}_{Unix秒}.quarantine` 发生变化，B5 的搜索模式将无法匹配。此格式在两个类中使用了相同的常量定义（`QuarantineFileNameFormat = "{0}_{1}_{2}.quarantine"`），但非编译期强制约束。
