# A6-本地化基础设施 — 阶段性关键决策与细节控制文档

---

## 一、 关键架构/设计决策记录

### 决策1：ResourceManager + .resx 标准方案

1. **最终决策**：使用 .NET 标准 `ResourceManager` + `.resx` 嵌入资源文件实现本地化，而非 JSON 配置方案。

2. **核心权衡（Trade-off）**：
   - 牺牲了 JSON 的简单性和运行时动态修改能力；
   - 换取了完整的 .NET 生态工具链支持（VS 设计器、自动生成强类型类、XAML `x:Static` 原生绑定、CultureInfo 回退链）；
   - 增加了一个 `System.Resources.ResourceManager` 的依赖，但这是 .NET 基础运行时自带的，无需额外 NuGet 包。

3. **被否决的替代方案**：
   - **方案A（纯 JSON 字典）**：放弃理由——无法利用 WPF XAML 的 `x:Static` 绑定能力，每个 UI 控件需要手动订阅语言变更事件刷新文本，增加大量样板代码和事件订阅开销。
   - **方案B（自定义 .NET Cultures 注册）**：放弃理由——需要注册自定义 CultureInfo，增加应用启动复杂度，且与 Windows 系统区域设置耦合风险高。

### 决策2：语义标签映射独立于通用资源

1. **最终决策**：语义标签通过 `SemanticLabelLocalizer` + `Dictionary<SemanticCategory, (string Zh, string En)>` 硬编码映射表管理，不放入 `.resx` 文件。

2. **核心权衡（Trade-off）**：
   - 牺牲了语义标签的对外可扩展性（新增标签需要修改代码而非仅修改 .resx 文件）；
   - 换取了编译时完整性检查（枚举驱动的类型安全，新增枚举值后编译器强制要求补全映射表）；
   - 8 个语义标签属于设计文档明确定义的有限集合，不需要运行时动态扩展。

3. **被否决的替代方案**：
   - **方案A（语义标签放入 .resx）**：放弃理由——枚举值到 `.resx` 键的映射关系需要手动维护命名约定，易出现键名拼写错误；且语义标签的稳定性超过了通用 UI 文本，不适合依赖字符串键查找。
   - **方案B（所有 UI 文本硬编码 + 语言字典）**：放弃理由——通用 UI 文本量大（>100 条），硬编码字典难以维护；`.resx` 的 ResourceManager 自动回退机制更适合通用 UI 场景。

### 决策3：保持 FileDecisionCache.SemanticCategory 为 string 类型

1. **最终决策**：`FileDecisionCache.SemanticCategory` 和 `RuleResult.SemanticCategory` 保持 `string` 类型不变，通过 `SemanticLabelLocalizer.LocalizeFromString()` 方法进行字符串→本地化文本转换。

2. **核心权衡（Trade-off）**：
   - 牺牲了编译时类型安全（允许规则引擎产生任意字符串标签）；
   - 换取了零级联修改——避免改动 Repository 的 SQLite 读写代码、测试数据构造代码、以及 DB 层字段映射；
   - `SemanticCategory` 枚举作为"推荐标签集"存在，自定义规则仍可产出非标准标签字符串。

3. **被否决的替代方案**：
   - **方案A（改为枚举类型）**：放弃理由——需要级联修改 `FileDecisionCacheRepository`（SQL 参数绑定、读取映射）、`RuleResult`、3+ 个测试文件中的字符串断言；对 0.5d 任务而言过度设计。

## 二、 隐含前提与关键假设清单

1. [假设1]：**假设 .resx 资源的 `ResourceManager` baseName 路径正确**。`Strings.resx` 位于 `DirectoryCleanAgent.Core/Localization/` 下，默认命名空间为 `DirectoryCleanAgent.Core`，因此 baseName 为 `DirectoryCleanAgent.Core.Localization.Strings`。若项目重命名命名空间或 `.resx` 文件移动位置，`ResourceManager` 将无法找到嵌入资源，所有 `GetString` 调用返回占位符 `[[key]]`。

2. [假设2]：**假设 .NET SDK 默认的 `<EnableDefaultEmbeddedResourceItems>true</EnableDefaultEmbeddedResourceItems>` 生效**。该属性确保 `**/*.resx` 文件自动编译为 `.resources` 并嵌入程序集。若项目或 Directory.Build.props 中有覆盖此默认值的设置，`.resx` 文件将不会被嵌入。

3. [假设3]：**假设 `CultureInfo("en-US")` 回退到 `CultureInfo("en")` 的父级回退链正常工作**。当用户选择 "en-US" 时，`ResourceManager.GetString(key, new CultureInfo("en-US"))` 会先查找 `Strings.en-US.resx`（不存在），然后回退到 `Strings.en.resx`。这是 .NET 标准区域回退行为，假设未被自定义 `ResourceManager` 配置覆盖。

4. [假设4]：**假设 8 个语义标签在 V3.7 范围内不增加**。当前 `SemanticCategory` 枚举和 `LabelMap` 硬编码了 8 个标签。若后续版本增加新的语义标签（如 V3.8 计划的 VSS 感知），需要同步更新枚举、映射表和单元测试。

## 三、 边界与范围界定

1. [明确不做]：**WPF XAML 资源绑定**。本阶段仅提供 `ILocalizationService` 接口和实现，不涉及 `<TextBlock Text="{x:Static ...}"` 等 XAML 绑定代码。XAML 绑定属于 C1-C8 UI 层任务。

2. [明确不做]：**设置页面的语言下拉菜单 UI**。本阶段仅提供后端 `SetLanguage()` 方法和 `LanguageChanged` 事件。语言选择 UI 控件属于 C4（设置页面）任务。

3. [明确不做]：**向导步骤的实际 UI 文本绑定**。虽然 `.resx` 文件已包含向导文本键值对，但向导 UI 的文本绑定属于 C3（首次启动向导）任务。

4. [明确不做]：**翻译完整性（i18n 校对）**。当前 `.resx` 文件包含约 100 个核心 UI 键的翻译，但后续任务可能增量补充新键。英文翻译基于技术文档语义完成，未经过专业本地化审校。

5. [延期处理]：**第三方组件（Everything、SQLite 等）的本地化**。第三方工具的错误消息和界面文本由其自身语言设置控制，不在本工具本地化范围内。

## 四、 待确认事项

1. [待确认项A]：**语义标签图标映射**。设计文档 3.3 节为每个语义标签定义了图标（🗑️、🧹、📥、♻️、📋、📁、🛡️、❓），当前 `SemanticLabelLocalizer` 仅处理文本映射，图标映射未纳入。需确认图标方案（Unicode emoji vs 自定义 SVG/Path 图标），如需要可在后续 C2（文件列表视图）中补充。

2. [待确认项B]：**语言切换后的 UI 刷新策略**。当前 `LanguageChanged` 事件在 `LocalizationService.SetLanguage()` 时触发，但是否需要"立即全量刷新所有打开的窗口和控件"还是"仅更新后续打开的新窗口"取决于 UI 框架的实现选择。需要 UI 团队确认刷新策略。

3. [待确认项C]：**英文 locale 标识符**。当前使用 "en-US" 作为英文标识符。若未来需要支持英式英语（en-GB）或其他英文变体，需评估是否需要父级区域 "en" 还是具体区域 "en-US"。
