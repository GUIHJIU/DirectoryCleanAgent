> 生成日期：2026-06-23 | V3.7 | 状态：代码已完成，34项测试通过，待人工复核

## 一、 关键架构/设计决策记录

### 决策1：MVVM 模式 — 手动实现 vs 社区工具包

1. **最终决策**：手动实现 `INotifyPropertyChanged` + `ICommand`（RelayCommand），不引入 CommunityToolkit.Mvvm 等第三方 MVVM 框架。
2. **核心权衡（Trade-off）**：
   - 牺牲了源生成器带来的样板代码减少（每个属性需要约 8 行手动代码 vs 源生成器的 1 行 Attribute）。
   - 换取了零外部依赖、与设计文档声明技术栈完全一致、以及未来团队成员的零学习成本。
3. **被否决的替代方案**：
   - **CommunityToolkit.Mvvm 8.2**：引入未在设计文档中声明的 NuGet 包，且其源生成器依赖 Roslyn 4.x+编译特性，若团队 CI 环境不一致会导致编译失败。
   - **Prism / ReactiveUI**：重型框架，对当前 C1 阶段仅需 1 个 MainViewModel 的场景严重过度设计。

### 决策2：依赖注入容器 — App.xaml.cs 统一管理

1. **最终决策**：在 `App.xaml.cs` 的 `OnStartup` 中构建 `ServiceCollection`，利用已有的 `AddConfigLayer()` / `AddLocalizationLayer()` 扩展方法，新增 `IAppStateService` / `IDashboardDataProvider` 注册后，通过 `IServiceProvider` 解析 `MainWindow`。
2. **核心权衡（Trade-off）**：
   - 所有服务通过 DI 容器获取，增加了 App.xaml.cs 的复杂度（约 180 行），但与后端模块（B1-B7）的 DI 模式保持一致，后续替换 Mock 服务只需修改 2 行注册代码。
3. **被否决的替代方案**：
   - **直接在 MainWindow 中 new ViewModel**：C1 阶段最简单，但 B 阶段后端就绪后必须大规模重构所有构造函数调用链，不采纳。

### 决策3：Mock 数据隔离策略 — 接口抽象

1. **最终决策**：在 UI 项目中定义 `IDashboardDataProvider` 接口，`MockDashboardDataProvider` 实现硬编码模拟数据。后续后端就绪后，新建 `RealDashboardDataProvider` 实现同一接口，在 DI 注册处替换。
2. **核心权衡（Trade-off）**：
   - Mock 数据硬编码在代码中（而非 JSON 文件），牺牲了非程序员调整测试数据的能力。
   - 换取了零文件 I/O 启动开销（满足 <3 秒可交互要求）和类型安全。
3. **被否决的替代方案**：
   - **读取 `docs/A1.5-Mock数据样例.json`**：增加了文件系统依赖和反序列化开销，对 Mock 阶段价值有限。

### 决策4：主题系统 — XAML ResourceDictionary 动态替换

1. **最终决策**：LightTheme.xaml / DarkTheme.xaml 通过 `ResourceDictionary.MergedDictionaries` 动态替换，FollowSystem 模式通过注册表读取 Windows 个性化设置。
2. **核心权衡（Trade-off）**：
   - 所有颜色在 XAML 中以 `SolidColorBrush` 静态资源定义，控件样式通过 `{StaticResource}` 引用——意味着主题切换时不会自动刷新已渲染控件的颜色（仅影响新创建的控件）。当前通过 ToggleTheme 命令中的手动替换 ResourceDictionary 来补救。
   - 正确的实现应使用 `{DynamicResource}`，但会引入运行时的资源查找开销。
3. **被否决的替代方案**：
   - **DynamicResource 全量替换**：性能开销在高频 UI 更新场景（如进度条）下不可忽视，且 C1 阶段主题切换为低频操作，不值得。

## 二、 隐含前提与关键假设清单（风险预警）

1. **[假设1]**：假设 `LocalizationService` 的 `zh-CN` 默认资源已包含 C1 阶段所有 UI 文本的键。当前实现中 UI 文本直接硬编码于 XAML 和 C# 中（如"一键清理安全项"、"本次释放"等），**尚未与 ILocalizationService 集成**。若后续本地化需求启动，需逐条替换为 `{Binding}` 绑定或资源引用。（风险等级：中）
2. **[假设2]**：假设 Everything SDK 的索引状态检查可通过 Mock 数据模拟，且 `IsIndexReady` 属性的 true/false 切换能充分覆盖 UI 状态组合。实际上 Everything IPC 的异常场景（如 SDK 版本不兼容、IPC 通道断开等）在本阶段无法测试。（风险等级：低）
3. **[假设3]**：假设管理员权限检测逻辑（`WindowsPrincipal.IsInRole`）在所有 Windows 版本上的行为一致。已知在某些域环境中，本地管理员可能不被 `WindowsBuiltInRole.Administrator` 识别。（风险等级：低）
4. **[假设4]**：假设 NuGet 包（Serilog、Microsoft.Extensions.*）在离线构建环境中的本地缓存可用。当前构建已验证离线 restore 可用，但若 CI 环境首次构建且无网络，需提前配置本地 NuGet 源。（风险等级：高）

## 三、 边界与范围界定（明确"不做什么"）

1. **[明确不做]**：C1 阶段不实现主内容区（左侧分组树 + 右侧文件列表）—— 属于 C2 任务，当前仅显示"更多功能开发中"占位。
2. **[明确不做]**：C1 阶段不实现设置窗口、帮助窗口 —— 对应 C6/C10 任务，当前按钮仅弹出 MessageBox 占位提示。
3. **[明确不做]**：C1 阶段不实现导出报告的真实逻辑（HTML/CSV 生成）—— 属于 C5 任务。
4. **[明确不做]**：C1 阶段不做真实的文件扫描、规则引擎调用、Everything IPC 通信 —— 全部 Mock 模拟。
5. **[明确不做]**：C1 阶段不做 VSS 卷影副本感知功能 —— 设计文档标注延后至 V3.8+。
6. **[明确不做]**：C1 阶段不做 UI 自动化测试（如 Appium/WinAppDriver）—— 仅提供 xUnit 单元测试覆盖 ViewModel 和 Converters。
7. **[明确不做]**：C1 阶段不做 ILocalizationService 的完整集成 —— UI 文本当前为中文硬编码，本地化切换框架已就绪但资源键映射表待 C6 统一处理。

## 四、 待确认事项（需要人工介入的模糊点）

1. **[待确认A]**：主窗口标题"智能磁盘清理"是否需要加版本号？当前 XAML Title 为"智能磁盘清理"，顶部栏显示"智能磁盘清理 V3.7"。需确认最终对外名称。
2. **[待确认B]**：仪表板 4 张卡片的图标（🟢💡⚠️🛡️）为 emoji 字符，在某些 Windows 字体（如 Segoe UI）下渲染效果可能不一致。是否需要替换为自定义 SVG/Path 图标？
3. **[待确认C]**：操作栏"分组依据"下拉框选项（路径/时间/大小/类型/语义标签）中的"语义标签"在非 AI 模式下是否应隐藏？当前始终显示。
4. **[待确认D]**："显示所有文件"Toggle 开关的默认状态是关闭还是开启？当前默认关闭（不显示所有文件）。
5. **[待确认E]**：非管理员只读模式下，"模拟运行"和"导出报告"按钮是否应该保持可用？当前实现中它们保持可用（符合设计文档"非管理员可分析"的要求），但需二次确认产品逻辑。
6. **[待确认F]**：进度条中 `Percentage = -1`（总数未知）时使用 `IsIndeterminate` 模式，但这依赖 `InverseBoolConverter` 对负数百分比的转换逻辑——当前实现可能不正确（Percentage 为 double，InverseBoolConverter 处理 bool）。需确认并修复或采用专门的 `ProgressIndeterminateConverter`。
