# B6-隔离区管理（空间+大文件绕过+异步SHA256）

**阶段**：B6 | **依赖**：B4, A3 | **预估工时**：3 d | **完成日期**：2026-06-23

---

## 一、关键架构/设计决策记录

### 1. 创建独立的 IQuarantineManager 接口集中管理隔离区

- **最终决策**：创建 `IQuarantineManager` 接口和 `QuarantineManager` 实现类，将隔离区目录解析、空间监控、容量强制、过期清理、文件命名等逻辑从 OperationExecutor 和 BackupManager 中提取到独立的集中式组件。
- **核心权衡（Trade-off）**：
  - 优势：消除 OperationExecutor 和 BackupManager 中 `GetQuarantineDirectory` 的重复代码（两处各 ~15 行完全相同）；将隔离区关注点从执行器和回滚器中解耦；新增的空间监控和过期清理功能有明确的归属。
  - 代价：增加了一个接口和一个实现类（~350 行）；OperationExecutor 和 BackupManager 各增加一个构造函数依赖参数；现有测试需要添加 `Mock<IQuarantineManager>`。
- **被否决的替代方案**：
  1. **在 OperationExecutor 中直接扩展隔离区管理功能**：被否决，因为会导致 BackupManager 继续维护重复的隔离区路径解析逻辑，且空间监控和过期清理与删除执行职责混淆。
  2. **将隔离区管理逻辑全部放入静态工具类**：被否决，因为无法进行依赖注入和单元测试 Mock，且静态类不支持 IProgress/CancellationToken 等异步模式。

### 2. 文件名中嵌入 Unix 时间戳作为过期判断依据

- **最终决策**：清理过期文件时，从隔离区文件名中解析 Unix 时间戳（如 `A1B2C3D4E5F6A7B8_file.txt_1718845200.quarantine`）来判断文件是否过期，而非使用文件系统的 LastWriteTime。
- **核心权衡（Trade-off）**：
  - 优势：文件复制/移动操作会改变文件系统时间戳，但文件名中的时间戳不可变，能准确反映文件最初被删除的时刻。
  - 代价：依赖文件命名规范的严格执行；如果外部程序重命名了隔离区文件，解析会失败（安全保守策略：跳过无法解析的文件）。
- **被否决的替代方案**：
  1. **使用文件系统 CreationTime**：被否决，因为跨卷复制会重置 CreationTime，导致过期判断失准。

### 3. ParseQuarantineFileName 采用首/末下划线二段式解析

- **最终决策**：解析 `{SHA256}_{FileName}_{UnixSeconds}.quarantine` 格式时，使用第一个 `_` 定位 SHA256 前缀边界，最后一个 `_` 定位 Unix 时间戳边界，中间部分（含所有 `_`）为原始文件名。
- **核心权衡（Trade-off）**：
  - 优势：支持文件名中含任意数量下划线（如 `项目_文档_v2.docx`）；时间复杂度 O(n)，无需正则表达式。
  - 代价：要求 SHA256 前缀固定 16 位（不含 `_`），Unix 时间戳为纯数字（不含 `_`）；若文件命名格式变更，此解析逻辑需同步更新。
- **被否决的替代方案**：
  1. **使用正则表达式 `^([^_]+)_(.+)_(\d+)\.quarantine$`**：被否决，因为 SHA256 前缀固定 16 位且文件名含下划线时，正则的 `(.+)` 会贪婪匹配导致 Unix 时间戳被吞入文件名段。

### 4. 过期清理仅匹配 .quarantine 文件

- **最终决策**：`CleanupExpiredAsync` 和 `PurgeAllAsync` 仅操作 `*.quarantine` 文件，不触及隔离区目录中其他扩展名的文件。
- **核心权衡（Trade-off）**：
  - 优势：防止误删用户手动放入隔离区目录的无关文件（如日志、配置文件）；与 B4 的 `.cleaning.tmp` 临时文件（跨卷复制中间状态）自然隔离。
  - 代价：如果未来隔离区文件扩展名变更，清理逻辑需同步更新。
- **被否决的替代方案**：
  1. **清空整个隔离区目录**：被否决，因为隔离区目录可能与用户数据目录重合，清空整个目录风险过高。

## 二、隐含前提与关键假设清单

1. **[假设1]**：隔离区目录位于 `{DataStoragePath}/quarantine/`，且该路径对应用程序有读写权限。若 DataStoragePath 配置了一个无权限的路径，`GetQuarantineDirectoryAsync` 的 `Directory.CreateDirectory` 会失败，后续所有隔离区操作将返回 null（禁用状态）。
2. **[假设2]**：隔离区文件命名规范 `{SHA256前16位}_{原始文件名}_{Unix秒}.quarantine` 由所有写入方（OperationExecutor、QuarantineManager）和所有读取方（BackupManager、QuarantineManager）严格遵守。若外部程序创建了不符合规范的 `.quarantine` 文件，过期清理会跳过它们。
3. **[假设3]**：隔离区文件数量在可枚举范围内（通常数千到数万）。若隔离区累积数十万文件，`Directory.EnumerateFiles` 的枚举性能可能成为瓶颈（当前无分页机制）。
4. **[假设4]**：`QuarantineRetentionDays` 使用自然天数（86400 秒/天），不考虑闰秒。
5. **[假设5]**：大文件绕过阈值（卷剩余空间的 20%）在当前磁盘容量下合理。对于剩余空间极小（<100MB）的磁盘，20% 阈值可能导致几乎所有文件都被绕过。

## 三、边界与范围界定

1. **[明确不做]**：本期 **不实现** 隔离区文件的压缩存储。隔离区文件以原始格式存储。
2. **[明确不做]**：本期 **不实现** 隔离区文件的自动定时清理（定时清理属于自动化调度器，延后至后续里程碑）。
3. **[明确不做]**：本期 **不实现** 隔离区 UI 界面（隔离区管理界面属于 C7 任务，依赖 B6 完成后实施）。
4. **[明确不做]**：本期 **不实现** 隔离区文件的跨设备同步或远程隔离区。
5. **[明确不做]**：本期 **不修改** `HashVerifier` 类。该类已被 B4/B5 使用，且在 B6 隔离区清理流程中不需要（清理使用 File.Delete 而非哈希校验）。
6. **[明确不做]**：本期 **不修改** `IBackupManager` 接口。BackupManager 的恢复逻辑保持完整，仅其内部实现从 `GetQuarantineDirectory(config)` 切换为 `_quarantineManager.GetQuarantineDirectoryAsync(ct)`。
7. **[延期处理]**：隔离区空间使用在接近上限时的自动通知提醒（如 Windows 通知或 UI 横幅）延后至 C7 UI 阶段。

## 四、待确认事项

1. **[待确认项A]**：`UserConfig.QuarantineMaxSizeBytes` 默认值为 10GB。若用户磁盘空间紧张（如 128GB SSD），10GB 可能过大。建议在首次启动向导中增加配置引导。
2. **[待确认项B]**：过期清理的时间判断使用文件名中嵌入的 Unix 秒时间戳。若因 NTP 同步导致系统时间回退，新创建的文件可能被错误标记为过期。当前代码未做时钟回退保护。
3. **[待确认项C]**：`GetQuarantineStatsAsync` 的 `DriveFreeBytes` 字段用于 UI 展示。当前实现查询隔离区所在卷时可能因权限问题失败（返回 0）。UI 层应处理此值为 0 的显示逻辑。
4. **[待确认项D]**：BackupManager 的 6 个集成测试（`RollbackAsync_Integration_*` 等）在 B6 重构前已存在问题——测试中隔离区文件内容与原始文件不一致导致 SHA-256 校验失败。这些测试需要在后续修复（修改测试文件内容或更新期望结果），但这是预存问题，非 B6 引入。

## 五、关键外部依赖锁定

*   **依赖项1**：`IConfigService`（A4 阶段产出）——QuarantineManager 依赖此服务获取 `QuarantineMaxSizeBytes`、`QuarantineRetentionDays`、`DataStoragePath` 配置。若 A4 的 JSON 配置序列化格式变更，QuarantineManager 的配置读取可能受影响。
*   **依赖项2**：`PathNormalizer`（A5 阶段产出）——QuarantineManager 使用 `PathNormalizer.Normalize` 标准化隔离区目录路径。若路径标准化逻辑变更（如 `\\?\` 前缀规则调整），隔离区目录创建和文件枚举可能受影响。
*   **依赖项3**：`ILogger<QuarantineManager>`（A1 阶段基础设施）——所有日志输出依赖 Microsoft.Extensions.Logging。若日志基础设施不可用，QuarantineManager 构造函数会因 null 检查抛出异常。
*   **依赖项4**：`Everything SDK`（A2 阶段产出）——B6 不直接依赖 Everything SDK，但隔离区文件命名中使用的 SHA-256 哈希来源于 B3 快照生成时的计算。若 B3 的哈希计算方式变更（如改用 SHA-512），隔离区文件名的 SHA256 前缀将失效。
*   **依赖项5**：`ShellFileOperation`（B4 阶段产出）——BackupManager 的隔离区恢复依赖此组件执行文件移动。B6 不直接依赖，但备份恢复回路依赖其稳定性。
