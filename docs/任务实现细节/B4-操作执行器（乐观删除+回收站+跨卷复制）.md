# B4-操作执行器（乐观删除+回收站+跨卷复制）— 阶段性关键决策与细节控制文档

---

## 一、 关键架构/设计决策记录

### 1. SHFileOperationW 替代 IFileOperation COM

- **最终决策**：采用 `SHFileOperationW`（shell32.dll P/Invoke）作为文件操作的主路径，而非设计文档中提到的 `IFileOperation` COM 接口。
- **核心权衡（Trade-off）**：
  - 牺牲：`IFileOperation` COM 接口提供更丰富的进度回调（`IFileOperationProgressSink`）、Better COM 生命周期管理。`SHFileOperationW` 的进度回调较粗粒度（仅支持进度对话框）。
  - 换取：大幅降低实现复杂度。`SHFileOperationW` 仅需 1 个结构体 + 1 个函数声明，而 `IFileOperation` 需定义 3+ 个 COM 接口（`IFileOperation`、`IFileOperationProgressSink`、`IShellItem`）+ `SHCreateItemFromParsingName` P/Invoke + COM 对象生命周期管理。代码量从 ~500 行降至 ~150 行，且 SHFileOperationW 是经过二十年验证的稳定 API。
- **被否决的替代方案**：
  1. **完全实现 IFileOperation COM 互操作**：否决原因 —— 手动 COM 互操作代码量巨大、易出错（GUID 对齐、vtable 布局、Marshal 释放），投入产出比低。
  2. **引入 Vanara.PInvoke.Shell32 NuGet 包**：否决原因 —— 该包体量庞大（数十 MB），对单一功能引入重型依赖，与项目"手动 P/Invoke"的现有模式（见 EverythingNative.cs）不一致。

### 2. 仓储接口跨层依赖解决方案

- **最终决策**：Operations 项目增加对 Data 项目的直接项目引用（`DirectoryCleanAgent.Data`），而非将仓储接口提升至 Core 层。
- **核心权衡（Trade-off）**：
  - 牺牲：违反了"核心能力层不应依赖持久化层"的清洁架构原则。Operations 现在编译时依赖于 Data 项目的接口定义。
  - 换取：以最小代码改动（仅修改 .csproj 一行）完成 B4 集成，避免了将 `ILocalTombstoneRepository`、`IDeletionRecordRepository`、`IAuditLogRepository` 三个接口迁移至 Core 层所需的大范围重构（涉及 Data 实现类、DI 注册、所有测试文件的 using 语句变更）。
- **被否决的替代方案**：
  1. **将仓储接口提升至 Core.Interfaces**：否决原因 —— 涉及 6+ 文件的变更，超出 B4 单阶段任务范围，且需同步修改所有现有引用。
  2. **通过 IServiceProvider 运行时解析**：否决原因 —— 失去编译时类型安全，增加运行时错误风险。

### 3. 错误处理策略：文件不存在 = 删除成功

- **最终决策**：`SHFileOperationW` 返回 `ERROR_FILE_NOT_FOUND`（0x80070002）时视为删除成功。
- **核心权衡（Trade-off）**：
  - 牺牲：丢失了对"文件已不在预期位置"这一异常状态的精确追踪能力。
  - 换取：在磁盘清理场景中，文件已被其他进程或用户删除属于正常状态（目标已达成），不应标记为"失败"触发不必要的用户关注。
- **被否决的替代方案**：
  1. **文件不存在视为独立状态（NotPresent）**：否决原因 —— 增加 DTO 字段和 UI 处理复杂度，用户不关心文件"如何消失"。

### 4. 隔离区备份与删除的一体化设计

- **最终决策**：隔离区备份逻辑内置在 `ExecuteAsync` 中（而非独立接口），通过 `UserConfig.QuarantineMaxSizeBytes > 0` 判断是否启用。
- **核心权衡（Trade-off）**：
  - 牺牲：`ExecuteAsync` 方法承担了双重职责（备份+删除），单一方法复杂度较高（~500 行）。
  - 换取：调用方无需协调两个独立操作（`BackupAsync` + `ExecuteAsync`），避免备份成功但删除失败时的状态不一致。
- **被否决的替代方案**：
  1. **拆分为 IQuarantineManager 独立接口**：否决原因 —— 接口爆炸（IOperationExecutor + IQuarantineManager 需协同），增加调用方的编排复杂度。

## 二、 隐含前提与关键假设清单（风险预警）

1. **[假设1]**：假设 `SHQueryRecycleBinW`（shell32.dll）在所有 Windows 10 1809+ 系统上均存在且行为一致。若该 API 在某些精简版 Windows 或 Wine/Proton 环境中不可用，`QueryRecycleBinCapacityAsync` 将保守返回 `TotalBytes=0`（无限制），可能导致回收站空间溢出。
2. **[假设2]**：假设 `SHFileOperationW` 在 `\\?\` 前缀路径下正常工作。Windows Shell API 传统上不接受 `\\?\` 扩展路径，但测试表明 Vista+ 版本已支持。若某些边缘版本不支持，删除操作将全部失败。
3. **[假设3]**：假设临时文件删除测试可在 `%TEMP%` 目录执行。若该目录权限受限或磁盘空间不足，涉及 `CreateTempFile` 的单元测试将失败。
4. **[假设4]**：假设 `FRN_AVAILABLE` 标志（来自 `UserConfig`）在 B4 执行时已由 B1 阶段的 Everything 启动检测正确设置。若该标志与实际 FRN 能力不一致，墓碑匹配可能失效。
5. **[假设5]**：假设隔离区目录的 `DriveInfo.AvailableFreeSpace` 在 B4 执行期间稳定不变。若磁盘空间在操作过程中急剧变化（如其他进程大量写入），大文件绕过隔离区的判断可能过时。

## 三、 边界与范围界定（明确"不做什么"）

1. **[明确不做]**：本期不做 `IFileOperation` COM 接口的完整实现。当前使用 `SHFileOperationW` 实现等价功能。`IFileOperation` COM 封装作为远期优化项（需评估进度回调细粒度需求后再决定）。
2. **[明确不做]**：本期不做回滚功能。`IOperationExecutor` 仅负责删除/备份操作。回滚（从回收站/隔离区恢复）属于 B5 `IBackupManager` 的职责范围。
3. **[明确不做]**：本期不做自动化调度清理。`IOperationExecutor` 仅响应手动触发的清理操作（通过 UI 按钮调用）。全自动定时清理延后至后续里程碑。
4. **[明确不做]**：本期不引入 `Vanara.PInvoke` 或 `Microsoft.Windows.SDK` 等重型 NuGet 包。所有 Windows API 调用采用手动 P/Invoke（与 Everything 模块一致）。
5. **[明确不做]**：本期不做跨平台兼容。所有代码依赖 Windows Shell API（`SHFileOperationW`、`SHQueryRecycleBinW`、`WindowsIdentity`），`net8.0-windows` 目标框架明确限定 Windows 平台。
6. **[明确不做]**：隔离区空间配额（`QuarantineMaxSizeBytes`）的动态监控与自动清理在 B5 实现。

## 四、 待确认事项（需要人工介入的模糊点）

1. **[待确认项A]**：`SHQueryRecycleBinW` 返回的 `SHQUERYRBINFO.i64NumItems` 字段含义存在模糊性。MSDN 文档中该字段表示"回收站中的对象数量"，但在部分实现中实际存储的是已用字节数。当前代码将 `i64Size` 作为 `TotalBytes`、`i64NumItems` 作为 `UsedBytes`。**需在 Windows 10/11 实体机上通过调试器验证实际返回值语义**，以确认 `AvailableBytes` 的计算公式（`TotalBytes - UsedBytes`）是否正确。
2. **[待确认项B]**：`LocalTombstone.CreateTombstone` 方法在 B4 阶段使用 `DateTime.UtcNow` 作为 `OriginalLastWriteTime` 的替代值，因为 `DeleteSnapshotEntry` 不包含原始文件的 `LastWriteTime` 字段。**若墓碑的指纹降级匹配需要精确的 `LastWriteTime`**，需在 `DeleteSnapshotEntry` 中增加该字段（影响 DTO 设计和 B3 快照生成逻辑）。
3. **[待确认项C]**：`GetQuarantineDirectory` 方法中隔离区目录默认为 `{DataStoragePath}/quarantine/`。若实际部署中隔离区位置需可配置（如独立磁盘），需在 `UserConfig` 中增加 `QuarantinePath` 字段。
4. **[待确认项D]**：永久删除（`DeleteMethod.Permanent`）时是否需要 SHFileOperationW 还是直接使用 `File.Delete`？当前使用 SHFileOperationW（与其他路径一致），但 `File.Delete` 配合 `\\?\` 路径可能性能更优。

## 五、 关键外部依赖锁定

*   **依赖项1：shell32.dll（Windows Shell API）**。`SHFileOperationW`、`SHQueryRecycleBinW` 依赖此 DLL。该 DLL 为 Windows 核心组件，在所有 Windows 10 1809+ 系统上均存在。若操作系统本身完整，此依赖无失效风险。
*   **依赖项2：kernel32.dll（Windows Kernel API）**。`GetVolumeInformationW`、`MoveFileExW` 依赖此 DLL。同上，为 Windows 核心组件。
*   **依赖项3：Microsoft.Data.Sqlite（NuGet v8.0.11）**。Operations 项目通过 Data 项目间接依赖此包。此依赖已在 A5 阶段锁定版本。若此包因网络策略无法下载，Operations 项目将无法还原。
*   **依赖项4：DirectoryCleanAgent.Data 项目**。Operations 项目依赖 Data 的仓储接口（`ILocalTombstoneRepository`、`IDeletionRecordRepository`、`IAuditLogRepository`）和 `ITombstoneCache` 接口（来自 Core）。若 Data 项目的接口签名发生变更，Operations 项目需同步更新。
*   **依赖项5：DirectoryCleanAgent.Core 项目**。Operations 依赖 Core 的 `CancellableOperationBase`（取消基类）、`PathNormalizer`（路径标准化）、`IConfigService`（配置读取）、所有 DTO 和枚举定义。Core 层变更会直接影响 Operations。
