# 阶段性关键决策与细节控制文档

**任务号**：B2 | **任务概要**：规则引擎（硬规则+启发式+热加载） | **完成日期**：2026-06-22

---

#### 一、 关键架构/设计决策记录（必须包含"被否决的方案"）

##### 决策1：规则接口统一为 IRule + 短路求值模式

1. **最终决策**：所有规则（硬规则、启发式规则、用户排除规则）实现统一 `IRule` 接口，`Evaluate()` 返回 `RuleVerdict?`（null = 不匹配），规则引擎按优先级排序后首个非 null 即返回（短路）。

2. **核心权衡（Trade-off）**：
   - 牺牲：不能同时获得所有匹配规则的完整列表（如一个文件同时位于 Temp 且扩展名为 .log 时，只能看到最高优先级匹配）。
   - 换取：流式处理中每条文件的决策时间 O(规则数)，内存零分配（无 List 收集）。

3. **被否决的替代方案**：
   - **方案A**：每条规则独立返回分数，加权求和后阈值判断。**致命缺陷**：无法解释"为什么删除"——设计文档明确要求可解释性（语义标签 + 规则理由），加权评分对此是黑盒。
   - **方案B**：规则引擎输出所有匹配规则的列表，由决策引擎做最终选择。**致命缺陷**：每条文件需遍历全部规则收集结果，200 万文件场景下多出大量不必要的枚举开销。

##### 决策2：用户排除规则（P3）动态评估 vs. 编译时缓存

1. **最终决策**：用户排除规则不在 `CompileRules()` 中编译，而是在 `Evaluate()` 方法中动态检查。采用"在 P1/P2 检查后、P4 之前插入 P3 检查"的策略。

2. **核心权衡（Trade-off）**：
   - 牺牲：每次 `Evaluate()` 调用需额外做路径前缀匹配（相比编译时缓存每条规则多一次字符串比较）。
   - 换取：用户修改排除列表后无需重建规则列表，`Evaluate(file, config)` 的 `config` 参数即时生效，避免了"编译时配置快照"与"运行时配置"不一致的问题。

3. **被否决的替代方案**：
   - **方案A**：在 `CompileRules()` 时为每个排除项创建 IRule 实例。**致命缺陷**：规则在构造时捕获 `UserConfig` 快照，后续 `Evaluate(file, config)` 传入的 `config` 参数被忽略，导致用户修改排除列表后不生效。

##### 决策3：启发式规则不引入第三方 Glob 库

1. **最终决策**：自实现简化 Glob 匹配（支持 `*`、`?`、`**`），不做完整 POSIX glob 兼容。

2. **核心权衡（Trade-off）**：
   - 牺牲：不支持字符类 `[a-z]`、花括号扩展 `{a,b}` 等高级 glob 语法。
   - 换取：零外部依赖，匹配逻辑完全可控，`**` 递归通配符针对 Windows 路径分隔符 `\` 优化。

3. **被否决的替代方案**：
   - **方案A**：引入 `Glob.cs` (Microsoft.Extensions.FileSystemGlobbing)。**致命缺陷**：该库为 ASP.NET 设计，依赖 `Microsoft.Extensions.*` 生态，引入后增加约 2MB 程序集体积，且其内部为每次匹配分配 `PatternMatchingResult` 对象，流式处理场景下 GC 压力不可接受。
   - **方案B**：正则表达式替代 Glob。**致命缺陷**：用户编写的规则以 Glob 形式呈现，转正则会引入转义问题（如 Windows 路径中的 `\` 在正则中需写 `\\\\`），增加用户配置复杂度。

##### 决策4：硬规则编译在代码中（不可通过配置文件禁用）

1. **最终决策**：优先级 1/2/4 的硬规则使用 `static class` + `Lazy<IReadOnlyList<IRule>>` 编译在程序集中，`RuleEngine` 构造时直接引用。

2. **核心权衡（Trade-off）**：
   - 牺牲：修改硬规则需要重新编译部署，不能通过 JSON 热修改。
   - 换取：杜绝用户或恶意软件通过修改配置文件禁用系统保护规则的风险，满足设计文档第 8 章安全要求。

3. **被否决的替代方案**：
   - **方案A**：所有规则（含硬规则）均从 JSON 加载。**致命缺陷**：若 JSON 文件被意外删除或损坏，系统关键目录保护将完全失效，可能导致灾难性误删。

#### 二、 隐含前提与关键假设清单（风险预警）

1. **[假设1]**：Everything SDK 返回的 `FileItem.FilePath` 始终以 `\\?\` 前缀格式化。若 Everything SDK 版本或配置变更导致路径格式不一致，硬规则的路径前缀匹配将失效（因为比较路径硬编码了 `\\?\` 前缀处理）。

2. **[假设2]**：`FileItem.LastWriteTime` 始终为 UTC 时间。文件年龄计算（`DateTime.UtcNow - file.LastWriteTime`）依赖此假设。若 Everything 返回本地时间，年龄判断将偏差时区偏移量。

3. **[假设3]**：系统盘为 `C:` 盘。`BuiltInForbidRules` 和 `BuiltInAutoDeleteRules` 中硬编码了 `Environment.SystemDirectory` 推导系统盘符。若用户系统安装在 D: 盘，路径保护仍正确（因为使用 `SystemDriveRoot` 动态获取），但若用户同时有双系统且当前运行的系统盘非目标扫描盘，保护范围可能不完整。

4. **[假设4]**：`UserConfig.UserExcludedDirs` 中的路径为用户输入的普通格式（非 `\\?\`）。`EvaluateUserExclusions()` 中同时检查了带 `\\?\` 前缀和不带前缀的版本，但仅做 `TrimEnd('\\')` 规范化，未调用 `PathNormalizer` 做完整路径标准化。

5. **[假设5]**：启发式规则 JSON 文件采用 UTF-8 编码。`HeuristicRuleLoader` 使用 `File.ReadAllText()` 读取，默认检测 BOM 但无 BOM 时假定 UTF-8。

#### 三、 边界与范围界定（明确"不做什么"）

1. **[明确不做]**：规则编辑 UI。本阶段仅实现规则引擎核心逻辑和 JSON 配置文件格式，不包含内置规则编辑器界面（属于 C 阶段 UI 任务）。

2. **[明确不做]**：规则有效性校验工具。当前 `HeuristicRuleLoader` 仅做基础 JSON 反序列化，不做规则冲突检测（如两条规则有相同的 name 但不同 priority）、不做死规则检测（条件永远不会匹配的规则）、不做规则性能分析。

3. **[明确不做]**：远程规则同步。规则文件仅从本地目录加载，不涉及网络更新或企业策略推送。

4. **[明确不做]**：规则执行统计与性能监控。不记录每条规则的命中率、平均评估耗时等指标（可延后到 B5 或 D2 阶段）。

5. **[明确不做]**：自动化调度器的规则触发。设计文档明确 V3.7 不做自动化调度，本阶段规则引擎不暴露定时触发接口。

#### 四、 待确认事项（需要我人工介入的模糊点）

1. **[待确认项A]**：`BuiltInProtectRules` 中的 WSL 发行版路径使用了硬编码的 `\Packages\CanonicalGroupLimited` 和 `\AppData\Local\Docker\wsl`，这些路径基于 WSL2 默认安装位置。若用户自定义 WSL 发行版安装路径（通过 `wsl --import` 指定自定义位置），这些路径将不被保护。**需要确认**：是否需要读取 WSL 注册表配置来动态获取实际发行版路径？

2. **[待确认项B]**：`BuiltInAutoDeleteRules` 中 Windows Temp 文件阈值暂定为 7 天，Prefetch 为 30 天，SoftwareDistribution 为 30 天。这些阈值来自设计文档 6.2 节的示例值。**需要确认**：这些阈值是否足够保守？是否需要在用户设置中暴露为可配置项？

3. **[待确认项C]**：`DefaultHeuristicRules.json` 中的 `nuget_npm_cache` 规则仅匹配了 `\.nuget\` 和 `node_modules\.cache` 路径。pip、cargo、go module 等其他包管理器的缓存路径未包含。**需要确认**：是否需要补充更多包管理器缓存路径？

4. **[待确认项D]**：`RuleVerdict.Exclude`（用户排除）在枚举中定义为独立值（value=2），但决策引擎（B3 阶段实现）将其映射为何种 `final_action`？设计文档 6.3 节决策表中 EXCLUDE 和 PROTECT 都映射为 `protected`。**需要确认**：`Exclude` 和 `Protect` 在 UI 展示上是否需要区分（如"用户排除" vs "系统保护"）？

5. **[待确认项E]**：当前 Glob 实现中 `**` 匹配跨越多级目录，但未处理 `**` 出现在模式中间时的边界情况（如 `a\**\b\**\c`）。默认启发式规则未使用此类复杂模式，但用户自定义规则可能触发。**需要确认**：是否需要为复杂 `**` 模式编写更完善的单元测试覆盖？
