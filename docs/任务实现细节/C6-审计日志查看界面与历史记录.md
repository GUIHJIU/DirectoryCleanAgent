# 阶段性关键决策与细节控制文档 — C6 审计日志查看界面与历史记录

> 文档用途：供项目负责人复核与纠偏，不对最终用户展示。
> 生成日期：2026-06-24 | V3.7 | 状态：代码已完成，待人工复核

---

#### 一、 关键架构/设计决策记录（必须包含"被否决的方案"）

##### 决策1：UI 容器 — 独立模态窗口 + TabControl vs 嵌入主界面

1. **最终决策**：采用独立模态窗口 `HistoryWindow`，内部 `TabControl` 包含"审计日志"和"删除历史"两个 Tab。
2. **核心权衡（Trade-off）**：
   - 牺牲了主界面一键直达的便利性（需多点击一次按钮打开窗口）。
   - 换取了主界面空间不被低频功能侵蚀、子 ViewModel 生命周期独立、与 SettingsWindow 模态模式一致。
3. **被否决的替代方案**：
   - **嵌入 MainWindow 主内容区（与 C2 文件列表共用 Row 4）**：主内容区已由 C2 文件列表独占，切换会丢失文件列表上下文（已选中的分组/排序状态）。不采纳。
   - **在操作栏增加下拉面板（Popup）**：Popup 不适合承载 DataGrid 等复杂控件（焦点管理、虚拟化滚动均受限），且回滚确认对话框在 Popup 上层嵌套时体验差。不采纳。

##### 决策2：ViewModel 拆分策略 — 两个独立 VM vs 统一 VM

1. **最终决策**：`AuditLogViewModel` 和 `DeletionHistoryViewModel` 各自独立，由 `HistoryWindowViewModel` 作为简单容器聚合。
2. **核心权衡（Trade-off）**：
   - 额外增加了一个容器 VM 层（HistoryWindowViewModel），但两个子 VM 各自短小精悍（AuditLogVM ~220 行, DeletionHistoryVM ~370 行），职责清晰。
   - 回滚操作中的状态变更不影响审计日志 Tab（隔离性好）。
3. **被否决的替代方案**：
   - **一个统一的 HistoryViewModel**：两个数据源（AuditLog / DeletionRecord）、两种交互模式（只读浏览 / 写操作+回滚），合并会导致 500+ 行巨型 VM，违反单一职责。不采纳。

##### 决策3：DI 全量切换 — Mock → 真实层

1. **最终决策**：在 C6 阶段将 DI 容器从 Mock 实现一次性切换到真实层（`AddDataLayer` + `AddEverythingServices` + `AddRulesLayer` + `AddDecisionLayer` + `AddOperationsLayer`），为所有后续 UI 任务提供真实数据基础。
2. **核心权衡（Trade-off）**：
   - 切换后 C1/C2 的 Mock 仪表板数据消失（文件列表需依赖真实 Everything SDK 和 SQLite），首次运行若 Everything 未启动或数据库为空则界面显示空状态。
   - 换取了 C6 之后所有功能（审计/回滚/隔离区/设置等）均工作在真实数据上，无需再次重构 DI。
3. **被否决的替代方案**：
   - **仅注册 C6 需要的 IAuditLogRepository + IDeletionRecordRepository + IBackupManager**：会导致部分容器内同时存在 Mock 和真实实现（如 MockFileDecisionCacheRepository 与真实 FileDecisionCacheRepository），行为不一致且难以排查。不采纳。
   - **推迟到 C8 阶段统一切换**：C6 验证时仍使用空数据库/Mock，无法验证回滚功能与真实数据的交互正确性。不采纳。

##### 决策4：回滚确认流程 — 详情预览→确认→进度执行

1. **最终决策**：选中批次后先加载文件详情和 `VerifyHashesAsync()` 哈希校验，展示摘要后用户点"执行回滚"→弹出二次确认对话框（显示文件数/大小/校验失败数）→确认后带进度条执行。
2. **核心权衡（Trade-off）**：
   - 多了一次点击和一次确认，但回滚是高风险操作（可能覆盖现有文件），必须确保用户知情。
   - 哈希预检可能增加加载延迟（需扫描回收站/隔离区），但预先告知用户哪些文件不可恢复比回滚中途失败体验更好。

#### 二、 隐含前提与关键假设清单（风险预警）

1. **[假设1]**：假设 Everything SDK 在用户机器上已安装并运行（≥1.4.1.1000）。若 Everything 未运行，`AddEverythingServices` 注册的 `EverythingFileListProvider` 初始化可能失败，导致 FileListViewModel 构造函数抛出异常，主界面文件列表区域显示空状态。（风险等级：高）
2. **[假设2]**：假设 `%LocalAppData%\SpaceClear\data.db` 路径可写入。若该路径被组策略限制或磁盘满，`DatabaseInitializer.InitializeAsync()` 将失败，当前未在启动流程中对此做 fallback 处理。（风险等级：中）
3. **[假设3]**：假设 `DeletionRecord` 表中的记录均由 B4 `OperationExecutor` 在删除时写入。若用户在 C6 之前运行过真实删除操作（非 Mock），表中已有记录；若从未真实删除，批次列表为空（UI 展示空状态提示）。（风险等级：低）
4. **[假设4]**：假设 `IBackupManager.RollbackAsync` 的所有依赖项（`ILocalTombstoneRepository`, `ITombstoneCache`, `IQuarantineManager`, `ShellFileOperation`）在 DI 容器中均可正确解析。`ShellFileOperation` 为具体类，依赖 P/Invoke `SHFileOperationW`，若在非管理员权限下调用 Move 操作可能失败。（风险等级：中）
5. **[假设5]**：假设 `AuditLogRepository` 中的 `BatchWriteQueue<AuditLogEntry>` 在应用退出前能正确 flush。当前 `App.OnExit` 未调用 `IAuditLogRepository.FlushAsync()`，若有未刷新的审计日志可能丢失。（风险等级：中）

#### 三、 边界与范围界定（明确"不做什么"）

1. **[明确不做]**：本期不做审计日志的导出功能（HTML/CSV/JSON）。导出属于 C5 报告导出范围。
2. **[明确不做]**：本期不做删除记录的编辑、删除或手动标记为"不可回滚"。审计日志和删除记录均为不可变数据。
3. **[明确不做]**：本期不做跨用户的回滚操作。回收站按用户 SID 隔离，跨用户回滚需要管理员权限枚举其他用户的回收站，B5 已排除。
4. **[明确不做]**：本期不做审计日志的图表/统计分析（如操作频率趋势图、Top-N 用户等）。仅提供列表浏览。
5. **[明确不做]**：本期不做回滚操作的审计日志写入（不在 AuditLog 表追加回滚记录）。回滚结果通过 ILogger 记录，不写入 AuditLog。B5 已明确排除此范围。
6. **[明确不做]**：本期不做网络驱动器 / UNC 路径文件的回收站恢复（`$Recycle.Bin` 仅本地 NTFS 卷）。B5 已排除。
7. **[明确不做]**：本期不做 Postman 测试用例。C6 是 WPF 桌面应用，不包含 REST API。测试方案为准 xUnit 单元测试（C# 测试项目）+ 手动 UI 验收测试。

#### 四、 待确认事项（需要我人工介入的模糊点）

1. **[待确认项A]**：审计日志过滤栏的"操作类型"下拉列表包含了 11 种操作类型（SCAN_STARTED, FILE_DELETED 等），但这些类型名称来源于设计文档 4.2 节的示例。实际业务中使用的操作类型枚举尚未在代码中统一定义（`AuditLogEntry.OperationType` 为 `string` 类型，非 enum）。需要业务方确认最终的操作类型清单。
2. **[待确认项B]**：回滚确认对话框中显示了"释放空间（回退）"的金额/大小。当 `OperationResult.FreedBytes` 为 0 时（回滚未释放任何新空间，反而是恢复了占用），此字段可能误导用户。当前直接显示 `result.FreedBytes`，需确认语义。
3. **[待确认项C]**：`HistoryWindow` 的 Tab 切换时，非活动 Tab 的 DataGrid 是否会因虚拟化回收而丢失滚动位置？WPF TabControl 默认保留可视化树（未设置 `TabControl.TabStripPlacement` ），但若用户频繁切换 Tab，DataGrid 的 `VirtualizingStackPanel` 可能重建容器。当前未做滚动位置保存，需确认是否可接受。
4. **[待确认项D]**： `DeleteMethod` 枚举当前仅有 `RecycleBin` 和 `Permanent` 两个值，但 `BackupManager.RollbackAsync` 还支持从隔离区恢复。这意味着 `OperationBatchInfo.Method` 与删除实际执行方式之间可能存在语义差距。需确认是否为 `DeleteMethod` 增加 `Quarantine` 值（影响 `DeletionBatchDisplayItem.MethodText` 的映射）。

#### 五、 关键外部依赖锁定

*   **依赖项1**：`DirectoryCleanAgent.Operations` 项目 — 包含 `BackupManager`（回滚管理器）、`OperationExecutor`（操作执行器）、`QuarantineManager`（隔离区管理）。已在 C6 阶段添加项目引用。若 Operations 项目构建失败或接口变更，C6 的回滚功能直接阻塞。
*   **依赖项2**：`DirectoryCleanAgent.Data` 项目 — 包含 `AuditLogRepository`、`DeletionRecordRepository`、`DatabaseInitializer`（SQLite 连接工厂 + 建表）。已通过 `AddDataLayer` 注册。若 SQLite 文件损坏或磁盘空间不足，数据库初始化失败将阻塞启动。
*   **依赖项3**：`Microsoft.Data.Sqlite` NuGet 包 — SQLite 的 ADO.NET 提供程序，由 Data 项目依赖。已在本地 NuGet 缓存中可用。
*   **依赖项4**：`DirectoryCleanAgent.Everything` 项目 — 包含 `EverythingFileListProvider`（Everything SDK 流式适配器）。若 Everything 服务未运行或 SDK 版本不兼容，`IFileListProvider` 的所有调用将失败，文件列表视图显示空数据。
*   **依赖项5**：用户配置文件 `%LocalAppData%\SpaceClear\config.json` — `IConfigService.LoadAsync()` 依赖此文件。首次运行时（`IsFirstRun=true`）会进入配置向导，向导完成后才写入配置文件。
