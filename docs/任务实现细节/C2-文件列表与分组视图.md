# C2 文件列表与分组视图 — 阶段性关键决策与细节控制文档

---

## 一、 关键架构/设计决策记录

### 1. 分组树数据源：FileDecisionCache 而非 Everything 全量

- **最终决策**：分组树基于 `FileDecisionCache` 缓存构建，而非直接查询 Everything 全量结果。
- **核心权衡**：牺牲了全量文件浏览的覆盖度，换取可控的内存占用和分组语义的准确性。语义标签是规则引擎产出，Everything 不索引此信息，基于缓存分组是唯一有意义的分组方式。
- **被否决的替代方案**：
  - A. 每次分组切换向 Everything 发起查询并实时仲裁 → 致命缺陷：200万文件场景下每次实时裁决 + 分组的延迟远超可接受范围（秒级→分钟级）。
  - B. 将所有文件缓存到内存 → 致命缺陷：违反流式约束，200万文件内存 >1GB。

### 2. 排序策略：混合下推

- **最终决策**：可排序列（名称/大小/路径/日期/扩展名）下推至 Everything 原生排序；语义标签列在当前 UI 子集上内存排序。
- **核心权衡**：牺牲了排序行为的一致性（不同列的排序延迟不同），换取大数据量下的性能保证。
- **被否决的替代方案**：
  - A. 全列内存排序 → 致命缺陷：全量文件内存排序违背 200MB 内存限制。
  - B. 将语义标签回写 Everything 自定义属性 → 致命缺陷：需要 Everything 1.5+ 且要求写入 Everything 数据库，引入持久化污染风险。

### 3. "显示所有文件"模式的分离数据路径

- **最终决策**：全量模式下绕过规则引擎和决策引擎，直接查询 Everything。
- **核心权衡**：全量模式下的文件缺少 AI 标签和语义标签，UI 显示可能不完整；换取数据加载速度。
- **被否决的替代方案**：
  - A. 全量模式下仍走完整裁决流程 → 致命缺陷：全量文件实时裁决延迟不可接受，且语义分组对未命中规则文件无意义。

### 4. 增量刷新节流策略

- **最终决策**：500ms 节流窗口 + Timer 批量处理，非每事件即时刷新。
- **核心权衡**：牺牲了 <100ms 的即时性，换取 UI 线程不被高频事件卡顿。
- **被否决的替代方案**：
  - A. 每个 FileChanged 事件即时刷新 → 致命缺陷：Everything 3秒轮询可能批量触发数十个事件，逐条刷新导致 UI 冻结。

### 5. WPF 虚拟化策略

- **最终决策**：TreeView 和 DataGrid 均启用 `VirtualizingStackPanel` (Recycling 模式, `EnableRowVirtualization`)。
- **核心权衡**：牺牲了某些复杂容器模板的直接支持（如不同高度的 TreeViewItem），换取大数据量下的滚动性能。
- **被否决的替代方案**：
  - A. 使用 `ListBox` + `Expander` 自建分组视图 → 致命缺陷：缺少 TreeView 的原生层级展开/折叠支持和虚拟化。

## 二、 隐含前提与关键假设清单

1. **[假设1]**：假设 B1(Everything 适配器) 的 `IFileListProvider.EnumerateFilesAsync()` 实现正确支持 `EverythingQueryParams.SortType` 参数，排序参数会原封不动传递给 Everything IPC。
   - **若该假设不成立**：C2 的"显示所有文件"模式下列头排序将无法下推，需要额外的内存排序逻辑。

2. **[假设2]**：假设 B2(规则引擎) 和 B3(决策引擎) 在处理每个 `FileItem` 后，会通过 `IFileDecisionCacheRepository.Upsert()` 将 `FileDecisionCache` 写入 SQLite，确保缓存数据与 UI 分组树同步。
   - **若该假设不成立**：`LoadDataAsync()` 读取的缓存可能为空，分组树无法构建。

3. **[假设3]**：假设 `SemanticLabelLocalizer.LocalizeFromString()` 能够正确解析 `FileDecisionCache.SemanticCategory` 中存储的字符串格式（目前是中文标签文本？枚举名称？设计文档未明确），假设当前存储格式为枚举名称（如 "SystemTempFiles"）。
   - **若该假设不成立**：语义标签分组和图标显示可能全部回退到"未分类"。

4. **[假设4]**：假设 Everything 的 `MaxResults` 参数在 `IFileListProvider` 中得到正确支持，能够限制全量模式下的结果数量。当前硬编码为 10,000。
   - **若该假设不成立**：全量模式下可能返回数十万条结果，超过 WPF DataGrid 虚拟化可承载范围。

5. **[假设5]**：假设用户的文件系统中不存在超过 10,000 个可操作文件的极端场景（在分组视图下不需要 `MaxResults` 限制）。
   - **若该假设不成立**：分组树可能包含过多一级节点，TreeView 性能下降。

## 三、 边界与范围界定（明确"不做什么"）

1. **[明确不做]**：C2 不实现文件内容的预览（缩略图、文本预览等），这些属于 C3 或 C4 阶段。
2. **[明确不做]**：C2 不实现多选文件后的批量操作 UI（如批量删除确认弹窗），这些属于 C4 阶段。
3. **[明确不做]**：C2 不实现文件列表的导出功能（导出为 CSV/HTML），导出功能在 C5 阶段。
4. **[明确不做]**：C2 不实现搜索/过滤栏（如搜索文件名、按大小范围过滤），这些属于 C7 阶段。
5. **[明确不做]**：C2 不实现 AI 分析的实际调用，AI 按钮点击仅显示占位提示，实际 AI 集成在 B6 阶段。
6. **[明确不做]**：二级分组的 ComboBox 控件未在 MainWindow 操作栏中实现（主 ComboBox 仅切换一级分组），二级分组的 UI 交互将在 C7 阶段完善。
7. **[明确不做]**：C2 不修改 `FileDecisionCache` 数据模型，不为其新增 `Extension` 属性（当前通过 `System.IO.Path.GetExtension(FilePath)` 运行时计算）。

## 四、 待确认事项（需要人工介入的模糊点）

1. **[待确认项A]**：语义标签图标方案 — 当前使用 Unicode Emoji（🗑️🧹📥♻️📋📁🛡️❓📄），在 Win10/Win11 上渲染效果不一致（Win10 可能显示为单色符号）。若需要统一视觉效果，应替换为 Segoe MDL2 Assets 字体图标或自定义 SVG 路径。
   - **建议**：运行业务方确认目标受众的 Windows 版本，若 Win10 占比高则建议改用 Segoe MDL2 Assets。

2. **[待确认项B]**：二级分组的交互设计 — 当前假设二级分组通过另一个下拉框选择，但 MainWindow 操作栏未预留位置。实际是弹出式菜单、右键菜单还是独立下拉框？
   - **建议**：C2 阶段先实现 `SubGroupModes` 数据绑定和逻辑，UI 控件位置在 C7 阶段与搜索栏一起布局。

3. **[待确认项C]**："显示所有文件"模式下 `MaxResults` 硬编码为 10,000。实际应用中是否需要分页加载或滚动加载更多？
   - **建议**：现阶段 10,000 对 WPF DataGrid 虚拟化是可承载的，若后续发现性能问题可改为滚动分页。

4. **[待确认项D]**：AI 状态的触发机制 — 当前 AI 分析触发由用户点击"请求AI分析"按钮（右键菜单或行操作列），还是应在扫描完成后自动批量发起？设计文档对此未明确。
   - **建议**：C2 阶段先实现手动触发，自动批量分析在 B6 阶段决策。

5. **[待确认项E]**：Mock 数据的语义标签存储格式 — 当前 Mock 数据使用枚举名称（如 "SystemTempFiles"）。真实 B2 规则引擎产出的 `SemanticCategory` 字符串格式需要与 `SemanticLabelLocalizer` 的解析逻辑一致。
   - **建议**：B2 产出格式应为枚举名称字符串（如 `SemanticCategory.SystemTempFiles.ToString()`），确保 `LocalizeFromString` 可正确解析。
