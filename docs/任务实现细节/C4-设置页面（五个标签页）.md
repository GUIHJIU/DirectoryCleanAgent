## 一、关键架构/设计决策记录

### 1. 克隆编辑模式（Clone Pattern）

- **最终决策**: SettingsViewModel 内部维护 `_editConfig = _configService.Current.Clone()` 深拷贝副本。所有 UI 绑定均指向此副本的属性，仅在用户点击"确定"或"应用"时，将副本数据同步回 `_configService.Current` 并调用 `SaveAsync()` 持久化。取消时直接丢弃副本。

- **核心权衡（Trade-off）**: 内存中同时存在两份 UserConfig（原始 + 副本），增加了约 2KB 内存开销。换取的是：
  - 取消操作零副作用（无需回滚复杂的状态变更）
  - 防止部分持久化（不会出现"改了3个字段但只保存了2个"的情况）
  - 多标签页间的编辑互不干扰

- **被否决的替代方案**:
  1. **直接绑定 IConfigService.Current**：放弃理由是——取消操作需要手动回滚每个字段，容易遗漏，且与"实时生效的语言/主题"产生复杂的状态冲突。
  2. **每个标签页独立保存**：放弃理由是——与设计文档"统一保存"的交互预期不一致，且增加"部分保存失败"的复杂性。

### 2. 主题/语言实时生效 vs 其余设置延迟生效

- **最终决策**: 语言和主题在用户切换 ComboBox 选项时立即生效（调用 `ILocalizationService.SetLanguage()` / `ApplyTheme()`），不等待"确定/应用"按钮。其余设置（扫描模式、AI 配置、排除目录等）需确认后生效。

- **核心权衡（Trade-off）**: 不一致的 UX 行为（部分立即生效、部分延迟生效）可能让用户困惑。换取的收益：
  - 用户切换语言/主题时可立即预览效果，无需反复开关设置窗口
  - 其余设置保持原子提交语义，防止误操作

- **被否决的替代方案**:
  1. **所有设置都立即生效**：放弃理由是——删除方式从"回收站"误改为"永久删除"并立即生效会导致安全风险。
  2. **所有设置都延迟生效**：放弃理由是——语言切换用户需要"所见即所得"的反馈，若需点确定才能看到效果，体验极差。

### 3. XAML TabControl vs Code-Behind 动态 UI 构建

- **最终决策**: 设置窗口使用 XAML 声明式 TabControl 定义五个标签页布局，而非像 SetupWizardWindow 那样在代码后台构建 UI。

- **核心权衡（Trade-off）**: XAML 在运行时无法动态增删标签页（如未来需求变更）。换取的收益：
  - 声明式布局可维护性远高于代码构建
  - 支持 Visual Studio 设计器预览
  - 数据绑定路径可静态检查

- **被否决的替代方案**:
  1. **代码后台构建 UI（继承 SetupWizardWindow 模式）**：放弃理由是——设置页面有 5 个固定标签页、30+ 控件，代码构建会形成 500+ 行的 BuildTabN 方法，可读性极差。
  2. **使用第三方 TabControl (如 ModernWpf)**：放弃理由是——引入额外依赖，且项目已确定使用原生 WPF。

### 4. PasswordBox 不支持绑定的变通方案

- **最终决策**: API Key 使用 WPF PasswordBox 控件（安全输入），在代码后台通过 `PasswordChanged` 事件手动同步到 ViewModel 的 `EditAIApiKey` 属性。

- **核心权衡（Trade-off）**: 失去了 MVVM 纯绑定的一致性，代码后台需要处理 PasswordBox 事件。换取的收益：
  - PasswordBox 使用 `SecureString` 内部存储，降低内存转储泄漏风险
  - 符合安全最佳实践（敏感字段不应使用明文 TextBox）

- **被否决的替代方案**:
  1. **使用 TextBox 直接绑定**：放弃理由是——安全审计不通过，API Key 会以明文形式出现在内存转储中。
  2. **自定义附加属性实现 PasswordBox 绑定**：放弃理由是——增加了不必要的复杂度，且自定义附加属性可能绕过 `SecureString` 安全机制。

### 5. COM 文件夹选择对话框 vs WinForms FolderBrowserDialog

- **最终决策**: 暂不实现原生文件夹选择对话框，"浏览"按钮提供提示说明，用户可直接在文本框中输入或粘贴路径。

- **核心权衡（Trade-off）**: 用户体验略有下降（需手动输入路径而非图形化选择）。换取的收益：
  - 避免了引入 Windows Forms 依赖导致的 `UserControl`/`ProgressBar`/`Timer` 类型歧义（WPF 项目添加 `<UseWindowsForms>true</UseWindowsForms>` 会导致全项目范围的命名空间冲突）
  - 数据存储路径有合理默认值（`%LocalAppData%\SpaceClear`），用户修改频率极低

- **被否决的替代方案**:
  1. **添加 `<UseWindowsForms>true</UseWindowsForms>`**：放弃理由是——导致 DashboardCard、FileListView、WarningBanner、SetupWizardWindow 等 5 个现有文件出现 `UserControl`/`ProgressBar` 类型歧义错误，需逐文件添加命名空间别名，侵入范围过大。
  2. **使用 COM IFileOpenDialog P/Invoke**：放弃理由是——COM 互操作在 .NET 8 中需要额外平台兼容性验证，`dynamic` 反射调用可靠性不足，且 CLSID 可能在不同 Windows 版本间有差异。

## 二、隐含前提与关键假设清单

1. **[假设1]**: 假设 `IConfigService.Current` 的 `Clone()` 方法（`MemberwiseClone() + 三个 List<string> 深拷贝`）覆盖了所有可变字段。若未来 UserConfig 新增引用类型集合字段而未更新 `Clone()`，设置编辑将产生"假取消"（取消后原始配置仍被修改）。

2. **[假设2]**: 假设系统盘 NTFS 格式为真（`DriveInfo.GetDrives()` 能返回至少一个 `DriveType == Fixed && DriveFormat == "NTFS"` 的卷）。若用户系统盘为 ReFS 或 FAT32（非 Windows 11 默认配置），卷列表将为空，UI 会显示红色警告文本但不会阻断设置操作。

3. **[假设3]**: 假设 `ILocalizationService.GetString()` 的回退机制正常（缺失键返回 `[[key]]` 占位符）。本次实现约 25 个新本地化键尚未添加到 `.resx` 资源文件，UI 将以 `[[Settings.Xxx.Yyy]]` 格式显示。本地化资源文件补充应在 C4 收尾阶段完成。

4. **[假设4]**: 假设 `Application.Current` 在 SettingsWindow 生命周期内不为 null。若在单元测试环境中 `Application.Current` 为 null，`ApplyThemeImmediately()` 和 `OnLanguageChanged()` 已添加 null 守卫，不会崩溃。

5. **[假设5]**: 假设用户以管理员权限运行应用时才能看到全部设置功能。非管理员只读模式的按钮禁用由 MainViewModel 层面的 `IsAdmin` 属性控制，设置窗口本身不检测权限状态。

## 三、边界与范围界定（明确"不做什么"）

1. **[明确不做]**: 本期不实现 **文件夹原生选择对话框**（BrowseForFolder）。数据存储路径仅支持手动输入/粘贴。若后续需要，可在引入 `Microsoft.Win32.OpenFolderDialog` NuGet 包后实现。

2. **[明确不做]**: 本期不实现 **API Key 加密存储**。`EditAIApiKey` 以明文形式存储在 `UserConfig.AIApiKey` 中（JSON 明文序列化）。加密存储方案（DPAPI / Windows Credential Manager）延后至 B7 或 C5 阶段。

3. **[明确不做]**: 本期不实现 **设置导入/导出**功能。用户无法通过 JSON 文件批量迁移设置到另一台机器。

4. **[明确不做]**: 本期不实现 **设置变更审计日志**的写入。AuditLog 表已存在（A3 阶段实现），但设置保存时不写入审计记录。延后至 C5 阶段统一处理。

5. **[明确不做]**: Tab 4（自动化清理）的完整实现。仅展示"即将推出"占位文本，不包含定时任务配置（Cron 表达式、任务列表等）。这部分属于设计文档明确标注的"延后至后续里程碑"功能。

6. **[明确不做]**: 本期不做 **设置搜索/过滤**功能。若未来设置项超过 50 项，可能需要全局搜索功能，当前 5 标签页约 30 项设置无需此功能。

7. **[明确不做]**: 本期不做 **设置变更冲突检测**。若用户在主界面打开设置窗口的同时，规则热加载或 AI 响应修改了 `IConfigService.Current`（如 `RuleCacheVersion` 递增），设置窗口不会检测此冲突，保存时可能覆盖外部变更。此场景概率极低（设置窗口为模态对话框，通常不会同时发生外部修改）。
