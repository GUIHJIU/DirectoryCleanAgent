> 生成日期：2026-06-23 | 对应任务：| B3 | 决策引擎（仲裁逻辑+快照生成） | B2 |

## 一、 关键架构/设计决策记录

### 1. 决策表实现方式：纯静态函数 vs 策略模式

- **最终决策**：采用纯静态函数 `ComputeFinalAction` + 两个子函数 `ArbitrateSuggestDeleteWithAi` / `ArbitrateKeepWithAi`，所有仲裁逻辑为无副作用纯函数。
- **核心权衡（Trade-off）**：牺牲了策略模式的可插拔性（新增 AI 标签类型需修改源码），换取了：
  - 零依赖的单元测试能力（无需 Mock，直接输入/输出断言）
  - 编译器可完全内联优化
  - 代码即文档（决策矩阵直接映射为 switch/case 分支）
- **被否决的替代方案**：
  1. **策略模式 + DI 注入规则链**：被否决——B3 仅涉及 4 种情况的决策表，策略模式的抽象成本远超收益；且设计文档明确决策表是固定规范，不需要运行时扩展。
  2. **JSON/XML 配置驱动决策表**：被否决——将决策表外置为配置文件会导致配置与代码行为不同步，且 AI 标签阈值（0.6/0.7/0.8）与业务语义强耦合，配置文件无法表达"高信任度时对 risky 标签更敏感"这类隐含逻辑。

### 2. 并发哈希方案：Parallel.ForEachAsync vs SemaphoreSlim + Task.Run

- **最终决策**：使用 `Parallel.ForEachAsync(MaxDegreeOfParallelism = Environment.ProcessorCount)`。
- **核心权衡（Trade-off）**：牺牲了 .NET Standard 2.0 兼容性（`Parallel.ForEachAsync` 为 .NET 6+ 新增），换取了：
  - 内置的 `CancellationToken` 传播与优雅取消
  - 不需要手动管理 `SemaphoreSlim` 的获取/释放配对
  - `ParallelOptions` 统一配置，代码更简洁
- **被否决的替代方案**：
  1. **SemaphoreSlim + 手动 Task.Run**：被否决——需要手动管理信号量生命周期，在取消场景下容易产生信号量泄漏（wait 后释放不对称导致的死锁风险）。
  2. **PLINQ with AsParallel()**：被否决——PLINQ 不支持真正的异步 I/O（`FileStream.ReadAsync`），对 CPU-bound 友好但对 I/O-bound 的哈希计算无法充分利用 IOCP。

### 3. DecisionSnapshot 的 JSON 序列化位置：Core 层 vs Decision 层

- **最终决策**：`ToCompactJson()` 方法直接添加到 `DecisionSnapshot` record（Core 层）中。
- **核心权衡（Trade-off）**：在 Core 层的 DTO 中引入了 `System.Text.Json` 依赖，打破了"DTO 纯数据结构"的边界。换取的是：
  - B3 产出 JSON 字符串，B4 直接取用，无需跨层适配
  - `DecisionSnapshotJson` 的数据内容与其定义同源，避免"DTO 在 Core，序列化工具在 Decision"导致的版本不同步
- **被否决的替代方案**：
  1. **独立 `DecisionSnapshotJsonSerializer` 工具类在 Decision 层**：被否决——B4（Operation Executor）引用 Core 即可，不应被迫引用 Decision 层仅为一个序列化工具方法。
  2. **在 B4 中读取 `Entries` 再自行序列化**：被否决——违背 B3 任务要求"序列化 JSON 写入 DeletionRecord"的职责边界。

## 二、 隐含前提与关键假设清单

1. **[假设 1]**：假设 `IConfigService.Current` 的读取是线程安全且一致的——`DecideAndSnapshotAsync` 在方法开头读取 `config.AIEnabled` 和 `config.AITrustLevel` 并捕获为局部变量，后续若配置被其他线程修改，本次快照内部不感知。**风险**：若 `UserConfig` 的某个属性在读取过程中被并发修改（如 List 的 Add 操作），可能导致不一致。当前假设配置写入发生在 UI 线程，读取发生在后台线程——不重叠。

2. **[假设 2]**：假设 `FileShare.Read` 在 SHA-256 计算期间不会遇到写锁——若另一个进程以独占方式打开同一文件，`FileStream` 构造会抛出 `IOException`，被单文件异常处理跳过。**风险**：高频读写的日志文件（如 SQLite WAL）可能频繁触发此场景。

3. **[假设 3]**：假设 `Environment.ProcessorCount` 并发度的哈希计算不会导致磁盘 I/O 饱和——在 SSD 上通常不是问题，但在 HDD 上可能需要限制为 `Math.Min(ProcessorCount, 4)`。当前未加入 HDD/SSD 检测逻辑。

4. **[假设 4]**：假设 AI 标签字符串为 `safe`/`unknown`/`risky`（三种值），使用大小写不敏感比较。若未来 AI 适配器返回其他标签值（如 `suspicious`），将被当作未识别标签，落入 `ArbitrateSuggestDeleteWithAi` 的 `_ => ManualReview` 兜底。

## 三、 边界与范围界定

1. **[明确不做]**：本期不实现 GZip 压缩与 Base64 编码——`DecisionSnapshot.ToCompactJson()` 仅输出原始紧凑 JSON。超 4KB 时的 "GZ:" 前缀 + GZip + Base64 由数据层（B4 或 A3 的 `DeletionRecord` 写入逻辑）负责。

2. **[明确不做]**：本期不实现哈希计算的磁盘 I/O 限流（throttling）——不根据磁盘类型（SSD/HDD）动态调整并发度。

3. **[明确不做]**：本期不实现决策结果的审计日志写入——决策日志仅通过 `ILogger` 输出到日志系统，不写入 `AuditLog` 表。审计日志写入属于 B4（操作执行器）职责。

4. **[明确不做]**：本期不实现 "decision_reason" 文本字段——设计文档 6.1 节提到输出包含 `decision_reason`，但 `FinalAction` 枚举未定义该字段，`DeleteSnapshotEntry` 也未预留该字段。当前由调用方根据 `FinalAction` 枚举值自行生成 UI 显示文本。

5. **[明确不做]**：本期不实现 `UserConfig.AIEnabled`/`AITrustLevel` 变更时的事件订阅（实时刷新）——`DecideAndSnapshotAsync` 仅在方法入口捕获一次配置。UI 刷新场景由 `Arbitrate` 方法单独处理（调用方传入最新的 `aiEnabled` 和 `aiTrustLevel`）。

## 四、 待确认事项

1. **[待确认项 A]**：`Parallel.ForEachAsync` 的 `MaxDegreeOfParallelism = Environment.ProcessorCount` 是否适合目标用户的硬件环境？——开发人员通常使用 SSD 和高核心数 CPU，但若目标用户包含低配设备（4 核 HDD），可能需要调整为 `Math.Min(Environment.ProcessorCount, 4)` 或通过配置文件覆盖。

2. **[待确认项 B]**：单文件哈希失败时的行为——当前设计为"记录警告 + 跳过该文件 + 继续处理其他文件"。是否需要向 UI 层返回失败文件列表？当前 `DecisionSnapshot` 不含 `FailedFiles` 字段，UI 层只能通过日志知道有文件被跳过。

3. **[待确认项 C]**：`DecideAndSnapshotAsync` 中的 `progress` 参数使用 `IProgress<int>`（仅上报已完成数量），是否需要更丰富的 `IProgress<OperationProgress>`（含阶段、当前文件路径等）？——设计文档 7.2 节提到 `OperationProgress` 含 `Phase.Hashing` 阶段，但接口当前仅定义 `IProgress<int>`。如果 UI 需要显示当前正在哈希的文件名，需要修改接口签名。

4. **[待确认项 D]**：AI 标签 `safe`/`unknown`/`risky` 的确切命名规范——当前使用大小写不敏感比较（`StringComparison.OrdinalIgnoreCase`）。若 AI 适配器（B7）使用不同的命名（如 `benign`/`uncertain`/`dangerous`），决策表将全部回退到 ManualReview。需与 B7 AI 顾问对齐标签枚举。
