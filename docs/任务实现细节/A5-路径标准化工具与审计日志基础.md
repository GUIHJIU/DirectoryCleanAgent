## 一、 关键架构/设计决策记录

### 决策 1：PathNormalizer 定位 — 静态工具类（而非 DI 注册的接口+实现）

- **最终决策**：`PathNormalizer` 为 `static class`，所有方法为纯函数，零外部依赖，不注册到 DI 容器。调用方直接 `PathNormalizer.Normalize(path)` 使用。
- **核心权衡**：牺牲了单元测试中 Mock 替换的可能性（无法注入不同的路径标准化策略），换取零分配、零接口抽象开销和极简调用链。路径标准化是数学意义上的纯函数——相同输入永远产生相同输出，不存在"不同环境需要不同策略"的场景，因此 DI 带来的灵活性在此处仅有理论意义而无实际价值。
- **被否决的替代方案**：
  - *方案 A*：`IPathNormalizer` 接口 + `PathNormalizer` 实现，通过 DI 注入。**致命缺陷**：路径标准化在 Everything 适配器、规则引擎、操作执行器等所有模块的**热路径**上被高频调用（扫描阶段每个文件至少调用一次），每层多一次接口虚调用分配，200 万文件场景下累积开销不可忽略，且接口抽象无实际替换场景。
  - *方案 B*：依赖 `System.IO.Path` 的 `GetFullPath` 自动处理。**致命缺陷**：`Path.GetFullPath` 在 .NET 8 下默认不添加 `\\?\` 前缀，且会对路径进行规范化（删除尾部空格和点号），这与设计文档"`\\?\` 前缀下保留原始路径"的要求矛盾。

### 决策 2：Validate 方法跳过扩展路径前缀后再检查非法字符

- **最终决策**：`Validate` 内部检测 `\\?\`、`\\?\UNC\`、`\\.\` 三种前缀，跳过前缀部分后，仅对实际路径内容进行非法字符检查。
- **核心权衡**：增加了 `Validate` 方法 4 行前缀检测逻辑，换取了对所有路径格式（含扩展路径）的准确校验。路径前缀中的 `?` 和 `.` 是设备命名空间语法的一部分，不应被误判为文件名非法字符。
- **被否决的替代方案**：
  - *方案 A*：在 switch 中为 `?` 增加"若位于 `\\` 之后的第 3 字符则放行"的复杂条件。**致命缺陷**：实现脆弱，若未来 Windows 引入新的扩展路径前缀变体（如 `\\?\GlobalRoot\`），则需修改校验逻辑，违反开闭原则。跳过前缀的方案对新变体天然兼容。

### 决策 3：审计日志模块本阶段零新增 — 前序阶段已超额交付

- **最终决策**：A5 不再新增任何审计日志相关代码。`IAuditLogRepository`（仅暴露 `Insert`/`QueryAsync`/`FlushAsync`）、`AuditLogRepository`（批量写入队列集成）、`AuditLogEntry` DTO、建表 DDL、3 个查询索引、6 条测试均已在 A3 阶段完成，且接口契约在编译期保证了"无 Update/Delete"的不可修改性。
- **核心权衡**：A5 的审计日志 DoD 为"仅暴露 Insert 方法；基础写入"——当前状态已超过此标准（额外提供了参数化分页查询、批量写入队列集成、多条件筛选），无需为"满足 DoD"而新增冗余代码。
- **被否决的替代方案**：
  - *方案 A*：在 A5 中新建一个 `AuditLogBasicWriter` 包装类。**致命缺陷**：增加无意义的间接层，`AuditLogRepository.Insert()` 已经是"基础写入"的最简实现。包装类会造成两种写入路径共存，反而增加维护负担和混淆。

## 二、 隐含前提与关键假设清单

1. **[假设 1]**：所有调用方传给 `PathNormalizer.Normalize()` 的是**有效的 Windows 文件系统路径**。`Normalize` 不做语义校验（如检查 C: 盘是否真实存在），只做格式转换 + 非法字符检查。若调用方传入格式合法但语义无效的路径（如 `Q:\nonexistent`），`Normalize` 会正常添加前缀而不报错——文件操作的失败由调用方在使用路径时捕获。
2. **[假设 2]**：`Validate` 方法的非法字符集基于 NTFS 限制。假设所有目标卷都是 NTFS（设计文档 1.4 节强制要求）。若未来支持 ReFS 或其他文件系统，非法字符集可能需要调整（ReFS 允许部分 NTFS 禁止的字符）。
3. **[假设 3]**：路径中不会出现 `\\?\` 前缀嵌套的情况（如 `\\?\\\?\C:\foo`）。`Normalize` 的幂等检查仅判断路径是否以 `\\?\` 开头，不处理嵌套前缀的异常输入。
4. **[假设 4]**：`Denormalize` 还原 `\\?\UNC\` 为 `\\` 后，调用方使用还原后的路径进行 UI 展示。还原后的 UNC 路径格式为 `\\server\share\...`，是用户可读的标准 UNC 格式，不包含任何转义字符。

## 三、 边界与范围界定

1. **[明确不做]**：本期不实现 `PathNormalizer` 与 `System.IO.Path` 的全面适配层（如 `Combine`、`GetDirectoryName`、`GetFileName` 等扩展方法）。调用方如需拼接路径，应使用 `Path.Combine` 后再调用 `Normalize`，或直接使用 `\\?\` 前缀的路径调用 .NET 原生 API（.NET 8 已原生支持 `\\?\` 路径）。若后续发现需要高频使用的组合操作，可在 `PathNormalizer` 中增加便捷方法。
2. **[明确不做]**：本期不实现路径的符号链接/交接点（Junction）解析。`Normalize` 不调用 `GetFinalPathNameByHandle` 或类似 API 解析符号链接的真实目标路径。
3. **[明确不做]**：本期不实现卷 GUID 路径（如 `\\?\Volume{guid}\...`）的生成与解析。此格式在墓碑匹配中可能涉及，但属于 B1 流式适配器阶段的职责。
4. **[明确不做]**：本期不实现路径大小写规范化（如 `c:\windows` → `C:\Windows`）。`Normalize` 保留原始大小写，因为 Windows 文件系统不区分大小写，且 Everything SDK 返回的路径已保持系统中的实际大小写。
5. **[明确不做]**：审计日志的写入量统计与告警（如单日超过 10 万条时触发警告）。此功能属于 C6 审计日志查看界面阶段。

## 四、 待确认事项

1. **[待确认项 A]**：`Validate` 方法在检测到第一个非法字符后立即抛出异常，不提供"收集所有非法字符后一次性报告"的模式。当前设计偏向快速失败，若用户需要完整的路径问题报告（如 UI 中实时高亮所有非法字符），需在调用层实现。
2. **[待确认项 B]**：`\\?\` 前缀路径在持久化到 SQLite 时的处理策略。当前约定是数据库存储原始路径（不带 `\\?\` 前缀），仅在调用 Windows API 时临时添加前缀。此约定未写入代码注释——依赖各模块开发者的理解一致性。若需要，可在 `PathNormalizer` 中补充 `NormalizeForStorage` / `NormalizeForApi` 语义化方法。
3. **[待确认项 C]**：`Normalize` 对 `C:` 格式的路径（驱动器号+冒号，无反斜杠）添加了 `\\?\` 前缀，但此格式的语义是"当前目录下的 C 盘相对路径"，而非" C 盘根目录"。当前行为与设计文档一致（只处理格式，不处理语义），但若调用方误用此格式，可能导致非预期的文件操作。
