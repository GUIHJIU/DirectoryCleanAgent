## 一、关键架构/设计决策记录

### 决策 1：测试分层策略 — 真实 Everything vs 纯文件系统

- **最终决策**：将 68 个测试用例分为两类：
  - **真实 Everything 测试（17 个）**：标记 `[Trait("Requires", "Everything")]`，使用 `EverythingTestHelper.SkipIfUnavailable()` 自动跳过
  - **纯文件系统集成测试（51 个）**：使用 `TestFileGenerator` + `IntegrationTestBase` 创建测试文件，通过真实 RuleEngine/DecisionEngine/OperationExecutor 实例验证管道
- **核心权衡（Trade-off）**：牺牲了"全真实环境"的完整性，换取了 CI 环境的可运行性（无需在 CI 机器上安装 Everything）
- **被否决的替代方案**：
  1. 全部使用真实 Everything — 致命缺陷：CI 环境无 Everything，导致全部 68 个测试被跳过，集成测试形同虚设
  2. 全部使用 Mock — 致命缺陷：无法验证跨层交互的真实行为（如 SHA-256 哈希、文件删除、SQLite 写入）

### 决策 2：测试类继承 IntegrationTestBase 而非独立管理资源

- **最终决策**：所有 11 个测试类继承 `IntegrationTestBase`，复用其 `TestRoot`、`CreateTestFile`、`ComputeSha256`、`UpdateConfig` 等辅助方法
- **核心权衡（Trade-off）**：牺牲了测试类构造函数的自由参数设计，换取了统一的临时目录管理和清理语义
- **被否决的替代方案**：
  1. 每个测试类自行管理临时目录 — 致命缺陷：大量重复代码，清理逻辑不一致，容易遗漏导致磁盘残留
  2. 使用 xUnit Collection Fixture 共享目录 — 致命缺陷：测试间文件相互干扰，违反隔离原则

### 决策 3：非管理员模式通过 Mock 验证而非真实非管理员进程

- **最终决策**：通过 `AppStateService.IsAdmin = false` + `AppMode.ReadOnly` 验证 UI 约束逻辑，不要求真实非管理员进程
- **核心权衡（Trade-off）**：牺牲了对 `WindowsPrincipal.IsInRole` 的真实调用验证，换取了测试的确定性和环境无关性
- **被否决的替代方案**：
  1. 使用 `runas /trustlevel` 启动测试进程 — 致命缺陷：需要 UAC 弹窗交互，无法在无人值守 CI 中运行
  2. 使用 `WindowsIdentity` 令牌模拟 — 致命缺陷：需要 SeImpersonatePrivilege 权限，CI 环境通常不具备

### 决策 4：索引未就绪通过模拟 CancellationToken 超时验证

- **最终决策**：通过 `CancellationTokenSource` 超时机制模拟索引未就绪场景，验证取消传播和超时处理逻辑
- **核心权衡（Trade-off）**：牺牲了对真实 Everything 索引构建过程的直接测试，换取了可控的、可重复的测试条件
- **被否决的替代方案**：
  1. 在测试前触发 Everything 强制重建索引 — 致命缺陷：重建索引耗时不可控（几分钟到数小时），且影响用户正常使用 Everything

### 决策 5：数据层使用真实 SQLite 而非 Mock

- **最终决策**：DeleteExecuteIntegrationTests 和 RollbackIntegrationTests 使用真实 `DatabaseInitializer` + `LocalTombstoneRepository` + `DeletionRecordRepository` + `AuditLogRepository`，在临时数据库上验证写入
- **核心权衡（Trade-off）**：牺牲了纯单元测试的速度优势，换取了端到端的数据完整性验证（墓碑、删除记录、审计日志）
- **被否决的替代方案**：
  1. 使用 SQLite 内存模式 — 致命缺陷：内存模式不支持 WAL，WAL 相关 PRAGMA 设置失败导致初始化异常
  2. 全部 Mock 仓储层 — 致命缺陷：无法验证批量写入队列延迟刷新（500ms/200条），可能遗漏数据丢失风险

## 二、隐含前提与关键假设清单

1. **[假设 1]**：假设 D1.5 搭建的 7 个基础设施类（IntegrationTestBase、TestFileGenerator、QuarantineSimulator、RecycleBinSimulator、EverythingTestHelper、PerformanceTestBase、TestConfigBuilder）API 稳定，不会有破坏性变更。若不成立，D2 全部 11 个测试类需要同步修改。
2. **[假设 2]**：假设真实 Everything 测试在开发机上运行（已安装 Everything 1.4.1+），CI 环境仅运行纯文件系统测试（`--filter "Requires!=Everything"`）。若不成立，需额外在 CI 机器上部署 Everything。
3. **[假设 3]**：假设 `OperationExecutor` 的 `ShellFileOperation` 在测试环境中可正常调用 `IFileOperation` COM 接口。若不成立，需要管理员权限运行测试或使用 Mock 替代。
4. **[假设 4]**：假设 `QuarantineManager.ConstructQuarantineFileName` 和 `ParseQuarantineFileName` 为 public static 方法，可被测试直接调用。若不成立（若为 internal），需要通过 `[InternalsVisibleTo]` 暴露。
5. **[假设 5]**：假设 SQLite 批量写入队列的 500ms 刷新延迟在测试中通过 `Task.Delay(800)` 等待足够覆盖。若不成立（高负载 CI 环境可能更慢），需增加等待时间或改用轮询检查。
6. **[假设 6]**：假设 `TestVolumeGenerator` 工具已生成测试卷但 D2 不使用（D2 使用 `TestFileGenerator` 生成小批量文件，200 万文件测试推迟到 D3）。若不成立，D2 中的 `GenerateForRuleEngine` 规模需要调整。

## 三、边界与范围界定

### 明确排除

1. **[明确不做]** 本期不做 200 万文件大规模性能测试（推迟到 D3 性能测试阶段）
2. **[明确不做]** 本期不做真实 UI 自动化测试（WPF 窗口启动、鼠标点击等），仅测试无窗口的逻辑层集成
3. **[明确不做]** 本期不做真实 AI API 调用集成测试（AI 层集成测试在 D1 单元测试中已通过 HttpClient Mock 覆盖）
4. **[明确不做]** 本期不做跨卷隔离区回滚测试（需要多分区环境，推迟到 D3 手动测试）
5. **[明确不做]** 本期不做真实非管理员进程测试（需要 UAC 环境，D2 采用 Mock 验证，真实验证在 D4 安全验证清单中覆盖）
6. **[明确不做]** 本期不修改现有 34 个单元测试文件（保持向后兼容，遵循 D1.5 中"不修改现有测试"的约束）
7. **[明确不做]** 本期不做 CI/CD 流水线集成（测试仅通过命令行验证，流水线配置推迟到 D5）

## 四、待确认事项

1. **[待确认项 A]**：`QuarantineManager.ConstructQuarantineFileName` 和 `ParseQuarantineFileName` 方法的访问修饰符。我在测试中以 `public static` 调用，若实际为 `internal`，需要在 `DirectoryCleanAgent.Operations.csproj` 中添加 `<InternalsVisibleTo Include="DirectoryCleanAgent.Tests" />`。
2. **[待确认项 B]**：`OperationExecutor.ExecuteAsync` 在传入已取消的 `CancellationToken` 时的具体行为。我在测试中覆盖了"抛出 OperationCanceledException"和"返回 IsCancelled=true 的 OperationResult"两种情况，实际行为取决于实现，需要确认。
3. **[待确认项 C]**：`DecisionSnapshot.ToCompactJson()` 方法是否存在。我在 `DecisionSnapshotIntegrationTests.cs` 的 IT-SNAP-06 中调用了此方法，若实际名称不同需要修正。
4. **[待确认项 D]**：`ILocalTombstoneRepository.DeleteByOperationIdAsync(string operationId)` 方法是否存在。我在回滚测试中调用了此方法清除墓碑，若不存在需要通过 `IConnectionFactory` 直接执行 SQL。
5. **[待确认项 E]**：`AppStateService` 的 `IsAdmin` 属性是否有 setter。我假设有 public setter（或 internal setter + InternalsVisibleTo），若没有需要通过反射设置。
6. **[待确认项 F]**：`QuarantineManager` 的 `EnforceCapacityAsync` 方法签名。我在测试中假设签名为 `Task EnforceCapacityAsync(long, CancellationToken)`，需要与实际实现核对。
