# D1-单元测试（核心逻辑）—— 阶段性关键决策与细节控制文档

## 一、关键架构/设计决策记录

### 1. B7 AI 内部类测试策略

- **最终决策**：通过 `<InternalsVisibleTo Include="DirectoryCleanAgent.Tests" />` 暴露所有 AI 模块的 `internal sealed` 类给测试项目，直接实例化测试。
- **核心权衡**：牺牲封装性换取可测试性。`internal` 类本意是防止外部程序集依赖，但完全无法测试。添加 InternalsVisibleTo 后可以在不改变类可见性的前提下进行单元测试。
- **被否决的替代方案**：
  1. 将所有 AI 类改为 `public` — 致命缺陷：破坏模块边界设计，UI 项目可直接引用内部实现。
  2. 仅通过 `AiAdvisorService`（public）间接测试 — 致命缺陷：无法覆盖熔断器状态机、限流器令牌桶、提示构建器等内部逻辑的边界条件。

### 2. 时间敏感测试的反射策略

- **最终决策**：使用反射操作 `_openStartedAt` 实例字段来模拟时间流逝，而非修改 `static readonly OpenTimeout`。
- **核心权衡**：反射依赖内部字段名称，若字段重命名则编译无法发现（运行时测试失败）。但避免了修改产品代码添加测试钩子。
- **被否决的替代方案**：
  1. 通过反射修改 `static readonly OpenTimeout` — 致命缺陷：.NET 8 运行时报 `FieldAccessException: Cannot set initonly static field after type is initialized`。
  2. 产品代码添加可注入的 `ITimeProvider` — 致命缺陷：过度设计，仅为测试引入抽象层违反项目简洁性原则。

### 3. HttpClient 模拟方案

- **最终决策**：`Mock<HttpMessageHandler>` + `Protected().Setup("SendAsync", ...)` 控制 HTTP 响应。
- **核心权衡**：Moq Protected 成员模拟依赖字符串方法名，不如接口抽象类型安全。但避免了引入 `IHttpClientFactory` 或自定义 `IHttpClient` 抽象。
- **被否决的替代方案**：
  1. 使用 WireMock.Net 启动本地 HTTP 服务器 — 致命缺陷：引入额外依赖，在 CI 环境中端口冲突风险高。
  2. 添加 `IHttpClient` 包装接口 — 致命缺陷：仅为测试修改产品代码，增加不必要的抽象层。

### 4. AiRateLimiter 耗时测试处理

- **最终决策**：将涉及自旋等待的 4 个测试标记为 `[Fact(Skip = "...")]`，推迟到集成测试阶段验证。
- **核心权衡**：单元测试无法完整覆盖令牌桶自旋等待逻辑，但避免了测试挂起和 CI 超时。
- **被否决的替代方案**：
  1. 将 `private const MaxTokens` 改为可配置 — 致命缺陷：const 值无法通过反射修改，需要重构产品代码。
  2. 测试中使用 `Task.Delay` 延长超时 — 致命缺陷：测试运行时间过长（每个测试可能达数秒），总计可能超时。

## 二、隐含前提与关键假设清单

1. **[假设1]**：假设 `Microsoft.Extensions.Logging.Abstractions` NuGet 包已作为传递依赖存在于测试项目中（通过 Core 项目引用）。若不成立，需显式添加该包引用。
2. **[假设2]**：假设 AI API 响应格式中，`CallAiApiAsync` 返回的原始 HTTP body 可直接传递给 `AiPromptBuilder.ParseResponse` 解析。当前产品代码中未做 `ChatCompletionResponse → content` 提取，假设上层调用方（如自定义 API 代理）已处理。
3. **[假设3]**：假设 `AiDailyUsageTracker` 中 `UsedCount` getter（使用 `lock`）与 `LoadAsync/IncrementAsync`（使用 `SemaphoreSlim`）的同步不一致性是已知的、可接受的设计权衡，而非需要修复的 Bug。
4. **[假设4]**：假设测试环境有文件系统写权限（`Path.GetTempPath()`），可创建临时目录和文件。若在受限环境（如 Docker 容器只读文件系统）中运行，相关 I/O 测试将失败。

## 三、边界与范围界定（明确"不做什么"）

1. **[明确不做]**：本期不做 AI 模型的真实 API 集成测试（不连接 OpenAI/Ollama 服务端）。所有 AI API 调用均通过 Moq HTTP 模拟验证。
2. **[明确不做]**：本期不做 Everything SDK 的真实集成测试（不在真实 Everything 环境中运行）。B1 适配器已有完善的 Moq 测试覆盖。
3. **[明确不做]**：本期不做 UI 自动化测试（不引入 WinAppDriver 或类似框架）。ViewModel 层测试已存在但属 C 阶段范畴。
4. **[明确不做]**：本期不追求 B1-B6 模块的覆盖率从 85% 提升到 95%+。重点填补 B7 的 0% → 80%+ 缺口。
5. **[明确不做]**：本期不做跨平台测试。项目锁定 `net8.0-windows`，所有测试在 Windows 环境下运行。
6. **[明确不做]**：不做 AI 模块的 DI 集成测试（`AiServiceRegistrationTests` 已计划但未实施，因 DI 容器测试需要引入 `Microsoft.Extensions.DependencyInjection` 包）。

## 四、待确认事项（需要人工介入的模糊点）

1. **[待确认项A]**：`AiAdvisorService.CallAiApiWithPipelineAsync` 方法直接对原始 HTTP body 调用 `ParseResponse`，未先提取 `choices[0].message.content`。需要确认：自定义 API 端点的响应格式是直接返回 `{label, confidence, explanation}` JSON，还是标准 OpenAI Chat Completions 格式？如是后者，需在 `CallAiApiWithPipelineAsync` 中添加 `AiChatCompletionResponse.ExtractContent()` 调用。
2. **[待确认项B]**：`AiDailyUsageTracker.UsedCount` 使用 `lock(_lock)` 而 `LoadAsync/IncrementAsync` 使用 `_lock.WaitAsync()`（SemaphoreSlim），两者不互斥。需确认：这是否为已知设计决策，还是需要修复的同步 Bug？
3. **[待确认项C]**：`ProcessSingleFileAsync` 在日限达到后调用 `_circuitBreaker.RecordFailure()`，将"日限达到"视为"失败"是否合理？当前设计中，日限达到与 API 失败共用熔断计数。
4. **[待确认项D]**：30 秒的 `ApiTimeout`（`AiAdvisorService`）和 30 秒的 `OpenTimeout`（`AiCircuitBreaker`）是否需从配置文件读取？当前均为硬编码常量。

## 五、关键外部依赖锁定

- **依赖项**：`Microsoft.Extensions.Logging.Abstractions` NuGet 包（≥ 8.0.0），通过 `DirectoryCleanAgent.Core` 项目传递引用。若该传递依赖缺失，所有使用 `NullLoggerFactory` 的测试将编译失败。
- **依赖项**：`Moq` 4.20.70 NuGet 包，已存在于测试项目中。若需要模拟 `HttpClient`，当前版本支持 `Protected().Setup()` 模式，无需升级。
- **依赖项**：`xUnit` 2.9.0 + `xunit.runner.visualstudio` 2.8.2，已存在于测试项目中。所有测试使用 `[Fact]` 和 `[Theory]` 属性。
- **依赖项**：`coverlet.collector` 6.0.2，已存在于测试项目中。覆盖率测量需通过 `/p:CollectCoverage=true` 参数启用。
