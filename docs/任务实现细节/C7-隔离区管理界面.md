# C7-隔离区管理界面 — 阶段性关键决策与细节控制文档

## 一、关键架构/设计决策记录

### 1. 接口扩展模式：最小侵入 vs 新建接口

- **最终决策**：在现有 `IQuarantineManager`、`IBackupManager`、`IDeletionRecordRepository` 三个接口上直接新增方法签名，而非创建独立的 `IQuarantineUIService` 门面接口。
- **核心权衡（Trade-off）**：牺牲接口的单一职责纯度（IQuarantineManager 同时承担存储管理和UI数据供给），换取最小化类爆炸和依赖链复杂度。3 个新增方法均匀分布在已有实现类中，无需额外的协调层。
- **被否决的替代方案**：
  1. 新建 `IQuarantineUIService` 门面接口聚合三个底层接口 → **致命缺陷**：引入了不必要的编排层，且 UI 层需要同时注入底层接口和新接口，增加 DI 复杂度而无实际收益。
  2. 将所有新方法放在 `IQuarantineManager` 中 → **致命缺陷**：单文件恢复依赖 DeletionRecord 查询（数据层职责），强行塞入隔离区管理器会破坏层次边界。

### 2. DataGrid 绑定模型：可变包装 vs 修改 DTO

- **最终决策**：新建 `QuarantineFileItem`（ViewModelBase 派生）作为 `QuarantineFileEntry`（readonly struct）的可变包装类，专用于 DataGrid CheckBox 双向绑定。
- **核心权衡（Trade-off）**：每个隔离区文件产生额外的堆分配（包装对象），换取与现有 MVVM 基础设施（ViewModelBase.SetProperty）的完全兼容，且不污染 Core 层 DTO 的不可变语义。
- **被否决的替代方案**：
  1. 将 `IsSelected` 直接添加到 `QuarantineFileEntry`（改为 class） → **致命缺陷**：Core 层 DTO 不应包含 UI 状态（IsSelected），违反分层原则，且破坏了与 `QuarantineStats` 一致的 readonly struct 值语义约定。

### 3. 隔离区文件枚举策略：全量 vs 分页

- **最终决策**：`ListQuarantineFilesAsync` 全量枚举隔离区中所有 `.quarantine` 文件，直接构建 `List<QuarantineFileEntry>` 返回。
- **核心权衡（Trade-off）**：假设隔离区文件数量通常较小（< 10K），全量加载简化 UI 逻辑（无需分页/虚拟滚动）。若隔离区膨胀至数万文件，内存和首屏加载时间将退化为性能瓶颈。
- **被否决的替代方案**：
  1. `IAsyncEnumerable<QuarantineFileEntry>` 流式返回 → **致命缺陷**：WPF DataGrid 无法直接绑定异步流，需要额外的缓冲/分页逻辑，实现复杂度远超当前阶段需求。

## 二、隐含前提与关键假设清单

1. [假设1]：隔离区文件数量通常不超过 5000 个（基于 10GB 上限、平均 2MB/文件的估算）。若超出此范围，`ObservableCollection` 全量加载将导致 UI 卡顿。
2. [假设2]：隔离区中每个 `.quarantine` 文件名格式严格遵循 `{SHA256前16位}_{原始文件名}_{Unix秒}.quarantine` 契约。若存在手工放置的非法格式文件，将被静默跳过。
3. [假设3]：`DeletionRecord` 表中的 `FileHash` 始终以小写十六进制存储，且 `COLLATE NOCASE` 查询能正确匹配。若历史数据中存在大小写混乱，恢复操作将失败。
4. [假设4]：恢复文件时，原始目标目录仍然存在。若原始目录已被删除，`Directory.CreateDirectory` 会自动创建，但若整个卷已被移除，恢复将失败。

## 三、边界与范围界定（明确"不做什么"）

1. [明确不做]：本阶段不提供隔离区文件的**单独查看**（如文件内容预览、属性面板）。详情面板仅显示 DataGrid 中的元数据列。
2. [明确不做]：本阶段不提供隔离区文件的**搜索/过滤/排序**功能。DataGrid 支持默认列排序（点击列头），但无自定义搜索框。
3. [明确不做]：本阶段不做**拖放恢复**（将文件从隔离区拖到资源管理器）。恢复仅通过"恢复选中"按钮执行。
4. [明确不做]：本阶段不做隔离区文件的**压缩/加密**存储。文件以原始内容存放在 `.quarantine` 文件中。
5. [明确不做]：本阶段不做隔离区与回收站之间的**跨区迁移**。恢复和删除操作均在隔离区范围内完成。

## 四、待确认事项

1. [待确认项A]：恢复文件时若原始路径已存在同名文件，当前策略是添加 `_restored_N` 后缀（复用 `BackupManager.ResolveRestorePath`）。**是否需要在恢复前弹窗让用户选择覆盖/重命名/跳过？** 当前直接自动后缀，无用户交互。
2. [待确认项B]：永久删除操作前的确认对话框文本为通用警告。**是否需要逐文件列出待删文件名？** 当前仅显示数量和总大小。
3. [待确认项C]：隔离区管理窗口的标题栏显示"隔离区管理"。**是否需要将标题本地化（en/zh-CN）？** 当前硬编码为中文。
4. [待确认项D]：`QuarantineFileItem.IsSelected` 通过 `PropertyChanged` 事件触发 `RefreshCommandStates()`。**高文件数（> 5000）场景下每次选中变更都全量重新计算 `HasSelection`，是否接受此性能开销？** 当前实现为 `QuarantineFiles.Any(f => f.IsSelected)` O(n) 遍历。

## 五、关键外部依赖锁定

| 依赖项 | 依赖说明 | 阻塞影响 |
|--------|---------|---------|
| `IQuarantineManager` (B6) | 提供 `ListQuarantineFilesAsync` / `DeleteQuarantineFileAsync` / `GetQuarantineStatsAsync` | 文件列表无法加载，删除按钮失效 |
| `IBackupManager` (B5) | 提供 `RestoreQuarantineFileAsync` 单文件恢复 | 恢复按钮失效 |
| `IDeletionRecordRepository` (Data) | 提供 `GetByFileHashAsync` 按哈希查原始路径 | 恢复操作无法确定原始路径，全部恢复失败 |
| `IConfigService` (A4) | 提供 `QuarantineMaxSizeBytes` / `QuarantineRetentionDays` / `DataStoragePath` | 容量展示错误，过期判断失效 |
| Everything SDK ≥ 1.4.1.1000 | 隔离区功能不直接使用 Everything（仅用文件系统 API） | **无直接阻塞** |
| SQLite (WAL 模式) | `DeletionRecord` 表存储原始文件路径信息 | 哈希查询失败，恢复功能不可用 |
