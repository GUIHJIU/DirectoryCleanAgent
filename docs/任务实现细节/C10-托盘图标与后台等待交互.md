## 一、 关键架构/设计决策记录（必须包含"被否决的方案"）

### 决策1：后台等待完成通知机制 — 两阶段完成模式

1. **最终决策**：ViewModel 后台轮询结束后**不直接触发 `WizardCompleted`**，而是先触发新的 `BackgroundWaitReady` 事件 → App 层显示气泡通知并启动 5 秒定时器 → 用户点击气泡/双击托盘/定时器超时后 → App 层回调 `ViewModel.CompleteBackgroundWait()` → 触发 `WizardCompleted` 完成窗口切换。

2. **核心权衡（Trade-off）**：
   - 牺牲了简单性：原本一个事件调用链变成了三阶段（VM→App 通知→App 回调→VM 完成），增加了 1 个新类（`BackgroundWaitReadyEventArgs`）、2 个新事件（`BackgroundWaitReady`/`BackgroundWaitCanceled`）、1 个新方法（`CompleteBackgroundWait`）。
   - 换取了：用户能在通知阶段看到完成状态（成功/超时），有机会主动点击气泡立即进入主界面（不等 5 秒自动转换），避免了静默切换的糟糕用户体验。

3. **被否决的替代方案**：
   - **方案A：在 ViewModel 中直接调用 TrayIconService 显示通知**：View 层关注点泄漏到 ViewModel，违反 MVVM 分层原则，且 ViewModel 不应依赖 UI 基础设施（P/Invoke `Shell_NotifyIcon`）。致命缺陷：ViewModel 需要引用 `TrayIconService`，后者依赖 WPF 窗口句柄，彻底破坏了 ViewModel 的可测试性。
   - **方案B：在 ExecuteStartBackgroundWait 完成后延迟 5 秒再触发 WizardCompleted**：只需一行 `await Task.Delay(5000)`，代码最简单。致命缺陷：延迟期间用户无法点击气泡或双击托盘主动唤回（因为 WizardCompleted 尚未触发，窗口切换未发生），且阻塞了线程池线程。

### 决策2：气泡通知点击交互 — NIN_BALLOONUSERCLICK vs Windows Toast

1. **最终决策**：使用 `Shell_NotifyIcon` 的 `NIN_BALLOONUSERCLICK` (0x0405) 回调消息实现气泡通知点击"查看结果"。当用户点击气泡通知时，`TrayIconService.HandleMessage` 识别该消息并触发 `BalloonClicked` 事件，App 层立即完成窗口转换。

2. **核心权衡（Trade-off）**：
   - 牺牲了通知中心的丰富交互（无操作按钮、无 Hero 图片、无输入框），且依赖 Win32 遗留 API（`NOTIFYICONDATA` 结构体最大 `szInfo` 仅 256 字符）。
   - 换取了零外部依赖（无需 `Microsoft.Toolkit.Uwp.Notifications` NuGet 包）、零 COM 注册（Windows Toast 需要快捷方式 + `INotificationActivationCallback`）、以及最低实现成本（仅新增 2 个常量和 1 个事件，不改变任何现有架构）。

3. **被否决的替代方案**：
   - **Windows 10/11 Toast 通知（`Microsoft.Toolkit.Uwp.Notifications`）**：支持"查看结果"按钮和内联操作，UI 美观，与 Windows 11 设计语言一致。致命缺陷：需要创建开始菜单快捷方式作为 AppId 锚点、注册 COM 激活回调、处理应用不在前台时的激活路径——整套机制至少需要 3 天开发+调试，远超 C10 的 1 天工时预算。而且引入了一个重量级 NuGet 依赖。
   - **WinForms `NotifyIcon` 控件**：内置 `BalloonTipClicked` 事件，代码最简洁。致命缺陷：已在 CLAUDE.md 中明确排除 WinForms 依赖（"避免 WinForms 依赖冲突"），且 WinForms `NotifyIcon` 在 WPF 应用中存在消息泵兼容性问题。

### 决策3：ViewModel 实例获取方式 — DataContext vs DI 新建

1. **最终决策**：修改 `App.xaml.cs` 的 `ShowSetupWizardAsync` 方法，从 `_wizardWindow.DataContext` 获取 ViewModel 实例（而非通过 `_serviceProvider.GetRequiredService<SetupWizardViewModel>()` 新建），确保事件订阅与窗口使用的是同一个实例。

2. **核心权衡（Trade-off）**：
   - 牺牲了 DI 容器的纯粹性：不再完全通过 DI 容器获取依赖，而是依赖 WPF 的 `FrameworkElement.DataContext` 属性——这是一种 Service Locator 反模式的变体。
   - 换取了事件路由的正确性：`SetupWizardViewModel` 注册为 Transient，DI 每次 `GetRequiredService` 会创建新实例，导致 App 层订阅的事件在一个实例上，而窗口操作的是另一个实例——事件永远不会触发。

3. **被否决的替代方案**：
   - **将 SetupWizardViewModel 注册为 Singleton**：总有一个实例，事件路由天然正确。致命缺陷：每次向导启动后 ViewModel 状态会被污染（前一用户的配置残留），且并发场景下不可复用；虽然当前无并发需求，但 Singleton 引入的隐式共享状态是未来 bug 的温床。
   - **保持 Transient 注册，但 App 层保存 ViewModel 引用**：在 `ShowSetupWizardAsync` 中获取 ViewModel 并存入字段。致命缺陷：当 `_wizardWindow` 被替换时，`_wizardViewModel` 字段需要同步更新——两个字段之间的状态耦合是典型的内存泄漏和状态不一致风险点。

## 二、 隐含前提与关键假设清单（风险预警）

1. **[假设1]**：假设 `Shell_NotifyIcon` 的 `NIN_BALLOONUSERCLICK` (0x0405) 在所有目标 Windows 版本（Windows 10 21H2+ / Windows 11）上行为一致。若某版本未正确发送此回调（已知部分 Windows 10 早期版本在"专注助手"开启时抑制气泡通知），用户将无法通过点击气泡唤回窗口。**缓解措施**：5 秒自动转换定时器作为兜底机制。

2. **[假设2]**：假设 `SetupWizardWindow.DataContext` 在 App 层调用 `CompleteBackgroundWaitAndTransition()` 时仍然有效（窗口未被 GC 回收或用户强制关闭）。`CompleteBackgroundWait` 方法通过 `_pendingBackgroundWaitResult != null` 的幂等保护，以及 `_wizardWindow?.DataContext is SetupWizardViewModel` 的空检查来防护此场景。

3. **[假设3]**：假设后台等待期间用户不会切换到其他 Everything 操作（如手动重建索引）。`WaitForIndexAsync` 仅检查 `is_indexing` 布尔值，不感知索引是否被外部进程重置。

4. **[假设4]**：假设 5 秒自动转换时间足够用户在通知弹出后做出反应。若用户离开电脑，5 秒后自动进入主界面是合理的默认行为（用户回来时已看到主界面）。但若用户在通知消失后才回来，将错过"查看结果"的交互窗口——此时只能从主界面操作。

## 三、 边界与范围界定（明确"不做什么"）

1. **[明确不做]** 本期不实现 Windows 10/11 Toast 通知（`Windows.UI.Notifications` / `Microsoft.Toolkit.Uwp.Notifications`）。气泡通知的"查看结果"通过 `NIN_BALLOONUSERCLICK` 回调实现，通知中心无法留存历史通知，也不支持操作按钮。

2. **[明确不做]** 本期不实现 MainWindow 的最小化到托盘（"关闭到托盘"）。C10 任务范围限定为"后台等待"场景（即 SetupWizard 的 Step0 索引等待流程），MainWindow 的托盘行为留待后续版本。

3. **[明确不做]** 本期不实现托盘图标的动画/状态切换（如等待中显示沙漏图标、完成显示绿色勾）。当前统一使用 `IDI_APPLICATION` 默认图标。

4. **[明确不做]** 本期不实现后台等待期间的工具提示（tooltip）定时刷新（显示"已等待 X 分 Y 秒"）。当前仅在后端等待完成时更新一次 tooltip（"索引已就绪 — 点击查看结果"或"索引等待超时 — 点击查看结果"）。实时刷新需要 Timer 回调，与 WPF 窗口消息循环的协调复杂度超出本阶段范围。

5. **[明确不做]** 本期不修改除 `App.xaml.cs`、`SetupWizardViewModel.cs`、`TrayIconService.cs`、`SetupWizardWindow.xaml.cs`（仅新增 `RestoreFromTray` 方法）、`BackgroundWaitReadyEventArgs.cs`（新增文件）以外的任何文件。不动 XAML 布局。DI 注册逻辑中 `TrayIconService` 由 Singleton 调整为 Transient（代码审计修复 #2）。

## 四、 待确认事项（需要人工介入的模糊点）

1. **[待确认项A]**：气泡通知的自动转换时间**暂定为 5 秒**。若用户反馈太短/太长，可在 `App.xaml.cs` 的 `BackgroundWaitAutoProceedSeconds` 常量处调整（当前为 `private const int BackgroundWaitAutoProceedSeconds = 5`）。

2. **[待确认项B]**：气泡通知文本中的措辞**暂定为"索引构建完成，正在进入主界面…"**（成功）和**"索引未就绪，将以警告模式运行，部分功能受限。"**（超时）。实际用词需产品/UX 确认，当前仅确保功能正确。

3. **[待确认项C]**：`SetupWizardViewModel` 的 DI 注册**仍为 Transient**（未改为 Singleton）。从 `DataContext` 获取实例的解决方式是 workaround，若后续架构评审认为应改为 Singleton 或其他生命周期，需全局评估对测试和状态管理的影响。

4. **[待确认项D]**：`BackgroundWaitReadyEventArgs` 类放置在 `ViewModels/` 目录下（与 `SetupWizardViewModel` 同目录）。若项目有事件参数的统一目录约定，需调整位置。

## 五、 关键外部依赖锁定

* **依赖项1：Everything SDK ≥ 1.4.1.1000** — 后台等待的核心轮询逻辑依赖 `IEverythingDetector.WaitForIndexAsync()` 方法。若 Everything SDK 版本不匹配或 IPC 通道异常，后台等待将提前进入超时/取消路径，用户体验降级。
* **依赖项2：Windows Shell32.dll (`Shell_NotifyIcon`)** — 托盘图标和气泡通知完全依赖此 Win32 API。Windows Server Core / Nano Server 没有 Shell 体验（explorer.exe），托盘图标将不可见。当前目标用户为桌面 Windows 10/11，此依赖无阻塞风险。
* **依赖项3：WPF `System.Windows.Threading.DispatcherTimer`** — 5 秒自动转换定时器依赖 WPF Dispatcher 消息循环。若应用程序在定时器触发前被强制终止（Task Manager / `taskkill /f`），定时器随进程终止，无资源泄漏。正常退出时 `OnExit` → `StopBackgroundWaitTimer` 确保停止。
