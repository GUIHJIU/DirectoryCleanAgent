## 一、关键架构/设计决策记录（必须包含"被否决的方案"）

### 1. 200 万测试文件生成：C# 控制台工具（TestVolumeGenerator）vs PowerShell

- **最终决策**：选择 C# 控制台工具（`tools/TestVolumeGenerator/`）作为主要生成工具，使用 `Parallel.ForEach` + NTFS 稀疏文件策略。同时保留 PowerShell 轻量脚本（`scripts/Generate-TestVolume.ps1`）用于 1-5 万文件的快速冒烟测试。
- **核心权衡（Trade-off）**：
  - 牺牲了零依赖的简便性（C# 工具需要 .NET 8 SDK + `System.CommandLine` NuGet 包恢复），换取了 50-100 倍的生成速度提升（5000-10000 文件/秒 vs 3-5 文件/秒）。
  - 牺牲了工具的即时可用性（需先 `dotnet build`），换取了断点续传（`--skip-existing`）和可复现性（`--seed` 确定性种子）。
  - 大文件（>10MB）采用稀疏文件策略：仅写入首尾 64KB 随机数据，中间区域为零填充（NTFS 稀疏文件机制）。牺牲了文件的完整随机性，换取了约 85% 的磁盘空间节省（35GB → 5-8GB 实际占用）。
- **被否决的替代方案**：
  1. **纯 PowerShell 脚本生成 200 万文件** — 致命缺陷：cmdlet 开销导致速率仅 3-5 文件/秒，生成 200 万文件需要 4-6 天，不切实际。
  2. **使用 Windows `fsutil file createnew` 批量创建零字节文件** — 致命缺陷：零字节文件无法触发规则引擎的大小相关规则（如"大文件清理建议"），也无法验证 SHA-256 哈希计算的正确性。

### 2. 测试基础设施设计：opt-in 非侵入式 vs 重构已有测试

- **最终决策**：所有新增基础设施放在 `src/DirectoryCleanAgent.Tests/Infrastructure/` 新文件夹中，作为**可选的 opt-in 层**。不修改任何已有测试文件。新测试类可选择性继承 `IntegrationTestBase` 或 `PerformanceTestBase`，也可直接实例化 `QuarantineSimulator`、`TestFileGenerator` 等工具类。
- **核心权衡（Trade-off）**：
  - 牺牲了统一性（已有测试继续使用自包含的 SetUp/TearDown 模式，新测试使用基类），代码风格短期不一致。
  - 但换取了零风险（已有 200+ 测试用例不受任何影响）和渐进式迁移（新测试自然采用新模式，旧测试在未来重构时可选迁移）。
- **被否决的替代方案**：
  1. **修改所有已有测试类继承 `IntegrationTestBase`** — 致命缺陷：改动范围过大（40+ 测试类），风险高（每个测试类的构造函数签名不同），且可能引入基类 Dispose 与现有 Dispose 的竞态条件。
  2. **将工具类作为 NuGet 包发布** — 致命缺陷：过度抽象，测试工具类仅在当前解决方案内使用，打包为 NuGet 增加维护负担且无外部消费者。

### 3. 隔离区/回收站模拟器：真实文件系统操作 vs Moq 纯模拟

- **最终决策**：`QuarantineSimulator` 和 `RecycleBinSimulator` 是**真实文件系统操作类**（在临时目录创建符合命名规范的物理文件），而非 Moq mock 对象。
- **核心权衡（Trade-off）**：
  - 牺牲了纯单元测试的隔离性（产生了真实的磁盘 I/O，测试速度略慢）。
  - 但换取了与产品代码的真实交互验证（如 `BackupManager.TryRestoreFromRecycleBinAsync` 需要实际的文件搜索和 SHA-256 匹配，纯 Moq 无法验证文件命名格式是否正确）。
- **被否决的替代方案**：
  1. **使用 Moq 模拟文件系统（`System.IO.Abstractions`）** — 致命缺陷：引入额外抽象层违反项目简洁性原则，且 `System.IO.Abstractions` 不支持所有 Windows 特有 API（如 `$Recycle.Bin` 目录遍历）。
  2. **使用 RAM Disk 提升 I/O 速度** — 弃用原因：增加环境复杂度（需要 ImDisk 或系统自带 RAM Disk 工具），且 RAM Disk 在 CI 环境中不可用。当前方案使用 `%TEMP%` 目录（通常为 SSD）已满足速度要求。

### 4. Postman 集合设计：独立文件夹 vs Runner 自动化脚本

- **最终决策**：Postman 集合包含 5 个文件夹（连接测试、文件分析、错误处理、边界条件、Ollama 本地服务测试），共计 14 个请求。每个请求包含 JavaScript 测试脚本（`pm.test`）。环境变量支持 3 种服务类型切换（openai/ollama/custom）。
- **核心权衡（Trade-off）**：
  - 牺牲了与 CI/CD 的直接集成（Postman 集合目前仅手动执行，未配置 Newman 命令行运行）。
  - 但换取了 AI API 测试的快速调试能力（开发者可一键导入 Postman，修改环境变量即可切换不同 AI 后端）。
- **被否决的替代方案**：
  1. **使用 xUnit 集成测试替代 Postman** — 致命缺陷：xUnit 测试需要启动完整的 AI 服务（HttpClient + 熔断器 + 限流器），无法独立验证 AI API 端点本身的可用性。
  2. **使用 `.http` 文件（VS Code REST Client）替代 Postman** — 致命缺陷：VS Code REST Client 不支持 JavaScript 测试断言，无法验证响应内容的语义正确性。

## 二、隐含前提与关键假设清单（风险预警）

1. **[假设1]**：假设测试目标磁盘为 NTFS 格式。Everything SDK 仅索引 NTFS 卷，若测试卷位于 ReFS/FAT/exFAT 磁盘，Everything 相关测试将静默失败。当前工具会打印警告但不阻断生成。
2. **[假设2]**：假设 `Everything64.dll` 位于系统 PATH 或 Everything 安装目录中。`EverythingTestHelper` 通过 `DllImport("Everything64.dll")` 进行运行时探测，若 DLL 不在搜索路径中，将返回 `IsAvailable = false`。
3. **[假设3]**：假设测试环境有 `%TEMP%` 目录的读写权限。所有模拟器（QuarantineSimulator、RecycleBinSimulator）和测试基类（IntegrationTestBase）的临时目录创建依赖此权限。若在受限环境（如 Docker Windows 容器）中运行，相关测试将失败。
4. **[假设4]**：假设 `System.CommandLine` NuGet 包（2.0.0-beta4.22272.1）可正常下载。TestVolumeGenerator 工具依赖此包提供 CLI 参数解析。若网络受限（中国 GFW），需手动配置 NuGet 镜像源。
5. **[假设5]**：假设 200 万文件测试卷的生成在 30-60 分钟内完成（NVMe SSD + 8 线程）。若测试环境使用 SATA HDD，生成时间可能延长至 4-6 小时。
6. **[假设6]**：假设隔离区文件命名格式 `{SHA256前16位}_{原始文件名}_{Unix秒}.quarantine` 与 B6 `QuarantineManager.ConstructQuarantineFileName` 的实现严格一致。若未来 B6 修改了命名格式，`QuarantineSimulator` 需同步更新。

## 三、边界与范围界定（明确"不做什么"）

1. **[明确不做]**：**本期不做 200 万文件的真实生成**。TestVolumeGenerator 工具已就绪，但实际 200 万文件的生成是 D3（性能测试）阶段的工作。D1.5 仅确保工具和脚本可用。
2. **[明确不做]**：**本期不做 Everything SDK 的真实集成测试**（不在真实 Everything 环境中运行测试）。`EverythingTestHelper` 提供了运行时检测能力，但实际的 Everything 集成测试将在 D2 阶段编写。
3. **[明确不做]**：**本期不做性能基准的 CI 集成**。`PerformanceTestBase` 提供了度量基础设施，但未配置 CI 流水线的性能阈值告警。这属于 D3 阶段的工作。
4. **[明确不做]**：**本期不做 TestVolumeGenerator 的单元测试**。控制台工具是开发辅助工具，不属于产品代码，不纳入覆盖率统计。
5. **[明确不做]**：**本期不做跨平台脚本**。所有 PowerShell 脚本锁定 Windows 平台（依赖 Windows 注册表、服务管理、NTFS 特性），不兼容 PowerShell Core on Linux/macOS。
6. **[明确不做]**：**本期不修改已有测试文件**。所有新增基础设施在 `Infrastructure/` 新文件夹中，已有 40+ 测试文件保持不变。
7. **[明确不做]**：**本期不做 DI 容器的集成测试配置**。`IntegrationTestBase` 创建的是手动构造的 Mock 对象，不涉及 `Microsoft.Extensions.DependencyInjection`。

## 四、待确认事项（需要人工介入的模糊点）

1. **[待确认项A]**：**Everything 下载地址与版本**。当前 `Check-EverythingEnvironment.ps1` 中硬编码了下载 URL `https://www.voidtools.com/Everything-1.4.1.1026.x64.zip`。需要确认：① 此 URL 在中国大陆是否可直接访问，是否需要备选镜像；② 1.4.1.1026 是否是最新稳定版，是否需更新。
2. **[待确认项B]**：**测试卷根目录默认路径**。当前 `Setup-TestEnvironment.ps1` 默认使用 `$env:TEMP\SpaceClearTestVolume` 作为测试卷根目录。若系统盘空间不足，需手动指定其他磁盘（如 `D:\TestVolume`）。是否需要在脚本中增加可用空间预检？
3. **[待确认项C]**：**Postman 集合中的 API Key 为占位符**。环境变量文件中的 `api_key` 默认值为 `sk-your-api-key-here`。需要使用者替换为真实的 API Key。是否需要在文档中增加获取 API Key 的指引？
4. **[待确认项D]**：**`EverythingTestHelper.SkipIfUnavailable` 的 Skip 机制**。当前使用自定义 `SkipException` 类，但 xUnit 本身不通过异常跳过测试。实际使用时需要通过条件判断（`if (!IsAvailable) return;`）或 xUnit 的 `[Fact(Skip = "...")]` 属性。是否需要改为 xUnit 兼容的实现？
5. **[待确认项E]**：**集成测试的测试卷是否需要持久化**。当前设计是每个 `IntegrationTestBase` 实例创建独立的临时目录并在 `Dispose()` 时删除。对于性能测试，可能需要跨测试共享一个预生成的测试卷（通过 xUnit Collection Fixture）。当前 `TestEnvironmentFixture` 在计划中但未实现（待 D2 阶段），是否需要在 D1.5 中就实现？
6. **[待确认项F]**：**`PerformanceTestBase._disposed` 字段警告**。编译产生 `CS0414` 警告（字段已被赋值但从未使用）。此字段为预留（遵循现有测试的 Dispose 模式），可选择移除或抑制。是否需要保留？
