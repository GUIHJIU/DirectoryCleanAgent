## 一、依赖关系修正

### 1.1 A5 依赖修正（路径标准化与审计日志基础）

**问题**：WBS 标注 A5 依赖 A1，但甘特图将其放在 A3 之后。
**结论**：A5 仅依赖 A1。路径标准化 (`\\?\`) 是纯工具函数，审计日志表创建在 A3 中完成，但 A5 只需要 A1 的基础设施。A5 可与 A2/A3/A4 并行。
**修正**：甘特图中 A5 应改为 `after a1`，而非 `after a3`。

### 1.2 A4 应纳入关键路径

**问题**：关键路径列为 A1→A1.5→A2→A3→A5→B1→B2→...，但 B2 的输入条件明确包含 A4（规则热加载基础设施）。A4 若延期，B2 无法启动。
**修正**：关键路径应更新为：A1 → A1.5 → A2 → A3 → A4 → A5 → B1 → B2 → B3 → B4 → B5 → D2 → D3 → D4 → D5。A4 串联在 A3 之后。

### 1.3 D1 依赖过窄（单元测试仅依赖 B7）

**问题**：D1 输入条件仅列 B7（AI 顾问），但"核心逻辑单元测试"应覆盖 B1-B5 全部模块。
**修正**：D1 依赖改为 **B5**（最后一个核心模块完成）。单元测试应覆盖：
- B1: Everything 适配器 Mock 测试（流式枚举、墓碑过滤、排序下推）
- B2: 规则引擎测试（硬规则、启发式规则、热加载缓存失效）
- B3: 决策引擎测试（四种决策表情况、快照生成、SHA-256 计算）
- B4: 操作执行器测试（乐观删除、回收站容量预检、锁失败降级）
- B5: 回滚管理器测试（恢复+哈希校验、墓碑精确清除）
- B6: 隔离区管理测试（跨卷复制、大文件绕过）
- B7: AI 顾问测试（速率限制、超时处理）

### 1.4 B5 与 B6 的隐式依赖

**问题**：回滚流程（7.6 节）需从隔离区恢复文件，但 B5 依赖仅列 B4, A3, A5，未列 B6。
**分析**：B5 的"从隔离区恢复"只需要读取隔离区文件系统的文件（通过路径映射），不需要 B6 的写入逻辑。B5 和 B6 可以并行开发，只需约定隔离区的目录结构和文件命名规范。
**确认**：当前依赖关系正确，无需修改。但需在实现时确保隔离区文件命名规范在 B5/B6 启动前就已约定（见下文 3.3 节）。

## 二、缺失任务

### 2.1 排除管理器（Exclusion Manager）——拆分说明

**问题**：设计文档 4.2 节将排除管理器列为独立模块，但任务分解中无独立任务。
**结论**：排除管理器职责已拆分到现有任务中，无需新增独立任务：

| 职责 | 归属任务 | 说明 |
|------|---------|------|
| 用户排除目录/扩展名的存储与读写 | A4（配置管理） | JSON 配置中的 `ExcludedDirs`、`ExcludedExtensions` 字段 |
| 自动保护目录检测（Docker/WSL/虚拟机） | B2（规则引擎） | 作为"保护规则"（优先级4）实现，启动时动态检测路径 |
| 排除规则在扫描流程中的应用 | B1（流式适配器）/ B2（规则引擎） | 流式处理第 K 步"排除规则过滤"——在规则引擎裁决前过滤 |

**实现约束**：排除管理器不需要独立类库项目。在 `DirectoryCleanAgent.Core` 中定义一个 `IExclusionManager` 接口（含 `IsExcluded(string path)` 方法），由配置模块和规则引擎共同实现。

### 2.2 国际化/本地化基础设施（新增任务 A6）

**问题**：设计文档要求中/英双语（第 2 节、3.7 节），但无对应任务。
**建议**：**新增任务 A6——本地化基础设施（0.5 d）**

| ID | 任务名称 | 输入条件 | 完成标准 | 预估工时 |
|----|---------|---------|---------|---------|
| A6 | 本地化基础设施 | A1 | .resx 资源文件建立（中/英）；`ILocalizationService` 接口；UI 控件绑定资源键；启动时根据 UserConfig.Language 切换 | 0.5 d |

**实现说明**：
- 在 `DirectoryCleanAgent.Core` 中定义 `ILocalizationService` 接口，含 `string GetString(string key)` 和 `event LanguageChanged`
- 默认语言：简体中文
- A6 可与 A2/A3/A4 并行（仅依赖 A1）
- 语义标签（3.3 节）的中英文映射表纳入此任务

### 2.3 VSS 卷影副本感知（延后处理）

**问题**：设计文档 2.0 节列出"VSS卷影副本感知"，但无任务。
**结论**：VSS 感知的实现复杂度高（需 `IVssBackupComponents` COM 互操作），且与核心清理流程耦合度低。**建议延后至 V3.8+**，当前版本在界面预留"VSS 快照"标签页（禁用状态，标注"即将推出"）。
**操作**：在 C1（主界面框架）中预留一个禁用标签页即可，不分配独立任务。

### 2.4 Everything 事件驱动的实时刷新（合并到 B1 + C2）

**问题**：设计文档 1.2 节要求"文件变更通过Everything事件在数秒内反映到界面"，但无独立任务。
**结论**：此功能拆分为两部分，合并到现有任务：
- **B1 流式适配器**：注册 Everything 的变更回调（`Everything_SetReply` 或轮询 Everything 的变更 IPC），产出 `IFileListProvider.FileChanged` 事件。
- **C2 文件列表视图**：订阅 `FileChanged` 事件，触发增量刷新（只刷新受影响的文件行，不重扫全盘）。
- 在 B1 DoD 中补充："暴露 `event FileChanged`，当 Everything 通知文件变更时触发（含新增/修改/删除）"。
- 在 C2 DoD 中补充："订阅 FileChanged 事件，增量更新文件列表，刷新延迟 <1s"。

## 四、关键实现细节补充（按任务）

### 4.1 A2（Everything SDK 集成）——FRN 探测细节

**FRN 能力检测实现**：
```
1. 调用 Everything_GetResultFullPathName + Everything_GetResultSize 
   查询 C:\Windows\explorer.exe 获取索引中的条目
2. 调用 Everything_GetResultFileReferenceNumber 获取 FRN
3. 若 FRN 返回 0 或 Everything_GetLastError 非 OK：
   - 设置 FRN_AVAILABLE = false
   - 墓碑匹配策略降级为指纹模式
   - 墓碑过期时间强制设为 3 天
4. 若 FRN 有效：
   - 设置 FRN_AVAILABLE = true
   - 墓碑过期时间使用默认 30 天
```

**120 秒索引超时处理**：
```
等待 Everything 索引完成时：
- 轮询间隔：2 秒
- 累计超过 120 秒无响应（Everything 进程仍存活但 is_indexing 始终为 true）：
  → 自动降级为"警告模式"：进入主界面，显示黄色横幅，禁用一键清理
- 无响应 ≠ Everything 崩溃（进程存活检测仍在）
```

### 4.2 A4（配置管理）——FileSystemWatcher 监控路径

```
监控目录：%LocalAppData%\SpaceClear\rules\
监控文件：*.json（启发式规则文件）
监控事件：Changed, Created, Deleted, Renamed
防抖策略：500ms 内同一文件的多次事件合并为一次处理
防抖实现：System.Timers.Timer，AutoReset=false，每次事件重置计时器
```

### 4.3 B1（Everything 流式适配器）——墓碑过滤集成

墓碑过滤发生在 `yield return` 流中，顺序如下：
```
Everything 返回原始条目
  → 提取 FRN（若可用）
  → 查询本地墓碑表（内存 Dictionary<IdentityKey, LocalTombstone> 缓存，启动时加载）
  → FRN 可用：按 file_identity_key (VolumeGuid+FRN) 精确匹配
  → FRN 不可用：按 Size+LastWriteTime 指纹匹配 + 额外校验路径和大小
  → 命中墓碑 → 跳过（已被删除）
  → 未命中 → 继续流式处理
```

**墓碑缓存策略**：启动时一次性加载全部墓碑到 `ConcurrentDictionary<string, LocalTombstone>`（墓碑数量远小于文件总数，内存可控）。新增墓碑时同时更新内存缓存和批量写入队列。

### 4.4 B3（决策引擎）——SHA-256 计算时机与并发

**计算位置**：在 `DecideAndSnapshotAsync` 方法内部，完成决策表仲裁后、生成 `ReadOnlyCollection<DeleteSnapshotEntry>` 快照前。

**并发策略**：
```
1. 获取所有 final_action != protected 的文件列表
2. 使用 SemaphoreSlim(Environment.ProcessorCount) 控制并发哈希计算
3. 每完成一个文件的哈希，调用 IProgress<int>.Report(completedCount)
4. 每次循环检查 CancellationToken
5. 哈希写入 DeleteSnapshotEntry.Sha256Hash
6. 全部完成后深拷贝为 ReadOnlyCollection
```

**注意**：回收站容量预检（B4 的 `SHQueryRecycleBin`）必须在哈希计算之前执行（见 7.2 节时序），因此决策引擎应在收到容量预检通过信号后才启动哈希计算。这意味着 B3 与 B4 之间有**运行时调用依赖**：B4 先调用容量预检 → B3 收到确认后再执行哈希 → B4 再执行实际删除。

### 4.5 B4（操作执行器）——容量预检与快照的协作

**实现时序**（与 B3 协作）：
```
1. B4.QueryRecycleBinCapacityAsync() — 仅一次
2. 若总大小 > 可用空间 → 弹出警告，用户选择缩小范围或确认永久删除
3. 用户确认后 → B3.DecideAndSnapshotAsync() — 生成快照 + 计算哈希
4. B4.ExecuteAsync(snapshot, method, progress, ct) — 执行删除
```

**B4 DoD 补充**："容量预检在决策引擎快照生成之前执行，遵循设计文档 7.2 节的快速失败原则。"

### 4.6 B6（隔离区管理）——与 B4 隔离区路径的交互

B4 在执行隔离区删除时，调用 B6 提供的以下方法：

```csharp
// 获取隔离区目录路径（自动创建，禁用时返回 null）
Task<string?> GetQuarantineDirectoryAsync(CancellationToken ct);

// 检查隔离区是否有足够空间容纳指定大小的文件
Task<bool> HasCapacityAsync(long fileSize, CancellationToken ct);

// 容量强制检查：当前已用 + 新增是否超出配额
Task<bool> EnforceCapacityAsync(long additionalBytes, CancellationToken ct);

// 判断文件是否过大而应绕过隔离区直接删除（阈值：卷剩余空间 20%）
Task<bool> IsLargeFileForQuarantineAsync(long fileSize, CancellationToken ct);

// 构造符合命名规范的隔离区文件名
string ConstructQuarantineFileName(string sha256Hash, string originalFileName);
```

**架构决策（B6 实现阶段确定）**：文件暂存执行逻辑（同卷移动/跨卷复制/SHA-256 校验/.cleaning.tmp 重命名）保留在 B4 的 `OperationExecutor.BackupToQuarantineAsync` 中，不在 B6 接口中暴露 `StageAsync`。QuarantineManager 负责策略与元数据层（命名/容量/过期/统计），OperationExecutor 负责执行机制层（IFileOperation/HashVerifier/文件系统操作）。此拆分避免了 QuarantineManager 对 ShellFileOperation、HashVerifier 的依赖，保持层次隔离。

B4 不需要知道 B6 的内部实现（过期清理、空间统计、文件命名解析），仅通过接口调用策略方法。

### 4.7 C2（文件列表视图）——Everything 原生排序请求映射

当用户点击列头排序时，C2 的 ViewModel 将列映射为 Everything 原生排序参数：

| 列 | Everything 排序参数 |
|----|-------------------|
| 文件名 | `EVERYTHING_SORT_NAME_ASCENDING` / `...DESCENDING` |
| 大小 | `EVERYTHING_SORT_SIZE_ASCENDING` / `...DESCENDING` |
| 修改日期 | `EVERYTHING_SORT_DATE_MODIFIED_ASCENDING` / `...DESCENDING` |
| 路径 | `EVERYTHING_SORT_PATH_ASCENDING` / `...DESCENDING` |
| 扩展名 | `EVERYTHING_SORT_EXTENSION_ASCENDING` / `...DESCENDING` |

**语义标签列不可委托 Everything**——因语义标签是规则引擎产出，不在 Everything 索引中。用户按语义标签排序时，仅对当前 `FileDecisionCache` 子集进行内存排序（此子集规模可控）。

## 五、模块间运行时调用时序（锁定时序契约）

以下是跨模块的关键调用链，必须在实现时严格遵守：

### 5.1 扫描与分析（无状态变更）

```
B1.EnumerateFilesAsync (流式)
  → 逐文件产出 FileItem
  → B2.Evaluate (规则裁决)
  → 命中规则 → A3.WriteQueue (写入 FileDecisionCache)
  → 未命中 → 丢弃
  → UI 定时刷新 (500ms/200条批刷后通知)
```

### 5.2 一键清理（有状态变更——严格时序）

```
用户点击"一键清理"
  → C8 显示进度条 + 取消按钮
  → B4.QueryRecycleBinCapacityAsync    // 步骤1: 容量预检（仅一次）
  → [若容量不足] 弹窗用户选择
  → B3.DecideAndSnapshotAsync           // 步骤2: 决策 + 哈希 + 快照
  → 弹窗确认对话框（汇总信息）
  → B4.ExecuteAsync(snapshot, ...)      // 步骤3: 执行删除
      → 逐文件：IFileOperation 删除
      → 锁失败 → manual_review
      → 成功 → 写墓碑（内存+队列）
  → A5 写审计日志
  → C8 隐藏进度条，刷新 UI
```

### 5.3 规则热加载（异步，不阻塞 UI）

```
FileSystemWatcher 检测到 .json 变更
  → 500ms 防抖
  → UserConfig.RuleCacheVersion++
  → 通知 DecisionEngine
  → 异步清空 FileDecisionCache 表
  → 异步触发全量重扫描 (B1 → B2 → B3)
  → 完成后刷新 UI
  → 整个过程不阻塞 UI 线程
```

## 六、风险项与缓解建议

| 风险 | 影响 | 缓解 |
|------|------|------|
| Everything IPC 在高负载下延迟增大 | B1 流式枚举变慢，UI 刷新延迟 | B1 实现时设置 Everything 查询超时（30s），超时后使用已有结果 |
| 200 万文件墓碑缓存内存 | 若墓碑数量异常大（>10万），`ConcurrentDictionary` 可能占 >100MB | B1 启动时检查墓碑数量，若 >10万则使用 LRU 缓存而非全量加载 |
| SHA-256 计算阻塞 UI | 大文件（>1GB）哈希计算耗时长 | B3 使用异步 I/O（`FileStream` + `SHA256.Create().ComputeHashAsync`），不阻塞 UI |
| 分区表/动态磁盘下 Everything FRN 稳定性 | FRN 可能跨重启变化 | 启动时重新探测 FRN 能力，动态调整墓碑策略 |
| 文件锁定（0x80070020）大量出现 | 大批文件无法删除，用户体验差 | B4 对锁失败文件提供"解锁后重试"和"跳过"两个选项 |
