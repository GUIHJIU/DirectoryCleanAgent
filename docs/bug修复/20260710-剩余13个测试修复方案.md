## A 类：隔离区哈希/文件名不匹配（#1~#6 BackupManager tests）

### A1. 问题概述

6 个 BackupManager 测试全部失败，错误信息均为 `SuccessCount = 0`（Expected: 1 或 2, Actual: 0）。这些测试都涉及隔离区文件恢复流程，根因分为两个子类：

- **子类 A1a**（#1, #2, #3, #6）：隔离区文件内容与原始文件不一致，导致哈希校验失败
- **子类 A1b**（#4, #5）：隔离区文件命名使用硬编码 SHA-256 前缀，与搜索模式不匹配

### A2. 根因详解

#### A1a：隔离区文件内容不匹配（#1, #2, #3, #6）

**涉及测试**：
- `RollbackAsync_WithMixedResults_ShouldReturnCorrectCounts` (#1)
- `RollbackAsync_WithSuccessfulRestore_ShouldClearTombstones` (#2)
- `RollbackAsync_PartialSuccess_ShouldStillCleanTombstones` (#3)
- `VerifyHashesAsync_AllPassed_ShouldReturnCorrectCounts` (#6)

**代码流程分析**：

1. 测试创建原始文件（随机内容）：
```csharp
// BackupManagerTests.cs line 100-107
var filePath = CreateRealFile(_tempDir, "mixed_success.txt");
// → 创建文件，内容 = "test content {Guid1}"
var hash = ComputeSha256ForFile(filePath);
// → 计算 原始文件 的 SHA-256
```

2. 测试创建隔离区文件（不同内容）：
```csharp
// BackupManagerTests.cs line 113-124
CreateQuarantineFile(PathNormalizer.Denormalize(filePath), hash, DateTime.UtcNow);
// → 创建隔离区文件，内容 = "quarantine content {Guid2}"  ← 不同内容!
// → 文件命名使用 hash[..16] 作为前缀（正确）
```

3. BackupManager 搜索隔离区文件：
```csharp
// BackupManager.cs line 579
var searchPattern = $"{sha256Prefix}_{originalFileName}_*.quarantine";
// → 搜索模式匹配文件名 ✓
```

4. BackupManager 验证哈希：
```csharp
// BackupManager.cs line 589
var hashOk = await HashVerifier.VerifyHashAsync(file, record.FileHash, ct);
// → 计算隔离区文件的 SHA-256 = hash("quarantine content {Guid2}")
// → 比较 record.FileHash = hash("test content {Guid1}")
// → 不匹配! ✗ → 文件被跳过，SuccessCount = 0
```

**根因**：`CreateQuarantineFile` 生成随机内容写入隔离区文件，而非复制原始文件内容。后续 `HashVerifier.VerifyHashAsync` 校验隔离区文件哈希与 `DeletionRecord.FileHash`（原始文件哈希）是否一致时，必然不匹配。

#### A1b：隔离区文件命名前缀不匹配（#4, #5）

**涉及测试**：
- `RollbackAsync_Integration_PathConflict_ShouldUseSuffix` (#4)
- `RollbackAsync_Integration_QuarantineRestore_Success` (#5)

**代码流程分析**：

```csharp
// 测试 #5 — RollbackAsync_Integration_QuarantineRestore_Success

// 隔离区文件命名（line 649）:
var sha256Prefix = "a1b2c3d4e5f6a7b8";  // ← 硬编码前缀!
var quarantineFileName = $"{sha256Prefix}_important.txt_{unixSeconds}.quarantine";
// → 文件名为: "a1b2c3d4e5f6a7b8_important.txt_xxx.quarantine"

// 隔离区文件实际哈希（line 655）:
var actualHash = ComputeSha256ForFile(quarantinePath);
// → actualHash = "e3f5a7b9..." (实际计算的 SHA-256)

// DeletionRecord 使用实际哈希（line 661）:
FileHash = actualHash  // → "e3f5a7b9..."

// BackupManager 搜索（line 579）:
var sha256Prefix = record.FileHash[..16];  // → "e3f5a7b9..."
var searchPattern = $"{sha256Prefix}_important.txt_*.quarantine";
// → 搜索模式: "e3f5a7b9..._important.txt_*.quarantine"
// → 实际文件名: "a1b2c3d4e5f6a7b8_important.txt_xxx.quarantine"
// → 不匹配! ✗
```

**根因**：测试用硬编码的 `sha256Prefix` 命名隔离区文件，但 `DeletionRecord.FileHash` 使用的是实际计算的哈希值。BackupManager 的搜索模式从 `FileHash[..16]` 提取前缀，导致文件名前缀与实际哈希前缀不一致，glob 模式匹配失败。

### A3. 修复方案

#### 修复 A1a：修改 `CreateQuarantineFile` 复制原始文件内容

**方案**：修改 `CreateQuarantineFile` 辅助方法，使其复制原始文件的内容到隔离区文件中，而非生成随机内容。

```csharp
// 修改前 (BackupManagerTests.cs line 113-124)
private string CreateQuarantineFile(
    string originalFilePath, string sha256Hash, DateTime deletionTime)
{
    var sha256Prefix = sha256Hash.Length >= 16 ? sha256Hash[..16] : sha256Hash;
    var originalFileName = Path.GetFileName(originalFilePath);
    var unixSeconds = new DateTimeOffset(deletionTime).ToUnixTimeSeconds();
    var quarantineFileName = $"{sha256Prefix}_{originalFileName}_{unixSeconds}.quarantine";

    var quarantinePath = Path.Combine(_quarantineDir, quarantineFileName);
    File.WriteAllText(quarantinePath, $"quarantine content {Guid.NewGuid():N}");  // ← 随机内容
    return quarantinePath;
}

// 修改后
private string CreateQuarantineFile(
    string originalFilePath, string sha256Hash, DateTime deletionTime)
{
    var sha256Prefix = sha256Hash.Length >= 16 ? sha256Hash[..16] : sha256Hash;
    var originalFileName = Path.GetFileName(originalFilePath);
    var unixSeconds = new DateTimeOffset(deletionTime).ToUnixTimeSeconds();
    var quarantineFileName = $"{sha256Prefix}_{originalFileName}_{unixSeconds}.quarantine";

    var quarantinePath = Path.Combine(_quarantineDir, quarantineFileName);
    // 复制原始文件内容到隔离区，确保哈希一致
    var denormalized = PathNormalizer.Denormalize(originalFilePath);
    File.Copy(denormalized, quarantinePath, overwrite: true);
    return quarantinePath;
}
```

**为何能修复**：隔离区文件内容 = 原始文件内容 → `HashVerifier.VerifyHashAsync` 计算的哈希值 = `DeletionRecord.FileHash` → 校验通过 → 文件成功恢复 → SuccessCount 正确。

#### 修复 A1b：使用实际哈希前缀命名隔离区文件

**方案**：移除硬编码的 `sha256Prefix`，改用实际计算的哈希值前缀。

**#5 `RollbackAsync_Integration_QuarantineRestore_Success`**：

```csharp

// 修改后：先创建文件，再计算哈希，用实际哈希构造文件名
```

**#4 `RollbackAsync_Integration_PathConflict_ShouldUseSuffix`**：同理，先计算实际哈希再命名。

**为何能修复**：隔离区文件名的 SHA-256 前缀 = `DeletionRecord.FileHash[..16]` → BackupManager 搜索模式 `{hash[..16]}_filename_*.quarantine` 匹配文件名 → 找到文件 → 哈希校验通过 → 恢复成功。

## B 类：文件路径混合分隔符（#7~#9 DecisionSnapshot tests）

### B1. 问题概述

3 个 DecisionSnapshot 集成测试全部失败：
- `DecideAndSnapshotAsync_Sha256Hash_MatchesExpected` → `Assert.Single() Failure: The collection was empty`
- `DecisionSnapshot_IsImmutable_AfterCreation` → `ArgumentOutOfRangeException: Index was out of range`
- `DecisionSnapshot_ToCompactJson_ValidFormat` → `Assert.Contains() Failure: Sub-string not found`

共同根因：`DecisionSnapshot.Entries` 为空（0 个条目）。

### B2. 根因详解

**代码流程分析**：

1. 测试创建文件时使用含 `/` 的相对路径：
```csharp
// DecisionSnapshotIntegrationTests.cs line 166
var testFile = CreateTestFile("hash_test/known_content.dat", content: content);
```

2. `CreateTestFile` 使用 `Path.Combine` 拼接路径：
```csharp
// IntegrationTestBase.cs line 71
string fullPath = Path.Combine(TestRoot, relativePath);
// TestRoot = "C:\Users\xxx\AppData\Local\Temp\snapshot_GUID"
// relativePath = "hash_test/known_content.dat"
// fullPath = "C:\Users\xxx\...\snapshot_GUID\hash_test/known_content.dat"
//                                                      ↑ 混合了 / 和 \
```

在 Windows .NET 8 中，`Path.Combine` **不会** 将第二个参数中的 `/` 转换为 `\`。结果路径包含混合分隔符。

3. 测试构造 `\\?\` 路径：
```csharp
// DecisionSnapshotIntegrationTests.cs line 171
FilePath = @"\\?\" + testFile,
// → "\\?\C:\Users\xxx\...\snapshot_GUID\hash_test/known_content.dat"
```

4. `DecisionEngine.ComputeSha256HashAsync` 打开文件：
```csharp
// DecisionEngine.cs line 147 (via FileStream)
var hash = await ComputeSha256HashAsync(cache.FilePath, token);
// → new FileStream("\\?\C:\...\hash_test/known_content.dat", FileMode.Open, ...)
```

5. **Windows `\\?\` 路径模式不规范化 `/` 为 `\`**：
   - 正常路径模式：Windows API 将 `/` 和 `\` 都视为路径分隔符
   - `\\?\` 前缀模式：Windows 使用"原始"路径解析，**不转换** `/` 为 `\`
   - 结果：系统尝试查找名为 `hash_test/known_content.dat`（含字面 `/`）的文件
   - 文件不存在 → `DirectoryNotFoundException` 或 `FileNotFoundException`

6. DecisionEngine 捕获异常，将文件放入 `SkippedEntries`：
```csharp
// DecisionEngine.cs line 206-215
catch (Exception ex) when (ex is FileNotFoundException or DirectoryNotFoundException)
{
    skippedEntries.Add(new SkippedEntry { ... });
    // → 文件进入跳过列表，不入快照
}
```

7. `Entries` 为空 → 所有 3 个测试断言失败。

### B3. 修复方案

**方案**：在 `CreateTestFile` 中对 `Path.Combine` 结果调用路径规范化。

```csharp
// 修改 IntegrationTestBase.cs CreateTestFile (line 69-90)
protected string CreateTestFile(string relativePath, byte[]? content = null, long sizeBytes = 1024)
{
    string fullPath = Path.Combine(TestRoot, relativePath);
    // 【新增】规范化路径分隔符，确保所有分隔符统一为 \
    fullPath = Path.GetFullPath(fullPath);
    
    string? dir = Path.GetDirectoryName(fullPath);
    if (dir != null && !Directory.Exists(dir))
    {
        Directory.CreateDirectory(dir);
    }
    // ... 其余不变
}
```

**替代方案**（更安全，在测试代码侧修复）：在 3 个 DecisionSnapshot 测试中，使用 `PathNormalizer.Normalize(testFile)` 而非手动拼接 `@"\\?\" + testFile`。

```csharp
// 修改前
FilePath = @"\\?\" + testFile,

// 修改后
FilePath = PathNormalizer.Normalize(testFile),
```

**建议**：两个方案都实施。`CreateTestFile` 修复从源头防止问题；测试侧修复增加防御性。

**为何能修复**：`Path.GetFullPath` 或 `PathNormalizer.Normalize` 会将 `/` 转换为 `\`，确保路径在 `\\?\` 模式下可被 Windows 正确解析 → 文件能被打开 → SHA-256 计算成功 → 条目进入快照 → 测试断言通过。

## C 类：时间桶硬编码日期（#13 GroupByTime_TimeBuckets）

### C1. 问题概述

`GroupByTime_TimeBuckets` 测试失败：
- Expected: 5 个分组
- Actual: 3 个分组

### C2. 根因详解

**代码流程分析**：

1. 测试数据使用硬编码参考日期：
```csharp
// TestDataFactory.cs line 12
private static readonly DateTime Now = new(2026, 6, 23, 12, 0, 0, DateTimeKind.Utc);

// line 47-54: TimeGroupedCache
CreateCache(..., daysAgo: 0),    // LastWriteTime = 2026-06-23
CreateCache(..., daysAgo: -3),   // LastWriteTime = 2026-06-20
CreateCache(..., daysAgo: -15),  // LastWriteTime = 2026-06-08
CreateCache(..., daysAgo: -60),  // LastWriteTime = 2026-04-24
CreateCache(..., daysAgo: -200), // LastWriteTime = 2025-12-05
```

2. 分组逻辑使用当前时间：
```csharp
// FileListViewModel.cs line 509
var now = DateTime.UtcNow;  // → 2026-07-10 (测试执行日期)
```

3. 时间桶计算：
```csharp
// FileListViewModel.cs line 674-682
var span = now - lastWriteTime;
```

4. 实际分组结果（测试执行于 2026-07-10）：

| daysAgo | LastWriteTime | span (天) | 实际桶 | 期望桶 |
|---------|--------------|-----------|--------|--------|
| 0 | 2026-06-23 | ~17 | **最近 30 天** | 今天 |
| -3 | 2026-06-20 | ~20 | **最近 30 天** | 最近 7 天 |
| -15 | 2026-06-08 | ~32 | **30-90 天** | 最近 30 天 |
| -60 | 2026-04-24 | ~77 | **30-90 天** | 30-90 天 |
| -200 | 2025-12-05 | ~217 | **90 天以上** | 90 天以上 |

**根因**：`TestDataFactory.Now` 是 17 天前的硬编码日期，与 `GetTimeBucket` 使用的 `DateTime.UtcNow`（测试执行时的真实时间）存在偏移。时间跨度计算基于真实当前时间，导致 5 个预期桶合并为 3 个实际桶。

### C3. 修复方案

**方案**：将 `TimeGroupedCache` 从静态只读字段改为静态属性，每次访问时基于 `DateTime.UtcNow` 动态计算。

```csharp
// 修改后

    {
        CreateCache(@"\\?\C:\tmp\f1.tmp", "SystemTempFiles", FinalAction.AutoDelete, 1_000_000, 0),

```

同时修改 `CreateCache` 使用 `DateTime.UtcNow` 替代硬编码的 `Now`：

## D 类：异常类型精确匹配（#12 WaitForIndexAsync_CancelDuringPolling_Throws）

### D1. 问题概述

`WaitForIndexAsync_CancelDuringPolling_Throws` 失败：
- Expected: `typeof(System.OperationCanceledException)`
- Actual: `typeof(System.Threading.Tasks.TaskCanceledException)`

### D2. 根因详解

与 P1-3 (`WaitForIndex_ThrowsAnyAsync` 修复) 完全相同的根因：

```csharp
// ExecutionCancellationTests.cs line 254
await Assert.ThrowsAsync<OperationCanceledException>(async () =>
{
    await pollingTask;
});
```

`Task.Delay` 被取消时抛出 `TaskCanceledException`（继承自 `OperationCanceledException`，但不精确匹配）。`Assert.ThrowsAsync` 要求精确类型匹配。

### D3. 修复方案

**方案**：将 `ThrowsAsync` 改为 `ThrowsAnyAsync`（同 P1-3 修复模式）。

```csharp
// 修改前 (ExecutionCancellationTests.cs line 254)
await Assert.ThrowsAsync<OperationCanceledException>(async () =>
{
    await pollingTask;
});

// 修改后
await Assert.ThrowsAnyAsync<OperationCanceledException>(async () =>
{
    await pollingTask;
});
```

**为何能修复**：`ThrowsAnyAsync` 接受派生类型 → `TaskCanceledException`（`OperationCanceledException` 的子类）满足断言 → 测试通过。

## E 类：隔离区目录嵌套（#10~#11 Quarantine tests）

### E1. 问题概述

- `QuarantineManager_GetStats_ReturnsCorrectCount` → `Assert.True(stats.TotalFileCount >= 30)` 失败
- `QuarantineManager_ListFiles_ReturnsSortedEntries` → `Assert.NotEmpty()` 失败（集合为空）

### E2. 根因详解

**代码流程分析**：

1. 测试创建隔离区目录和模拟器：
```csharp
// QuarantineIntegrationTests.cs line 21-33
_quarantineDir = Path.Combine(TestRoot, "quarantine");
// → _quarantineDir = "C:\Temp\quarantine_int_xxx\quarantine"

_quarantineSim = new QuarantineSimulator(_quarantineDir);
```

2. `QuarantineSimulator` 构造函数追加 `"quarantine"` 子目录：
```csharp
// QuarantineSimulator.cs line 37-43
public QuarantineSimulator(string basePath)
{
    _quarantineDir = Path.Combine(basePath, "quarantine");
    // → _quarantineDir = "C:\Temp\quarantine_int_xxx\quarantine\quarantine"  ← 双层嵌套!
}
```

3. `QuarantineManager` 从配置获取隔离区目录：
```csharp
// QuarantineManager.GetQuarantineDirectoryAsync
// DataStoragePath = TestRoot = "C:\Temp\quarantine_int_xxx"
// quarantineDir = "C:\Temp\quarantine_int_xxx\quarantine"  ← 单层
```

4. 不一致：
   - `QuarantineSimulator` 写入：`TestRoot\quarantine\quarantine\*.quarantine`
   - `QuarantineManager` 搜索：`TestRoot\quarantine\*.quarantine`
   - 两个目录不同 → 文件未被找到

**根因**：`QuarantineSimulator` 构造函数固定追加 `"quarantine"` 子目录名，但测试传入的 `_quarantineDir` 已经包含 `"quarantine"`，导致双层嵌套。而 `QuarantineManager` 的隔离区目录只有单层。

### E3. 修复方案

**方案**：将 `QuarantineIntegrationTests` 中传给 `QuarantineSimulator` 的参数从 `_quarantineDir` 改为 `TestRoot`。

```csharp
// 修改前 (QuarantineIntegrationTests.cs line 33)
_quarantineSim = new QuarantineSimulator(_quarantineDir);
// → QuarantineSimulator 内部追加 "quarantine"
// → 实际目录: TestRoot\quarantine\quarantine

// 修改后
_quarantineSim = new QuarantineSimulator(TestRoot);
// → QuarantineSimulator 内部追加 "quarantine"
// → 实际目录: TestRoot\quarantine
// → 与 QuarantineManager 获取的目录一致 ✓
```

**为何能修复**：`QuarantineSimulator` 内部目录 = `TestRoot\quarantine` = `QuarantineManager` 搜索目录 → 文件位于正确位置 → `GetQuarantineStatsAsync` 和 `ListQuarantineFilesAsync` 能正确枚举到文件 → 统计和列表断言通过。
