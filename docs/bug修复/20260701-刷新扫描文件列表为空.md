# 刷新扫描后文件列表为空

**日期**：2026-07-01
**严重程度**：高（核心用户流程断裂）
**影响范围**：`SimulationService` → `FileListViewModel` 数据链路
**状态**：已修复

---

## STAR 分析

### Situation（情境）

用户点击主界面的"刷新扫描"按钮后，仪表板正常更新（显示处理了 40253 个文件、可释放约 25MB 空间），但下方的文件列表始终为空，显示"暂无文件数据，请先执行磁盘扫描"。仪表板和文件列表对同一次扫描给出了矛盾的结果。

### Task（任务）

定位"刷新扫描"流程中导致文件列表无法加载数据的断裂点，修复该数据链路，使文件列表能够正确展示扫描产出的可操作文件及其分组树。

### Action（行动）

1. **日志对比**：对比刷新扫描的 4 条关键日志，发现"处理 40253 文件"与"从缓存加载 0 条记录"之间的矛盾
2. **数据流追踪**：沿 `ExecuteRefresh()` → `SimulateAsync()` → `LoadDataAsync()` 调用链，逐方法审查数据传递
3. **全局搜索验证**：搜索 `Upsert` 在整个项目中的调用点，确认缓存写入仅存在于 `AiAdvisorService`，扫描流程从未写缓存
4. **源头修复**：在 `SimulationService` 中注入 `IFileDecisionCacheRepository`，在流式扫描过程中执行 Clear → Upsert × N → Flush 三段式写入
5. **返回值重构**：将 `ProcessSingleFile` 的返回值从 `SimulationFileEntry` 扩展为 `(SimulationFileEntry, FileDecisionCache)` 元组，零额外计算开销

### Result（结果）

- 刷新扫描后，文件列表可正确加载全部命中规则的文件（之前为 0 条）
- 分组树（按路径/时间/大小/类型/语义标签/操作建议）可基于缓存数据正常构建
- 仪表板和文件列表的数据来源保持一致（同一次扫描 → 同一份缓存）
- `StreamAnalysisAsync`（导出流）同步适配，不受影响

## 详细分析：如何找到问题

### 第一步：日志矛盾

用户提供的日志中存在明显的数据不一致：

```
[15:04:54 INF] 模拟运行完成: 处理=40253, 可释放=26340915字节, 跳过错误=0, 耗时="00:01:07"
[15:04:54 INF] 仪表板已根据模拟结果更新
[15:04:54 INF] 刷新扫描完成: 处理=40253, 可释放=26340915字节     ← 扫描产出 40253 条
[15:04:54 INF] LoadDataAsync (FileListViewModel.cs:332) — 开始加载文件列表数据
[15:04:54 INF] 从缓存加载了 0 条记录                               ← 加载 0 条
[15:04:54 INF] 文件列表数据加载完成: 可操作=0, 总计=0               ← UI 展示空列表
```

**核心矛盾**：同一次扫描，SimulateAsync 处理了 40253 个文件，但 LoadDataAsync 从缓存读到 0 条记录。两者之间数据一定没有传递。

### 第二步：追踪数据流，定位断裂点

从 UI 按钮入手，沿调用链向上追溯：

```
用户点击"刷新扫描"按钮
  └─ MainViewModel.ExecuteRefresh()              [MainViewModel.cs:869]
       ├─ _simulationService.SimulateAsync()     ← 生产者：产出 SimulationResult
       └─ _fileListViewModel.LoadDataAsync()     ← 消费者：从缓存读取
```

**审查生产者** — `SimulationService.SimulateAsync()`（`SimulationService.cs:60`）：

```
foreach file in EnumerateFilesAsync():
    ProcessSingleFile(file)  →  构建 FileDecisionCache  →  调用 Arbitrate()
    →  聚合到 SimulationResult
    →  ❌ 没有调用 _cacheRepo.Upsert()
```

关键发现：`ProcessSingleFile`（第 286 行）内部创建了 `FileDecisionCache` 对象，交给决策引擎 `Arbitrate()` 后，将仲裁结果映射到 `SimulationFileEntry` 后**丢弃了 `FileDecisionCache`**。整个方法没有任何持久化操作。

**审查消费者** — `FileListViewModel.LoadDataAsync()`（`FileListViewModel.cs:330`）：

```csharp
var allEntries = await _cacheRepo.GetAllAsync(ct);  // 直接从 SQLite 读
```

`LoadDataAsync` 完全依赖 `IFileDecisionCacheRepository` 作为数据源，不做任何实时计算。

**断裂点确认**：Producer 生产了数据但不写入缓存；Consumer 只从缓存读。两者之间缺少写缓存的环节。

### 第三步：全局搜索，确认问题范围

搜索 `Upsert` 在整个项目中的调用：

```
AiAdvisorService.cs:599  →  _cacheRepo.Upsert(rearbitrated)   ← 仅此一处
```

整个项目中只有 AI 重仲裁时会调用 `Upsert` 写入缓存。以下场景均不写缓存：

- `SimulationService.SimulateAsync()` — 刷新扫描
- `SimulationService.StreamAnalysisAsync()` — 导出流
- `MainViewModel.CollectCandidatesAsync()` — 一键清理/智能选择

这确认了问题是一个**系统性的设计遗漏**——缓存写入在整个扫描管道中被忽略，不仅仅是某一个方法的问题。

### 第四步：为什么没有更早暴露？

`ExecuteQuickClean()`（一键清理）和 `ExecuteSuggestDelete()`（智能选择）虽然也不写缓存，但它们绕过了缓存直接调用 `CollectCandidatesAsync()`。所以这些功能可用，但文件列表因为是唯一走缓存路径的消费者，率先暴露了缺失。

## 详细分析：如何修复

### 方案选择

| 方案 | 描述 | 评估 |
|------|------|------|
| **A：SimulationService 写缓存** | 在流式扫描中逐文件 Upsert | ✅ 架构正确，Producer 负责持久化 |
| B：MainViewModel 绕过缓存 | 把 SimulationResult 直接传给 FileListViewModel | ❌ SimulationResult 只有聚合统计+10条样本，不包含全量 |
| C：LoadDataAsync 改为实时扫描 | 消费端自己跑 B1→B2→B3 | ❌ 重复扫描，性能差，违背缓存设计 |

选择方案 A：在数据源头修复，让 Producer 承担持久化职责。

### 前提验证

修复前验证了三个前提：

**前提 1：缓存 API 是否支持流式写入模式？**

审查 `IFileDecisionCacheRepository` 接口：
- `ClearAsync()` — 清空整表（扫描前重置）
- `Upsert(entry)` — 非阻塞入队（适合高频逐文件调用）
- `FlushAsync()` — 强制刷新队列（扫描后确保落盘）

底层使用 `INSERT OR REPLACE` + `file_path` 主键，天然支持幂等重写。完全匹配 Clear → Upsert × N → Flush 三段式模式。

**前提 2：决策数据是否已经计算好？**

`ProcessSingleFile` 内部流程：
```
FileItem → RuleEngine.Evaluate() → FileDecisionCache → DecisionEngine.Arbitrate() → SimulationFileEntry
```

`FileDecisionCache` 已经构建且经过仲裁，只是最后被丢弃。修复只需将其返回出来，**零额外计算开销**。

**前提 3：DI 容器是否已注册依赖？**

`App.xaml.cs` 中的注册顺序：
```
AddDataLayer()                              ← 注册 IFileDecisionCacheRepository
AddSingleton<ISimulationService, SimulationService>()   ← 后注册，可注入前者
```

`IFileDecisionCacheRepository` 在 `SimulationService` 之前注册，直接构造函数注入即可。

### 代码变更

**文件**：`src/DirectoryCleanAgent/Services/SimulationService.cs`

**变更 1**：添加命名空间引用
```csharp
using DirectoryCleanAgent.Data;
```

**变更 2**：注入缓存仓储
```csharp
private readonly IFileDecisionCacheRepository _cacheRepo;

public SimulationService(
    // ... 原有参数 ...
    IFileDecisionCacheRepository cacheRepo,  // 新增
    ILogger<SimulationService> logger)
{
    _cacheRepo = cacheRepo ?? throw new ArgumentNullException(nameof(cacheRepo));
}
```

**变更 3**：SimulateAsync 中添加三段式缓存写入
```csharp
// 扫描前：清空旧缓存
await _cacheRepo.ClearAsync(ct);

// 扫描中：逐文件写入决策
var (entry, cache) = ProcessSingleFile(fileItem, userConfig, aiEnabled, aiTrustLevel);
if (entry.RuleVerdict != RuleVerdict.Keep)
{
    _cacheRepo.Upsert(cache);  // 非阻塞入队
    // ... 聚合统计 ...
}

// 扫描后：刷新写入队列，确保 LoadDataAsync 读到完整数据
await _cacheRepo.FlushAsync(ct);
```

**变更 4**：ProcessSingleFile 返回值扩展为元组
```csharp
// 之前：private SimulationFileEntry ProcessSingleFile(...)
// 之后：
private (SimulationFileEntry Entry, FileDecisionCache Cache) ProcessSingleFile(...)
{
    // ... 相同的计算逻辑 ...
    return (entry, arbitrated);  // 同时返回分析条目和决策缓存
}
```

**变更 5**：StreamAnalysisAsync 适配新返回值
```csharp
(entry, _) = ProcessSingleFile(...);  // 导出流不写缓存，丢弃 Cache
```

### 修复后的数据流

```
用户点击"刷新扫描"
  └─ ExecuteRefresh()
       ├─ SimulateAsync()
       │    ├─ ClearAsync()              ← 清空旧缓存
       │    ├─ foreach file:
       │    │    ├─ ProcessSingleFile()  → (entry, cache)
       │    │    └─ Upsert(cache)        ← 写入缓存（非阻塞入队）
       │    ├─ FlushAsync()              ← 刷新队列，确保落盘
       │    └─ 返回 SimulationResult
       │
       ├─ UpdateDashboardFromSimulation()   ← 仪表板正常 ✅
       │
       └─ LoadDataAsync()
            └─ GetAllAsync()  → 返回 N 条记录  ← 文件列表正常 ✅
```
