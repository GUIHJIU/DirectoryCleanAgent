# 20260715 - AI 分析完成后跨线程 UI 访问导致应用崩溃

## 1. 基础元数据
- **严重级别**：P0（应用崩溃，功能不可用）
- **关联版本/Tag**：master 分支，commit `245fdad` 之后
- **影响范围**：所有触发 AI 批量分析（手动/自动）后，应用必然崩溃
- **触发概率**：100%

## 2. STAR 场景分析
- **S (情境)**：用户在专家模式下，选中文件后点击工具栏"AI 分析"按钮（或扫描完成后自动触发 AI 分析）。AI API 调用约 5 秒后返回，应用在更新 UI 状态时抛出 `System.InvalidOperationException`，进程崩溃。
- **T (任务)**：AI 分析完成后，应正确更新状态栏进度文本、刷新文件列表分组树、在应用最小化时弹出 Toast 通知，全程不应崩溃。
- **A (行动)**：采用纵深防御策略，分两层修复——
  1. **调用方层面**：移除 `MainViewModel` 中对 `LoadDataAsync` 的 `Task.Run` 包装，改用直接 `await`；将 `MainViewModel` 的两个 AI 事件处理器改为 `async void` + `await Dispatcher.InvokeAsync`，确保所有 UI 对象访问均在 Dispatcher 内执行。
  2. **被调用方层面**：`FileListViewModel.LoadDataAsync` 内部所有 UI 绑定属性（`IsLoading`、`HasData`、`IsEmpty`、`EmptyMessage`）的设置均包裹在 `RunOnUIThreadAsync` 中；事件处理器 `OnAiAnalysisCompleted`、`OnAiProgressChanged`、`OnAppStateChanged` 统一改为 `async void` + `await` + try-catch 模式。
- **R (结果)**：构建 0 错误，44 个相关单元测试全部通过。跨线程异常从 P0 崩溃降为不可能发生（所有 UI 访问路径均受 Dispatcher 保护）。

## 3. 根因分析（Root Cause）
> AI 分析 HTTP 回调延续至线程池线程，`MainViewModel.OnAiAnalysisCompleted` 在线程池线程上直接访问 `Application.Current.MainWindow`（WPF `DispatcherObject`），触发跨线程校验异常。

### 4.2 排除掉的干扰项
- **排除 WPF Binding 自动跨线程封送不可靠的假设**：初步怀疑 `PropertyChanged` 从线程池触发时 WPF 绑定引擎未正确封送。经验证，`PropertyChanged` 的 WPF 绑定处理器内部确实会尝试封送，但此处崩溃点在 `DispatcherObject.VerifyAccess()`，与绑定封送无关。
- **排除 `HttpClient` 内部 `ConfigureAwait(false)` 的假设**：经查阅 .NET 8 源码，`HttpClient.SendAsync` 公开 API 已不再内部使用 `ConfigureAwait(false)`（自 .NET 5 起修正）。但实际运行中 HTTP 回调延续至线程池线程仍可能发生（受 `SocketsHttpHandler` 内部连接管理影响）。
- **排除 `FileListViewModel` 单一责任假设**：第一轮修复仅针对 `FileListViewModel` 中的 UI 绑定属性，但崩溃仍然发生，确认还存在另一个订阅者 `MainViewModel` 的处理器。

## 5. 方案推演与决策（Trade-off Analysis）

### 5.1 构思修复方向（头脑风暴）

- **方向A（调用方封送）**：在 `AiAnalysisCoordinator.OnAnalysisCompleted` 触发事件前，主动 `Dispatcher.InvokeAsync` 将事件封送至 UI 线程，确保所有订阅者都在 UI 线程上执行。
- **方向B（订阅方防护 — 治本）**：在每个事件订阅者（`MainViewModel`、`FileListViewModel`）内部，将所有 UI 对象访问包裹在 `Dispatcher.InvokeAsync` 中，形成各自独立的防护层。
- **方向C（移除 ThreadPool 调度）**：在 `AiAnalysisCoordinator` 中将 `OnAnalysisCompleted` 的事件触发从 `QueueUserWorkItemCallback` 改为直接同步触发或 Dispatcher 触发。
- **方向D（全局异常兜底）**：在 `App.xaml.cs` 中注册 `AppDomain.UnhandledException` 和 `DispatcherUnhandledException` 处理器，捕获跨线程异常后自动重试在 UI 线程上执行。

### 5.2 多维度的可行性论证（优缺点量化）

| 方案 | 优点 | 缺点/风险 | 实施成本 | 回滚难度 |
|:---|:---|:---|:---|:---|
| **方案A** | 一处修复，所有订阅者受益；架构层面解决 | 修改核心协调器，涉及事件触发语义变更；`AiAnalysisCoordinator` 作为 Core 层不应依赖 WPF Dispatcher | 低（1 处改动） | 低 |
| **方案B** | 每个 ViewModel 独立防护，符合防御性编程原则；不修改 Core 层 | 需修改多处；每处都要记得包裹；未来新增订阅者容易遗漏 | 中（5-6 处改动） | 低 |
| **方案C** | 直接消除线程池调度 | `QueueUserWorkItemCallback` 是框架内部行为，无法直接控制 | 高（需 hack） | 高 |
| **方案D** | 兜底保护，面向未来 | 掩盖问题而非解决问题；全局异常处理器中的重试逻辑复杂且危险 | 低（1 处改动） | 低 |

### 5.3 最终决策 & 预期

- **最终选定**：**方案B（订阅方防护）**，辅以额外改进（移除不必要的 `Task.Run`、fire-and-forget 改为 await）。
- **决策理由**：
  1. `AiAnalysisCoordinator` 位于 Core 层（`DirectoryCleanAgent.AI`），不应引入 WPF 依赖（`Dispatcher`），方案A 违反分层架构原则。
  2. 方案B 虽然改动点多，但每处改动都很小且独立，不相互影响，可逐步验证。
  3. 同时移除 `MainViewModel` 中的 `Task.Run` 包装，减少不必要的上下文丢失场景。
- **实施前预期（Prediction）**：
  - *稳定性预期*：AI 分析完成后不再发生跨线程崩溃，崩溃率从 100% 降为 0。
  - *行为预期*：AI 分析完成后，状态栏文本正确更新，文件列表正确刷新，Toast 通知正常弹出。
- **副作用预期**：`MainViewModel.InitializeAsync` 和扫描刷新路径中，`LoadDataAsync` 从 fire-and-forget 变为 `await`，初始化/刷新完成时间可能略微延长（增加等待文件列表加载的时间），但用户体验更好（数据一致性保障）。

## 7. 改进措施（举一反三）

- **代码规范**：
  1. **ViewModel 中所有 UI 绑定属性的 setter 必须包裹在 `RunOnUIThreadAsync` 中**，或使用基类层面的 AOP/代码生成统一处理。
  2. **事件处理器模式统一**：所有订阅非 UI 线程事件的处理器统一使用 `async void` + `await Dispatcher.InvokeAsync` + try-catch 模式，禁止 fire-and-forget 和同步 `Dispatcher.Invoke`。
  3. **禁止 `Task.Run` 包装 ViewModel 方法**：`Task.Run` 会丢失 UI 同步上下文，应直接 `await` 异步方法，让内部 I/O 自然释放 UI 线程。

- **监控告警**：建议在 `App.xaml.cs` 的 `DispatcherUnhandledException` 中增加结构化日志，记录跨线程访问的具体调用栈。

- **后续规划**：
  1. 考虑在 `ViewModelBase` 中增加 `AssertUIThread()` 调试断言，在 Debug 构建中自动检测跨线程属性设置。
  2. 评估将 `RunOnUIThreadAsync` 提升到 `ViewModelBase` 基类，避免每个 ViewModel 重复实现。
  3. 技术债：`AiAnalysisCoordinator` 中的 `OnAnalysisCompleted` 通过 `QueueUserWorkItemCallback` 触发（异步），应明确文档化事件触发线程语义（当前无文档说明）。
