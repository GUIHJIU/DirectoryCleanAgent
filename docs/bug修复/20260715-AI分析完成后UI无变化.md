# 20260715 - AI 分析完成后 UI 不显示结果（批量队列刷新竞态）

## 1. 基础元数据

- **严重级别**：P1（核心功能不可用 — 用户手动触发 AI 分析后无法看到任何结果，功能表现为无响应）
- **关联版本/Tag**：V3.7 开发阶段，commit `245fdad` 附近
- **影响范围**：手动批量 AI 分析（工具栏按钮 / 右键菜单触发）、自动 AI 分析（扫描完成后触发）
- **发现方式**：用户手动测试反馈

## 2. STAR 场景分析

- **S (情境)**：用户在文件列表中勾选文件，点击工具栏「AI 分析」按钮（或右键菜单触发），日志显示 AI API 调用成功、结果已写入缓存，但 UI 上文件列表无任何变化——AI 标签列、置信度、解释文本均未更新。

- **T (任务)**：AI 分析完成后，UI 应自动刷新文件列表，用户能在 DataGrid 行中看到 AI 返回的标签（如 `unknown`）、置信度百分比和解释文本。

- **A (行动)**：
  1. 在 `FileListViewModel.OnAiAnalysisCompleted` 中，**`LoadDataAsync()` 之前**增加 `await _cacheRepo.FlushAsync()` 调用，强制刷新批量写入队列，确保 AI 结果已持久化到 SQLite。
  2. 移除 `ExecuteAnalyzeSelectedFilesAsync` 中 `AnalyzeBatchAsync` 返回后的手动回写结果代码（与 `LoadDataAsync` 存在竞态，冗余且有风险）。

- **R (结果)**：
  - 修复前：AI 分析日志完整但 UI 零变化（100% 复现）
  - 修复后：AI 分析完成后 UI 立即刷新，DataGrid 行正确显示 AI 标签、置信度和解释文本

## 3. 根因分析（Root Cause）

> **批量写入队列 500ms 延迟导致 `LoadDataAsync` 在队列刷新前从 SQLite 读到无 AI 标签的过期数据。**

`AiAdvisorService.ApplyResultToCache()` 调用 `_cacheRepo.Upsert()` 是**非阻塞入队**（进入 `ConcurrentQueue`，由 500ms 定时器触发批量写入），而 `OnAiAnalysisCompleted` 事件处理器**同步地**调用 `LoadDataAsync()` → `_cacheRepo.GetAllAsync()` 从数据库读取——此时批量队列尚未刷新，AI 结果不在此次查询返回的数据中，UI 用过期数据重建，表现为"无变化"。

## 5. 方案推演与决策（Trade-off Analysis）

### 5.1 构思修复方向

> 基于根因（`Upsert` 入队与 `GetAllAsync` 之间存在未刷新的时间窗口），可行的解决方向：

- **方向A（治标 — 手动刷新）**：在 `OnAiAnalysisCompleted` 中 `LoadDataAsync()` 之前，显式调用 `await _cacheRepo.FlushAsync()` 强制刷新队列。
- **方向B（治本 — 绕过 DB）**：不重新从数据库加载，直接从内存中的 `_allActionableCache` 重建分组树。`ApplyResultToCache` 已修改了同一内存对象的 `AiLabel/AiConfidence/AiExplanation` 字段。
- **方向C（架构层 — 同步写入）**：将 AI 结果的 `Upsert` 改为同步写入（跳过批量队列），或使用写穿（write-through）缓存策略。
- **方向D（兜底 — 延迟刷新）**：在 `OnAiAnalysisCompleted` 中加 `Task.Delay(600)` 等待队列自然刷新后再调用 `LoadDataAsync`。

### 5.2 多维度的可行性论证

| 方案 | 优点 | 缺点/风险 | 实施成本 | 回滚难度 |
|:---|:---|:---|:---|:---|
| **A — 手动 FlushAsync** | 改动最小（+2 行）；不改变现有架构；确保 DB 一致性 | 仍有一次 DB 往返；FlushAsync 可能刷新其他不相关的批量条目 | 极低（5 分钟） | 极低（删 2 行） |
| **B — 内存重建** | 零 I/O，速度最快；不依赖 DB 状态 | 如果 `_allActionableCache` 与传入 `AnalyzeBatchAsync` 的对象非同一引用（如深拷贝场景），则失效；需确认整个调用链的对象引用一致性 | 低（15 分钟） | 低 |
| **C — 同步写入** | 从根源消除问题；任何读取方都不会读到过期数据 | 破坏批量写入的架构优势（吞吐量降低）；AI 分析是低频操作（≤500 文件/次），批量写入的优势在此场景不明显 | 中（需改 Repository 接口） | 中 |
| **D — 延迟刷新** | 零代码改动（仅加 Delay） | 不可靠（500ms 不一定够，取决于队列积压）；浪费 500ms+ 无意义等待；测试不稳定 | 极低 | 极低 |

### 5.3 最终决策 & 预期

- **最终选定**：**方案A（主）+ 方案B（备选考虑）**。

- **决策理由**：
  - 方案A 改动最小（仅 1 行 `FlushAsync` + 1 行注释），不改变现有架构契约，风险最低
  - `FlushAsync` 是批量队列已暴露的公开方法（`IFileDecisionCacheRepository` 接口已定义），使用其设计意图内的能力
  - 方案B 作为备选已记录在注释中，若未来出现对象引用不一致的问题可快速切换
  - 方案C 改动面大，且 `Upsert` 批量写入对扫描阶段（数千文件/秒）至关重要，不应为 AI 分析场景（低频）牺牲整体吞吐

- **实施前预期**：
  - *功能预期*：AI 分析完成后 UI 立即显示 AI 标签，不再出现"日志显示成功但 UI 无变化"
  - *性能预期*：`FlushAsync` 通常在 < 5ms 内完成（1 条记录的批量 Upsert），`LoadDataAsync` 耗时不变（258 条记录约 50ms）
  - *副作用预期*：`FlushAsync` 会将队列中其他待刷新的条目一并写入，这些条目本就在 500ms 内即将刷新，无负面影响

- **实施后验证**：修复后重新触发批量 AI 分析，日志顺序变为：
  ```
  AI 结果已写入缓存 → FlushAsync(队列刷新) → LoadDataAsync(读到含 AI 标签的数据) → UI 正确显示
  ```

## 7. 改进措施（举一反三）

- **架构层面**：批量写入队列的"非阻塞"语义与上层"写入即读"的预期存在隐式契约违背。建议在 `IFileDecisionCacheRepository` 接口文档中明确标注 `Upsert` 的异步语义（"数据在 ≤500ms 内持久化"），并考虑为需要"写入后立即读取"的场景提供 `UpsertAndFlushAsync` 组合方法。

- **事件时序文档**：`AnalysisCompleted` 事件的触发时机与数据持久化完成时机之间的时序关系，应在 `IAiAnalysisCoordinator` 接口文档中明确说明。

- **测试覆盖**：当前单元测试中 `Upsert` 后显式调用了 `FlushAsync`（如 `FileDecisionCacheRepositoryTests:74`），说明测试编写者知道这个时序问题，但业务代码未遵循同一模式。建议新增集成测试：模拟 `ApplyResultToCache` → `GetAllAsync` 的时序，验证不调用 `FlushAsync` 时读到的是过期数据。

- **监控告警**：当前无。若后续 AI 分析频率增加，可添加 `ai_result_stale_read_count` 指标（当 `LoadDataAsync` 加载的数据中 AI 标签为空但 `_fileStates` 标记为已完成时计数）。
