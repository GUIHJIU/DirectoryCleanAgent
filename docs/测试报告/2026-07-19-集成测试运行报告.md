## 二、崩溃详情

### 崩溃信息

```
Fatal error. System.AccessViolationException: 
  Attempted to read or write protected memory. 
  This is often an indication that other memory is corrupt.
```

### 调用栈

```
EverythingNative.Everything_GetNumResults()           ← Native 崩溃点
  ↑ EverythingSdkWrapper.GetNumResults()
    ↑ EverythingFileListProvider.EnumerateFilesAsync()
      ↑ ScanCancellationTests.EnumerateFilesAsync_CancelDuringStream_StopsMidway
```

### 崩溃测试

| 属性 | 值 |
|------|-----|
| 测试类 | `ScanCancellationTests` |
| 测试方法 | `EnumerateFilesAsync_CancelDuringStream_StopsMidway` |

| 类别 | Cancellation / 扫描取消 |
| 描述 | 在流式枚举过程中取消，验证中途停止能力 |

### 根因分析

#### 核心原因：Everything SDK v2.0 进程全局状态 + xUnit 并行测试竞争

**Everything64.dll 使用进程级全局状态**（SDK 2.0 特性）。`Everything_Query()` 将查询结果写入共享内存，`Everything_GetNumResults()` 从共享内存读取结果数。关键问题是 **Everything64.dll 是进程级单例，整个测试进程共享同一个原生 DLL 的全局状态**。

#### 三层线程安全问题

| 层级 | 问题 | 严重程度 |
|------|------|----------|
| **Everything64.dll 原生层** | SDK 2.0 所有 `SetSearch`/`Query`/`GetNumResults` 操作共享进程全局内存 | **原生不支持并发** |
| **EverythingFileListProvider 层** | 通过 `_sdkLock`（`SemaphoreSlim`）串行化访问，但锁是**实例级**而非**进程级** | 同实例内安全，跨实例不安全 |
| **EverythingDependencyDetector 层** | 静态方法 (`PingIpc`, `CheckVersion`, `DetectFrnCapability`, `CheckIndexReady`) 直接调用 `EverythingNative.*`，**完全不使用锁** | **完全不受保护** |

#### 崩溃机制（四种独立原因）

##### 原因 A（最可能）：xUnit 并行测试导致全局状态被覆盖

`EverythingFileListProvider` 的 `_sdkLock` 锁是实例级别的。xUnit 默认并行执行测试类，如果另一个 `[EverythingFact]` 测试类（如 `EverythingScanIntegrationTests`）在**同一进程内并行执行**，它会通过自己的 `EverythingFileListProvider` 实例调用 `Everything_Query()`，直接覆盖 `Everything64.dll` 内部的全局结果缓冲区。

当 `ScanCancellationTests` 随后调用 `Everything_GetNumResults()` 时，读取到的已是被其他测试覆盖/损坏的缓冲区 → `AccessViolationException`。

> 证据：`EverythingFileListProvider.cs:168-170` 的注释明确说明：
> "SDK 2.0 使用全局进程状态 -- 查询结果集是全局的。若在 yield return 期间释放锁，变更轮询定时器可能执行新查询并覆盖结果集，导致 AccessViolationException。"

##### 原因 B：EverythingDependencyDetector 绕过锁直接调用原生 API

`EverythingDependencyDetector.cs` 中的静态方法直接调用 `EverythingNative.Everything_Query()`、`EverythingNative.Everything_GetNumResults()` 等，完全**绕过了 `_sdkLock` 保护**。如果这些方法在 `EnumerateFilesAsync` 持有锁期间被调用（例如在应用启动流程中），会直接覆盖全局状态。

##### 原因 C：async IAsyncEnumerable 状态机的 try/finally 时序问题

`EnumerateFilesAsync` 在 `try` 块内获取锁，在 `finally` 块中释放。C# 编译器生成的异步迭代器状态机在 `DisposeAsync` 时执行 `finally`，但存在已知的 .NET 边缘情况，`OperationCanceledException` 可能导致状态机未能正确展开 `try/finally`，锁不会被释放。

##### 原因 D：MaxResults=50000 大结果集缓冲区溢出

测试指定 `MaxResults = 50000`（EverythingFileListProvider 第 244 行），Everything64.dll 的共享内存缓冲区可能不足以承载 50000 条带完整元数据（含 FRN、VolumeSerial）的结果，导致 `Everything_Query(true)` 返回 `true` 但内部缓冲区溢出/损坏。

#### 进程终止波及

由于 `AccessViolationException` 是 .NET **corrupted state exception**（CSE），CLR 无法捕获和处理，它直接终止了 `dotnet test` 宿主进程，导致后续 9 个测试无法执行。

   ```
   dotnet test --filter "FullyQualifiedName~Integration&FullyQualifiedName!~ScanCancellationTests" --no-build
   ```

   验证其余测试全部通过。

2. **验证 Everything 环境**: 在测试机上确认 Everything ≥ 1.4.1.1000 正在运行，Everything64.dll 路径在 PATH 中。

### 🟡 短期修复（技术方案）

#### 方案 A（优先级最高）：所有 Everything 集成测试串行化

创建测试集合定义，确保所有需要 Everything SDK 的测试**串行执行**：

```csharp
// 新建文件: EverythingTestCollection.cs
[CollectionDefinition("Everything", DisableParallelization = true)]
public class EverythingTestCollection : ICollectionFixture<EverythingTestFixture> { }

// 在每个 [EverythingFact] 测试类上添加：
[Collection("Everything")]
```

**成本最低、风险最小** — 只需添加 Attribute，不需要修改生产代码。

#### 方案 B（推荐）：进程级全局锁

将 `_sdkLock` 从实例级 `SemaphoreSlim` 提升为**进程级静态锁**：

```csharp
// EverythingFileListProvider.cs
private static readonly SemaphoreSlim _globalSdkLock = new(1, 1);
```

同时让 `EverythingDependencyDetector` 的静态方法也使用同一个静态锁。

#### 方案 C：EverythingDependencyDetector 安全加固

将所有 `EverythingDependencyDetector` 中对 `EverythingNative.*` 的直接调用改为通过注入的 `IEverythingSdk` 实例间接调用，确保走锁保护路径。

涉及文件：
- `src/DirectoryCleanAgent.Everything/EverythingDependencyDetector.cs` 
  - `PingIpc()` (行 372): 直接调用 `EverythingNative.Everything_Query()`
  - `CheckVersion()` (行 401): 直接调用 `EverythingNative.Everything_GetMajorVersion()`
  - `DetectFrnCapability()` (行 422): 直接调用 `EverythingNative.Everything_GetNumResults()`
  - `CheckIndexReady()` (行 464): 直接调用 `EverythingNative.Everything_IsDBLoaded()`

#### 方案 D：缩减测试查询规模

将 `ScanCancellationTests` 中的 `MaxResults` 从 `50000` 减小到 `5000`，降低共享内存缓冲区压力。

### 🟢 长期改进

5. **Mock IFileListProvider**: 为 `IFileListProvider` 创建可控的 mock 实现，避免集成测试对 Everything 的硬依赖
6. **测试超时控制**: 为 `IsIndexingAsync_DelegatesToNative_ReturnsBool`（5s）等慢测试添加超时机制
7. **分离测试套件**: 将 `[EverythingFact]` 测试与纯逻辑测试分为独立的测试运行配置
