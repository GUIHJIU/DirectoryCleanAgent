### 2.1 偏差1：AiDailyUsageTracker.UsedCount 同步原语不一致

| 项目 | 内容 |
|:---|:---|
| **序号** | 1 |
| **审查路径** | `src/DirectoryCleanAgent.AI/AiDailyUsageTracker.cs:54-60` → `UsedCount` getter |
| **如何发现问题** | 对照实现细节文档 §三 的"线程安全：SemaphoreSlim(1,1) 序列化所有读写操作"声明，发现 getter 使用 `lock (_lock)`（Monitor）而非 `_lock.Wait()`（SemaphoreSlim），两种同步原语不互斥 |
| **影响范围（风险等级）** | **中等** — `_currentCount`（int）读写是原子的，不会撕裂；但 Monitor 与 SemaphoreSlim 不互斥，getter 可能读到过期值（CPU缓存一致性问题）。对日限计数器影响有限，但违反设计的线程安全契约 |
| **思考修复方案（权衡）** | 方案A：改用 `_lock.Wait()` / `_lock.Release()`（同步等待，读操作极快无性能影响）；方案B：将 `SemaphoreSlim` 改为 `object` + `lock` 但会破坏异步方法。选择方案A：最小改动，保持与异步方法互斥 |
| **修复方案解析（代码对比）** | **修复前**：`lock (_lock) { return _currentCount; }` — Monitor.Enter 与 SemaphoreSlim.WaitAsync 互不阻塞。**修复后**：`_lock.Wait(); try { return _currentCount; } finally { _lock.Release(); }` — 与 LoadAsync/IncrementAsync 共用同一个 SemaphoreSlim，正确互斥 |

### 2.2 偏差2+4：AnalyzeFileAsync 绕过速率限制器+熔断器恢复通道缺失

| 项目 | 内容 |
|:---|:---|
| **序号** | 2 |
| **审查路径** | `src/DirectoryCleanAgent.AI/AiAdvisorService.cs:110-181` → `AnalyzeFileAsync` 方法 |
| **如何发现问题** | 对比 `ProcessSingleFileAsync`（批量路径）的完整管道（EnsureNotOpen → TryAcquireAsync → 日限检查 → API调用 → IncrementAsync → Release），发现 `AnalyzeFileAsync`（单文件路径）直接调用 `CallAiApiWithPipelineAsync`，完全跳过了速率限制器和日限计数器。同时发现单文件路径未调用 `EnsureNotOpen()`，无法触发熔断器 Open→HalfOpen 转换 |
| **影响范围（风险等级）** | **高** — (1) 当批量分析正在消耗令牌时，单文件分析可绕过 30RPM 限制直接调用API；(2) 单文件分析的API调用不被计入日限；(3) 批量分析触发熔断后，单文件路径永无法独立恢复（需等待批量路径"救活"熔断器） |
| **思考修复方案（权衡）** | 方案A：让 `AnalyzeFileAsync` 调用 `ProcessSingleFileAsync`（需构造 FileDecisionCache 包装）— 增加内存分配；方案B：在 `AnalyzeFileAsync` 内联完整的限流+熔断+日限管道。选择方案B：管道步骤清晰可见，与 `ProcessSingleFileAsync` 注释风格一致，且单文件分析场景不需要缓存对象 |
| **修复方案解析（代码对比）** | **修复前**：`if (!IsAvailable) return null;` → `CallAiApiWithPipelineAsync(userPrompt, ct)` → 无速率限制、无日限递增。**修复后**：增加完整管道 — `EnsureNotOpen()` → `TryAcquireAsync()` → 日限检查 → API调用 → `IncrementAsync()` → `finally { Release() }`，并新增 `CircuitBreakerOpenException` 捕获分支 |

### 2.3 偏差3：日限达到时错误记录熔断器失败

| 项目 | 内容 |
|:---|:---|
| **序号** | 3 |
| **审查路径** | `src/DirectoryCleanAgent.AI/AiAdvisorService.cs:356-361` → `ProcessSingleFileAsync` 中日限检查分支 |
| **如何发现问题** | 审查熔断器 `RecordFailure()` 调用点时，发现日限达到（应用层配额限制）也被计入熔断器失败计数，这不符合熔断器"保护API不被连续失败打垮"的设计意图 |
| **影响范围（风险等级）** | **中等** — 极端场景：用户日限用完后连续尝试3次 → 熔断器错误打开（3次"失败"）→ 第二天日限重置，但熔断器仍需30秒才能恢复。用户体验：日限恢复后仍被拒绝服务，困惑不解 |
| **思考修复方案（权衡）** | 唯一方案：删除该行。日限是应用层状态，与API服务健康无关，不应触发熔断保护 |
| **修复方案解析（代码对比）** | **修复前**：`if (_usageTracker.IsLimitReached) { _circuitBreaker.RecordFailure(); return ...; }`。**修复后**：`if (_usageTracker.IsLimitReached) { return ...; }` — 仅记录警告日志并返回失败结果，不影响熔断器状态 |
