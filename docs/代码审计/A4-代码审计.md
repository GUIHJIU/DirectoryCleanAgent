| 1 | `JsonConfigService.cs:125` — `SaveAsync()` 在 SemaphoreSlim 锁内调用 `OnConfigChanged()` | 对照实现细节文档"决策 3 — ConfigChanged 事件在锁外触发"逐行审查 `SaveAsync` 和 `IncrementRuleCacheVersion` 两条路径时，发现 `IncrementRuleCacheVersion`（第 177 行）正确地在 `finally { _semaphore.Release(); }` 之后调用 `OnConfigChanged()`，但 `SaveAsync`（第 125 行）在 `try` 块内调用，此时锁尚未释放 | **死锁风险（中）** — 若 ConfigChanged 事件处理器回调 `IConfigService.SaveAsync()`，由于 `SemaphoreSlim` 不支持重入，调用方将永远等待，线程卡死。虽然正常使用时事件处理器不太可能再次 SaveAsync（可能引发无限递归），但防御性设计不应依赖调用方行为 | **方案 A（选用）**：引入 `saved` 标记，在 `try` 块中保存成功后设 `saved=true`，在 `finally` 释放锁后 `if (saved) OnConfigChanged()`。确保：① 成功路径在锁外触发事件；② 异常路径（取消/IO 错误）不误触发事件。方案 B：像 `IncrementRuleCacheVersion` 一样把整个 SaveAsync 改为 fire-and-forget 模式 — 否决，因为 SaveAsync 是用户直接调用的 API，需要同步返回结果。方案 C：使用 `lock` 替代 `SemaphoreSlim` — 否决，lock 不支持异步。 | **修改前**：`OnConfigChanged()` 在 `await SaveInternalAsync(ct)` 之后、`catch` 之前调用，锁仍未释放。**修改后**：引入 `var saved = false`；保存成功后 `saved = true`；`finally { Release(); }`；`if (saved) { OnConfigChanged(); }`。事件触发仅在成功路径执行，且锁已释放，与 `IncrementRuleCacheVersion:177` 行为完全一致。 |

| 3 | `JsonConfigService.cs:38-50` — 构造函数未调用 `EnsureDefaults()` | 修复问题 2 后发现 `StartupFlow_LoadConfig_Failure_ReturnsDefaults` 测试仍失败，追踪根因：测试通过 DI 容器获取 `IConfigService.Current`，从未调用 `LoadAsync()`。构造函数中 `_current = new UserConfig()`（字段初始化器），`DataStoragePath` 始终为 `""`。此问题在问题 2 修复后才暴露——因为 `EnsureDefaults()` 原本只能在 `LoadAsync` 内部被调用 | **默认配置不完整（中）** — DI 构造后、`LoadAsync()` 之前访问 `Current` 属性得到不完整默认值。影响面：启动流程中若任何组件在 `LoadAsync()` 完成前读取 `Current.DataStoragePath` 将得到空字符串。虽然设计文档假设 `LoadAsync()` 是启动第一批调用，但防御性设计应保证 `Current` 在任何时刻都返回可用默认值 | 在构造函数末尾添加 `EnsureDefaults()` 调用。构造函数中 `_configDirectory` 已赋值（第 41 行），`EnsureDefaults()` 依赖该字段且在其后执行，安全。注意 `EnsureDefaults()` 仅补充空值字段（幂等），不会覆盖已有值。构造函数中 `_current` 为全新 `UserConfig()`，所有可补字段为空。 | **修改前**：构造函数仅初始化路径和 `JsonSerializerOptions`，`_current` 依赖字段初始化器 `= new()`。**修改后**：构造函数末尾添加 `EnsureDefaults()` — 确保 DI 实例化后、任何调用之前，`Current` 属性已返回完整默认值（含 `IncludedVolumes` 系统盘 + `DataStoragePath` 默认路径）。 |
