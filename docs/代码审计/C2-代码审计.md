### 问题 #1 — DataGrid 列头排序事件未连接到 ViewModel

| 项目 | 内容 |
|:---|:---|
| **序号** | 1 |
| **审查路径** | `Controls/FileListView.xaml:167-184` → `ViewModels/FileListViewModel.cs:157-164, 825-978` |
| **级别** | ❌ 缺失 |
| **如何发现问题** | 对照设计文档 §4.7 "Everything 原生排序请求映射" 和补充说明 §4.7 排序列映射表。发现 DataGrid 设置了 `CanUserSortColumns="True"` 但未注册 `Sorting` 事件处理器。ViewModel 中完整实现了 `SortByColumnCommand`（含 Everything 下推逻辑 + 语义标签内存排序），但该命令从未被 XAML 调用。此外 `SortMemberPath` 值（"Name"/"Size"/"DateModified"）指向 `FileListItem` 不存在的属性名，WPF 默认排序静默失败。 |
| **影响范围（风险等级）** | **高** — 列头排序功能完全失效。用户点击列头时，WPF 因 SortMemberPath 无效而静默失败，无任何排序效果。分组视图和全量模式均受影响。 |
| **思考修复方案（权衡）** | **方案A**：在 DataGrid 注册 `Sorting` 事件，`e.Handled=true` 阻止 WPF 默认排序，将 `SortMemberPath` 作为逻辑键转发给 ViewModel。**方案B**：修正 SortMemberPath 为实际属性名，让 WPF 执行内存排序（但违反设计文档"排序下推至 Everything"的架构决策）。选择方案A——最小改动、保持架构一致性、不引入内存排序开销。 |
| **修复方案解析（代码对比）** | 详见下方代码对比 |

**FileListView.xaml**：
```diff
- <DataGrid ItemsSource="{Binding CurrentFileList}"
-           ... CanUserSortColumns="True"
+ <DataGrid ItemsSource="{Binding CurrentFileList}"
+           ... CanUserSortColumns="True"
+           Sorting="DataGrid_Sorting"
```

**FileListView.xaml.cs**：
```diff
+ /// <summary>
+ /// 拦截 DataGrid 列头排序事件，阻止 WPF 默认内存排序，
+ /// 将 SortMemberPath 作为逻辑键转发给 ViewModel 处理排序下推。
+ /// </summary>
+ private void DataGrid_Sorting(object sender, DataGridSortingEventArgs e)
+ {
+     e.Handled = true;  // 阻止 WPF 默认排序
+     var sortKey = e.Column.SortMemberPath;
+     if (string.IsNullOrEmpty(sortKey)) return;
+     if (DataContext is FileListViewModel vm)
+     {
+         vm.SortByColumnCommand.Execute(sortKey);
+     }
+ }
```

### 问题 #2 — 分组切换未取消前一次重建操作

| 项目 | 内容 |
|:---|:---|
| **序号** | 2 |
| **审查路径** | `ViewModels/FileListViewModel.cs:72-96` |
| **级别** | ⚠️ 偏差 |
| **如何发现问题** | 审查 `GroupByPrimaryIndex` / `GroupBySecondaryIndex` setter 实现：每次属性变更调用 `RebuildGroupTreeAsync(CancellationToken.None)`。ViewModel 中有 `_currentLoadCts` 字段用于取消操作，但分组切换未使用。用户快速切换分组时，前一次 `Task.Run` 中的 `BuildGroupTree` 仍在后台线程运行，可能与新结果产生竞态条件。 |
| **影响范围（风险等级）** | **中** — 用户快速切换分组时可能出现：旧分组结果覆盖新分组结果（竞态）；CPU 浪费于已过时的中间计算。 |
| **思考修复方案（权衡）** | 在 setter 中先 `Cancel()` + `Dispose()` 旧 CTS，再创建新 CTS 传入 `RebuildGroupTreeAsync`。需注意 `_currentLoadCts` 被三个入口共享（LoadDataAsync、GroupByPrimaryIndex、GroupBySecondaryIndex），Dispose 方法已正确处理释放。 |
| **修复方案解析（代码对比）** | 详见下方代码对比 |

```diff
  public int GroupByPrimaryIndex

      set
      {
          if (SetProperty(ref _groupByPrimaryIndex, value))

+             _currentLoadCts?.Dispose();

      }
  }
```

### 问题 #4 — ProcessPendingChangesAsync 仅更新不增删

| 项目 | 内容 |
|:---|:---|
| **序号** | 4 |
| **审查路径** | `ViewModels/FileListViewModel.cs:1141-1218` |
| **级别** | ⚠️ 偏差 |
| **如何发现问题** | 审查增量刷新逻辑：`OnFileChanged` 用 `_pendingFileChanges[e.FilePath] = DateTime.UtcNow` 记录变更，但完全忽略 `FileChangedEventArgs.ChangeType` 字段。`ProcessPendingChangesAsync` 仅在 `CurrentFileList` 中查找已存在项进行更新——文件被删除（`FileChangeType.Deleted`）时不会从列表中移除，新文件（`FileChangeType.Created`）不会添加。此外，`ConcurrentDictionary` 索引器在锁外使用与 `lock` 保护的快照操作之间存在不一致。 |
| **影响范围（风险等级）** | **中** — 增量刷新不完整：已删除文件仍显示在列表中（并可能被用户尝试操作），新创建文件不可见直到手动刷新。 |
| **思考修复方案（权衡）** | 1) 将 `_pendingFileChanges` 值类型从 `DateTime` 改为 `FileChangeType`；2) 使用 `AddOrUpdate` 替代索引器（Deleted 优先级最高）；3) `ProcessPendingChangesAsync` 按 ChangeType 分别处理——Deleted→移除行、Modified→更新行、Created→暂不自动添加（保守策略，因为缺少完整 FileDecisionCache 信息）。 |
| **修复方案解析（代码对比）** | 详见下方代码对比 |

**字段类型变更**：
```diff
- private readonly ConcurrentDictionary<string, DateTime> _pendingFileChanges = new();
+ private readonly ConcurrentDictionary<string, FileChangeType> _pendingFileChanges = new();
```

**OnFileChanged 合并策略**：
```diff
- _pendingFileChanges[e.FilePath] = DateTime.UtcNow;
+ // 合并策略：Deleted 优先级最高
+ _pendingFileChanges.AddOrUpdate(e.FilePath, e.ChangeType,
+     (_, existing) => existing == FileChangeType.Deleted
+         ? FileChangeType.Deleted
+         : e.ChangeType);
```

**ProcessPendingChangesAsync 差异化处理**：
```diff

+ }
+
  // 处理文件修改/创建/重命名：在列表中查找并更新
  var existingItem = CurrentFileList.FirstOrDefault(...);
  if (existingItem != null) { ... }
+ // 注：Created 场景下文件不在 CurrentFileList 中，暂不自动添加。
+ // 新增文件需通过重新扫描发现（由用户手动刷新或规则热加载触发）。
```
