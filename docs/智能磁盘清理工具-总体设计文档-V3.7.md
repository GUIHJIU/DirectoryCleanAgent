## 1. 引言

### 1.1 项目背景

现有磁盘清理工具普遍存在以下问题：
- 需要用户手动选择目录，不懂技术的人无从下手；
- 评分机制不透明，用户不敢轻易删除；
- 无法自动保护开发环境（如 Docker、WSL）；
- AI 引入缺乏可解释性，信任度低；
- 操作繁琐，需要逐个勾选文件。

本项目旨在设计一款**智能、安全、可解释、可自动化**的磁盘清理工具，特别针对开发者和高级用户，同时兼顾普通用户。

### 1.2 设计目标

- **智能**：自动定位垃圾热点，无需用户选目录。
- **透明**：每个文件的删除建议附带**语义标签**（简单模式）或规则理由（专家模式）。
- **安全**：多级保护机制（硬规则、白名单、回收站、隔离区、回滚）。
- **高效**：
  - **就绪速度**：工具启动后即显示分析结果（依赖Everything索引，流式处理，200万文件内存峰值 <200MB）。
  - **动态刷新**：文件变更通过Everything事件在数秒内反映到界面。
  - **排序性能**：所有排序/分组委托给 Everything 原生能力，不对未缓存的原始文件集进行内存排序。
- **自动化**：本版本（V3.7）仅提供手动与半自动（一键清理）操作，**全自动定时清理延后至后续里程碑**。
- **可扩展**：规则引擎配置化，AI服务作为辅助顾问，支持命令行。
- **健壮性**：严格保护系统资源，所有长耗时操作可取消、有进度反馈；低权限环境自动降级为只读；避免高频系统调用和临时文件残留。

### 1.3 目标用户

- 普通家庭用户（简单模式，语义标签引导）
- 开发者（专家模式，保护 Docker/WSL，可调规则）
- 高级用户/极客（命令行、自定义规则）
- 企业管理员（只读审计、批量配置）

### 1.4 最低系统要求

- 操作系统：Windows 10 版本 1809 或更高（Windows 11 推荐）
- **强制依赖**：**[Everything](https://www.voidtools.com/) ≥ 1.4.1.1000**（该版本起稳定支持 `VolumeGuid + FileReferenceNumber`，以下简称 FRN），且必须保持服务运行并启用 IPC。工具启动时将执行以下检测序列：
  1. **进程存活检查**：检查 Everything 进程与服务是否运行。
  2. **版本检查**：调用 `Everything_GetVersion()`，获取主版本号与构建号。若 < 1.4.1.1000，弹窗提示"需要 Everything 1.4.1.1000 或更高版本"，提供下载链接并阻断进入主界面。
  3. **IPC ping**：发送 IPC 探测指令验证通信正常。
  4. **索引就绪检测**：查询 `is_indexing` 状态。
  5. **FRN 能力检测**：尝试查询一个已知系统文件（如 `C:\Windows\explorer.exe`）的 FRN 字段。若返回空或错误，标记 `FRN_AVAILABLE = false`，后续墓碑匹配降级为 Size+LastWriteTime 指纹模式，并强制墓碑 3 天过期（详见 5.1 节）。
  以上任一检测（除 FRN 能力外）不通过，则禁止进入主界面并引导用户排查。
- **管理员权限**：工具在启动时检测当前令牌。若未以管理员身份运行，工具自动进入 **"只读模式"**：所有分析、仪表板、查看功能正常，但**删除、移动、清理等修改操作按钮全部禁用**，并显示提示"请以管理员身份重新运行以启用清理功能"。若需要执行清理，用户必须右键"以管理员身份运行"重新启动。
- 磁盘格式：**NTFS（强制）**。工具完全依赖 Everything 的 NTFS 索引，非 NTFS 卷（如 FAT、ReFS）内的文件将不可见且无法分析。若系统盘非 NTFS，工具将提示"不支持的文件系统"并退出。
- 支持 BitLocker 加密卷（Everything 正常读取，无需特殊处理）

> **设计决策**：完全取消传统递归遍历回退方案。Everything 不可用则视为工具不可用，避免边界场景的不可靠体验。

## 3. 用户体验与交互设计

### 3.1 首次启动配置向导

首次运行工具时，弹出配置向导，引导用户完成基本设置。用户可跳过使用默认配置，后续在设置中修改。

- **步骤0（前置检查）**：在向导显示前，执行 Everything 依赖检测：

  1. 检查 Everything 进程与服务；失败弹窗引导并阻断。
  2. 调用 `Everything_GetVersion()` 检查版本号。若主版本 < 1 或 (主版本 = 1 且构建号 < 1000)，弹窗提示"需要 Everything 1.4.1.1000 或更高版本，[下载链接]"，提供"重试"和"退出"按钮。版本符合后继续。
  3. IPC ping 探测；失败提示检查防火墙/端口并阻断。
  4. 查询 Everything 索引状态（`is_indexing`）：
     - 若正在索引，显示"Everything 正在构建索引，当前结果可能不完整，请稍候..."，同时提供三个选项：
       - **"等待完成"**：每 **2 秒**轮询一次 `is_indexing`，界面显示已等待时长（如"已等待 5 分 30 秒"）。不显示百分比（Everything 仅返回布尔值）。轮询直到索引完成或用户取消。若轮询超过 **120 秒**无响应，自动进入警告模式。
       - **"后台等待"**：工具最小化到系统托盘，索引完成后通过 Windows 通知提醒用户，随后自动进入主界面。**用户可通过双击系统托盘图标随时手动恢复主窗口，或点击Windows通知中心的“查看结果”按钮唤回界面。托盘图标同时提供右键菜单（显示主界面 / 退出），确保用户在任何时候都能主动控制应用状态。**
       - **"跳过"**：直接进入主界面，但界面顶部显示醒目的黄色横幅"索引构建中，结果不完整，建议稍后刷新"，并禁用"一键清理"按钮，直到索引完成。
     - 120 秒无响应自动进入警告模式。

- **步骤1：欢迎与模式选择**
  - 介绍工具价值。
  - 选择使用模式：🌿 简单模式（推荐） / 🔧 专家模式。
  - 简单模式后续步骤简化；专家模式展示更多选项。

- **步骤2：用户画像（仅专家模式显示）**
  - 职业：普通用户 / 软件工程师 / 设计师 / 运维 / 学生 / 其他
  - 设备用途：办公 / 开发 / 娱乐 / 图形设计 / 服务器

- **步骤3：保护与排除（可选）**
  - 自动检测并推荐保护目录（Docker、WSL、虚拟机等），用户可确认或修改。
  - 手动添加排除目录（支持通配符）。

- **步骤4：AI配置（可选）**
  - 是否启用AI顾问（默认否）。
  - 若启用：填写API地址、Key、模型、每日限额。
  - 测试连接。

向导完成后，进入主界面并自动执行首次智能扫描。

### 3.2 主界面布局与交互

主界面分为五个区域：顶部栏、仪表板、操作栏、主内容区（左侧分组树 + 右侧文件列表）、底部状态栏。

#### 顶部栏
- 左侧：应用图标 + 标题"智能磁盘清理"
- 右侧：设置⚙️、帮助❓、用户菜单（模式切换、退出）

#### 仪表板
显示清理概览卡片（实时更新）：

```
┌─────────────────────────────────────────────────────────────┐
│ 🟢 安全可自动清理: 18.2 GB (113项)                           │
│ 💡 建议清理: 12.5 GB (89项)                                  │
│ ⚠️ 需人工确认: 3.1 GB (6项)                                 │
│ 🛡️ 已保护: 50.1 GB (Docker/WSL等)                          │

```

点击任意卡片可快速筛选对应类别的文件。状态栏上方可能出现黄色警告横幅（索引未完成、权限受限等）。

#### 操作栏
按钮从左到右：
- **一键清理安全项**：删除所有 `auto_delete` 文件。
- **智能选择建议项**：自动勾选所有 `suggest_delete` 文件，用户确认后删除。
- **模拟运行**：预览删除效果，不实际执行。
- **导出报告**：导出当前分析结果为 HTML/CSV。
- **刷新**：重新执行增量扫描。
- **分组依据**下拉菜单：路径 / 时间 / 大小 / 类型 / 语义标签。

#### 主内容区（左侧分组树 + 右侧文件列表）
- **左侧分组树**：根据选定的分组依据动态生成树结构。节点显示：图标 + 名称 + (项数, 总大小)，并带复选框。
- **右侧文件列表**：显示当前选中分组下的文件。列：☑ 复选框 | 图标+语义标签 | 文件名 | 大小 | 最后访问时间 | AI状态 | 操作（详情、排除）。支持列排序和右键菜单（详情、排除、请求AI分析）。
- 排序/分组时，主内容区数据源切换为 **Everything 实时结果 + 本地墓碑过滤（基于物理ID）**，界面调用 Everything 查询时传递原生排序参数，不再在内存中排序原始结果。

#### 底部状态栏
- 显示：本次释放空间（实时更新） | 历史累计清理 | 最后扫描时间 | 隔离区使用情况（链接）。当后台有批量删除或校验操作时，状态栏显示进度条和"取消"按钮。

### 3.4 REVIEW区域分组筛选

**分组与排序实现原则**：用户点击"按大小/时间/路径/语义标签"排序或分组时，直接向 Everything 适配器请求携带原生排序参数的查询。仅当用户选择"按语义标签分组"或"仅查看命中规则的文件"时，才基于 `FileDecisionCache` 中的子集进行内存分组/排序。

**分组数据范围（重要设计约束）**：分组功能**仅作用于"已命中规则或用户标记"的可操作文件**。未命中规则的文件（`final_action = KEEP`）不进入任何分组，也不会出现在默认的分组视图中。此设计基于两个原因：
1. 未命中文件因其 `semantic_category` 为空，无法进行有意义的语义分组。
2. 若将所有全盘文件纳入分组视图，将违背"排序/分组不下拉至内存"的性能承诺。

UI 在主内容区底部显示一行灰色说明文字：*"当前视图仅包含可操作文件（建议清理 / 需确认 / 自动清理）。点击'显示所有文件'切换为全量浏览模式。"*

- **"显示所有文件"切换开关**：位于操作栏右侧。开启后，关闭分组模式，改为平面列表展示所有扫描到的文件（含未命中规则的），但**不对全量数据进行内存排序**——排序请求仍下推至 Everything。**在该平面列表模式下，用户仍可点击列头（名称、大小、修改日期等）触发 Everything 原生排序（支持升序/降序切换），但此模式下不支持树状多级分组（因分组逻辑依赖缓存中的语义标签）。**
- **多级分组**：先选一个主维度，再选子维度（如先按路径，再按时间）。

分组依据支持：
- **路径**：按顶层目录聚合（Windows, Program Files, Users等）
- **时间**：7天内 / 30天内 / 30-90天 / 90天以上
- **大小**：>1GB / 100MB-1GB / <100MB
- **类型**：扩展名分类（临时文件、安装包、压缩包等）
- **语义标签**：与简单模式标签一致

### 3.5 AI触发机制与展示

#### 触发方式与限制
- **手动触发**（默认）：无次数限制，但单次最多选择 **200** 个文件。
- **自动触发**（专家模式可配置）：扫描完成后，筛选出 `semantic_category = "未分类"` 且大小 > 100MB 的文件。**单次自动触发的总文件数上限为 500**，按文件大小降序取前 500 个发起批量请求。
- **速率限制**：所有 AI 请求全局限制每秒最大请求数（RPM）为 30，单个批次内并发不超过 5。
- **从不自动触发**：设置中关闭"自动为未分类文件请求AI"。

#### 展示
- 未分析文件：显示"❓ 点击分析"图标。分析中：显示"⏳ 分析中"。分析完成：显示AI标签图标（✅安全 / ❓未知 / ⚠️风险）。
- AI 响应永不改变已提交的删除操作：当用户点击"执行删除"时，系统冻结当前所有目标文件的 `final_action` 快照，AI 后续返回仅更新界面标签，绝不对已执行的删除进行回溯修正。

### 4.2 关键模块职责

| 模块 | 职责 |
|------|------|
| **Everything 流式适配器** | 通过 Everything IPC 获取文件列表时采用 **`yield return` 流式枚举**，不一次性全量加载到内存。**所有排序/分组请求转译为 Everything 原生查询参数**。逐个文件产出后立即转入规则引擎处理，未命中规则的文件直接丢弃。路径统一转为 `\\?\` 格式。维护本地墓碑表。启动时完成 IPC ping、版本检测、FRN 能力探测及索引就绪检测。 |
| **规则引擎** | 硬规则（只读） + 启发式规则（可配置），输出 `rule_verdict` 和 `semantic_category`。支持 **热加载**：使用 `FileSystemWatcher` 监控启发式规则 JSON 文件所在目录，检测到文件变更（含外部编辑器如 VS Code 直接修改）后，自动递增 `UserConfig.RuleCacheVersion`，并在 500ms 防抖后异步通知决策引擎——决策引擎收到通知后清空缓存并触发全量重扫描，不阻塞 UI。若用户通过内置编辑器修改规则，保存时同步递增版本号。两种路径最终效果一致：**任何规则文件写入都触发版本号递增和缓存清空**。 |
| **AI 顾问** | 异步调用大模型，旁路填充 `ai_label`。内置**单次触发上限 500 文件**，**全局速率限制 30 RPM，最大并发 5**。首次决策完全依赖规则。 |
| **决策引擎** | 结合规则裁决、AI 标签（若已有）和用户策略输出 `final_action`。执行删除时，**即时深拷贝所有待删除文件信息，生成不可变的内存快照 `ReadOnlyCollection`**，绝不依赖外部可变缓存。启动时检查 `FileDecisionCache` 的版本号，若与当前规则版本不一致则清空缓存并重新裁决。 |
| **操作执行器** | 执行删除，采用**乐观删除**策略；所有文件操作使用 `\\?\` 前缀；删除大文件前调用 `SHQueryRecycleBin` 仅一次（在所有删除路径之前）；所有批量操作绑定 `CancellationToken`；跨卷复制使用 `.cleaning.tmp` 临时扩展名。 |
| **回滚管理器** | 基于 SHA-256 从回收站或隔离区恢复；恢复成功后根据 `operation_id` 精确清除对应墓碑记录。 |
| **自动化调度器** | **当前版本不实现**，仅保留接口与预留数据结构。 |
| **排除管理器** | 管理用户白名单，优先级最高。 |
| **审计日志模块** | 记录所有配置变更、扫描、删除等行为。**强制实施“不可修改”策略：数据访问层（DAL）仅向审计日志表提供 `Insert` 方法，不提供任何 `Update` 或 `Delete` 的公开 API 或仓储方法；应用程序使用的数据库连接账号仅拥有写入和查询权限，从代码根源杜绝篡改可能。** |

## 5. 数据设计

### 5.1 核心数据实体

**FileDecisionCache（文件决策缓存）**
*仅存储被规则引擎命中或用户标记的文件，不再存储全盘文件元数据。*

- `file_path` (主键，字符串)
- `size_bytes` (整数)
- `last_write_time` (时间戳)
- `rule_verdict` (枚举)
- `semantic_category` (字符串)
- `ai_label` (可选)
- `ai_confidence` (浮点数)
- `ai_explanation` (文本)
- `final_action` (枚举)
- `user_decision` (可选)
- `cache_version` (整数)，记录生成该缓存时的规则版本号。

**LocalTombstone（本地墓碑）**
*基于文件物理标识进行精确匹配，解决路径复用误判。*

| 字段 | 类型 | 说明 |
|------|------|------|
| `file_path` | TEXT | 原路径 |
| `file_identity_key` | TEXT (主键) | **Everything 提供的 `VolumeGuid + FileReferenceNumber`**，即文件在 NTFS 上的唯一物理 ID。若 FRN 不可用（启动时 `FRN_AVAILABLE = false`），则降级为 `Size + LastWriteTime` 组合并**强制 3 天过期**（降低同一路径下同大小同时间文件的碰撞窗口）。降级模式下，每次匹配时额外校验 `file_path` 和 `original_size` 一致性，进一步降低误匹配概率。**注：降级指纹模式因强制3天过期且叠加额外校验，匹配率预期 > 99.9%，剩余碰撞概率对业务影响可忽略。** |
| `operation_id` | TEXT (UUID) | 关联的删除操作 ID，用于回滚时精确清除 |
| `original_size` | INTEGER | 删除时文件大小（字节） |
| `original_last_write_time` | DATETIME | 删除时文件最后写入时间（用于降级指纹匹配） |
| `deleted_at` | DATETIME | 删除时间戳 |

**DeletionRecord（删除记录）**
*记录每次删除操作的完整信息。*

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | INTEGER (主键) | 自增 ID |
| `operation_id` | TEXT (UUID) | 操作批次唯一标识 |
| `file_path` | TEXT | 原始文件路径 |
| `file_hash` | TEXT | 完整 SHA-256 哈希 |
| `file_size` | INTEGER | 文件大小（字节） |
| `deletion_method` | TEXT | 删除方式：`RECYCLE_BIN` / `QUARANTINE` / `PERMANENT` |
| `decision_snapshot` | TEXT (JSON) | **删除时冻结的决策快照**。使用 `System.Text.Json` 序列化为**紧凑 JSON**（不缩进），若 JSON 长度超过 4KB 则额外应用 GZip 压缩后 Base64 编码存储。快照结构定义如下：<br>`{`<br>`  "op": "<operation_id>",`<br>`  "files": [`<br>`    {`<br>`      "path": "<file_path>",`<br>`      "size": <file_size>,`<br>`      "hash": "<sha256_hex>",`<br>`      "action": "<final_action>",`<br>`      "verdict": "<rule_verdict>"`<br>`    }, ...`<br>`  ],`<br>`  "frozen_at": "<ISO8601_UTC>"`<br>`}`<br>快照仅存储于此字段用于审计和回滚溯源，**不另存为独立文件**。执行删除操作时，代码层面使用不可变 `ReadOnlyCollection<DeleteSnapshotEntry>` 遍历，数据结构为：`{ file_path, file_size, sha256_hash, final_action, operation_id }`。 |
| `created_at` | DATETIME | 记录创建时间 |

**AuditLog**
- `timestamp`, `user_sid`, `operation_type`, `target`, `result`, `details` (JSON)
- **不可修改性实现**：详见 4.2 节审计日志模块职责描述——数据层仅暴露追加方法，无任何更新/删除入口。

**UserConfig**
- 增加字段 `RuleCacheVersion` (整数)，由规则引擎在规则文件变更时自动递增（无论是内置编辑器保存还是 `FileSystemWatcher` 检测到外部修改，500ms 防抖后递增）。
- 增加 `IncludedVolumes` (字符串数组)，存储用户选择的盘符列表，默认仅含系统盘。
- 增加 `FRN_AVAILABLE` (布尔值)，记录启动时检测到的 FRN 能力状态，用于墓碑匹配策略选择。

### 5.2 存储方案

- **SQLite**：存储决策缓存、墓碑表、删除记录、审计日志。启用 **WAL 模式**，**PRAGMA synchronous=NORMAL**。
- **批量写入机制**：所有对 SQLite 的写操作不直接执行，而是先进入内存 `ConcurrentQueue`。后台任务每 **500ms** 或队列累积达到 **200 条** 时，执行一次批量 `UPSERT` 事务，彻底消除写锁竞争与刷写风暴。
- **缓存版本控制**：读取 `FileDecisionCache` 时，检查 `cache_version` 与当前 `UserConfig.RuleCacheVersion` 是否一致，不一致则清空缓存并重新裁决。`RuleCacheVersion` 的更新由 `FileSystemWatcher`（500ms 防抖）或内置编辑器保存操作自动触发，触发后**异步**清空缓存并触发重新扫描，不阻塞 UI。
- **JSON 文件**：用户配置（含规则版本号）、启发式规则库。
- **隔离区**：独立文件夹，默认路径 `%LocalAppData%\SpaceClear\Quarantine`。

## 6. 决策引擎（核心逻辑）

### 6.1 输入与输出

**输入**：`rule_verdict`, `ai_label` 和 `confidence`（如果启用），用户策略（`mode`, `ai_trust_level`, `ai_enabled`）。

**输出**：`final_action` ∈ {`auto_delete`, `suggest_delete`, `manual_review`, `protected`}，`decision_reason`。

### 6.2 规则引擎输出定义

| 优先级 | 规则类型 | 示例 | 输出 verdict |
|--------|----------|------|---------------|
| 1 | 硬禁止规则 | `C:\Windows\System32\*` | `FORBID` |
| 2 | 硬自动删除规则 | `C:\Windows\Temp\*`（>7天） | `AUTO_DELETE` |
| 3 | 用户排除规则 | 用户添加的白名单 | `EXCLUDE` → `FORBID` |
| 4 | 保护规则 | Docker/WSL路径 | `PROTECT` → `FORBID` |
| 5 | 启发式建议规则 | `.log` >100MB 且 >90天 | `SUGGEST_DELETE` |
| 6 | 启发式保留规则 | `.docx` 位于文档目录 | `SUGGEST_KEEP` |
| 7 | 默认规则 | 无匹配 | `KEEP` |

### 6.3 决策表（完整）

**AI 标签仅在已有缓存时参与决策；首次扫描无 AI 标签时，规则为 SUGGEST_DELETE 则直接输出 suggest_delete，KEEP 则 protected。**

#### 情况1：决定性规则（不依赖AI）
| rule_verdict | final_action |
|--------------|--------------|
| `FORBID` | `protected` |
| `AUTO_DELETE` | `auto_delete` |
| `SUGGEST_KEEP` | `protected` |
| `EXCLUDE` / `PROTECT` | `protected` |

#### 情况2：启发式建议删除（`SUGGEST_DELETE`）且AI启用
| AI_label | AI_confidence | 用户AI信任度 | final_action |
|----------|---------------|--------------|---------------|
| `safe` | 任意 | 任意 | `suggest_delete` |
| `unknown` | <0.8 | 任意 | `suggest_delete` |
| `unknown` | ≥0.8 | low/medium | `suggest_delete` |
| `unknown` | ≥0.8 | high | `manual_review` |
| `risky` | <0.6 | 任意 | `suggest_delete` |
| `risky` | ≥0.6 | low | `suggest_delete` |
| `risky` | ≥0.6 | medium/high | `manual_review` |

#### 情况3：规则无建议（`KEEP`）且AI启用
| AI_label | AI_confidence | 用户AI信任度 | final_action |
|----------|---------------|--------------|---------------|
| `safe` | ≥0.7 | high | `suggest_delete` |
| `safe` | ≥0.7 | low/medium | `manual_review` |
| `safe` | <0.7 | 任意 | `manual_review` |
| `unknown` | 任意 | 任意 | `manual_review` |
| `risky` | 任意 | 任意 | `manual_review`（带警告） |

#### 情况4：AI未启用或调用失败
- `SUGGEST_DELETE` → `suggest_delete`
- `KEEP` → `protected`（不显示）

### 6.4 快照机制

在执行删除操作时，系统**冻结**当前所有待删除文件的 `final_action`，生成一个不可变的 `ReadOnlyCollection<DeleteSnapshotEntry>` 内存快照（结构见 5.1 节 `DeletionRecord.decision_snapshot`）。即使规则变更导致缓存清空，已开始的删除操作不受任何影响，AI 返回或规则更新不对已冻结的决策产生影响。

快照生成后，同时序列化为 JSON 存入 `DeletionRecord.decision_snapshot` 字段（超 4KB 时压缩），用于审计溯源。快照不另存为独立文件。

### 6.5 规则热加载与缓存失效

**触发机制**：
1. **内置编辑器保存**：用户通过工具内置规则编辑器修改并保存时，保存操作完成后立即递增 `UserConfig.RuleCacheVersion`。
2. **外部修改**：`FileSystemWatcher` 监控启发式规则 JSON 文件所在目录（默认 `%LocalAppData%\SpaceClear\rules\`），检测到 `Changed` 或 `Created` 事件后，进行 **500ms 防抖**（合并连续写入事件），然后递增版本号并通知决策引擎。

**响应行为**：
- 决策引擎收到版本变更通知后，**异步**清空 `FileDecisionCache` 表，并触发全量重新扫描。
- 该过程不阻塞 UI 线程，用户可在重新扫描期间继续浏览旧结果，扫描完成后自动刷新界面。
- 启动时，决策引擎对比 `FileDecisionCache.cache_version` 与 `UserConfig.RuleCacheVersion`。若不一致，清空缓存并重新裁决。

### 7.1 扫描与分析流程（流式，可取消，基于Everything）

```mermaid

```

- 所有路径统一添加 `\\?\` 前缀。
- 墓碑过滤基于物理 ID（FRN），若启动时 `FRN_AVAILABLE = false`，则降级为 `大小+最后写入时间` 组合，并强制墓碑 3 天过期；降级模式下每次匹配额外校验路径和大小。
- UI 排序/分组请求会转换为 Everything 查询参数，不在内存中排序全量文件列表。

### 7.2 一键安全清理流程（优化时序）

1. 用户点击"一键清理安全项"或类似批量操作。

2. **汇总待删文件元数据**：系统根据当前筛选/勾选状态，汇总待删除文件的**基础信息列表**（路径 + 大小，此阶段无需计算哈希）。

3. **回收站容量预检（必须最先执行）**：在生成任何快照或计算哈希**之前**，调用**一次** `SHQueryRecycleBin` 获取回收站当前可用空间。计算待删文件总大小（仅限拟通过回收站删除的文件）。若总大小 > 可用空间，弹出警告，强制用户选择：
   - 缩小选择范围（仅选中部分文件）；
   - 对超出部分改用永久删除（需勾选确认"我明白永久删除不可恢复"）。
   - **设计意图**：此顺序确保系统不会在容量不足时白白耗费 CPU/IO 去计算大量文件的 SHA-256 哈希，遵循“快速失败”（Fail-Fast）原则，提升用户体验。
   - **注意**：此检查必须在所有删除路径（含隔离区流程）之前完成。只有回收站容量检查通过后，才进入后续步骤。

4. **深拷贝快照**：容量预检通过且用户最终确认删除集合后，系统立即生成待删文件的不可变只读集合快照 `ReadOnlyCollection<DeleteSnapshotEntry>`（每条包含 `{ file_path, file_size, sha256_hash, final_action, operation_id }`，此时执行 SHA-256 哈希计算），后续所有操作只遍历此快照，不依赖外部可变缓存。同时将快照序列化为 JSON 写入 `DeletionRecord.decision_snapshot`。

5. 弹出确认对话框，汇总待删除文件信息（按语义标签分组，显示总大小、文件数），并包含"取消"按钮。

6. 用户确认后，创建 `CancellationTokenSource`，将 `CancellationToken` 传递给操作执行器。UI 层实时更新进度条，并提供"取消"按钮。

7. 操作执行器遍历快照，每次检查 `IsCancellationRequested`。对于因独占锁而失败的文件，降级到 `manual_review`。

8. 删除成功后，向墓碑表插入记录（含 `operation_id`），并通过批量队列写入。记录审计日志，刷新 UI。

### 7.3 乐观删除与文件锁处理

- 所有删除/移动操作统一使用 `\\?\` 前缀路径。
- 执行删除时直接调用 `IFileOperation`。
- 若返回错误 `HRESULT` 表明文件被独占锁定（如 `0x80070020`），捕获后将该文件移入 `manual_review` 列表，标注"文件被占用，无法删除"。
- 取消任何事前文件锁预检。

### 7.4 隔离区与空间安全

**前置条件**：回收站容量检查（7.2 节步骤 3）必须已通过。以下流程假设回收站有足够空间接收最终删除的文件。

**隔离区写入规则：**

1. 对于待删除文件，若 **源与隔离区位于同一卷**：
   - 若文件大小 ≤ 隔离区剩余容量，执行 **移动** 操作。
   - 若文件大小 > 隔离区剩余容量，但 ≤ 磁盘总剩余空间，执行 **跨卷降级流程**：复制到隔离区 → SHA-256 校验 → 校验通过后删除原文件。

2. 若源与隔离区**不同卷**：**一律采用"复制+校验+删原文件"**。

3. **超大文件（>隔离区所在卷剩余空间 20%）**：**不经过隔离区**，直接删除到回收站（回收站容量已在前置检查中验证通过）。

**跨卷复制与临时文件机制**：
- 复制时目标文件命名为 `<原文件名>.cleaning.tmp`。
- 复制过程中若用户取消或发生错误，**立即删除 `.cleaning.tmp` 临时文件**，原文件不受影响。
- 复制完成后，在后台线程计算 SHA-256，通过 `IProgress` 更新进度，支持取消。校验过程中若取消或失败，删除临时文件，保留原文件。
- 校验通过后，将临时文件**重命名**为正式隔离文件（含哈希信息的规范文件名），然后删除原文件（放入回收站）。

### 7.5 模拟运行

与真实删除相同的扫描、规则匹配、AI调用（如启用），但跳过操作执行器。所有路径处理统一为 `\\?\`，并模拟回收站容量检查。展示将删除的文件列表、释放空间总量，可导出模拟报告。

### 7.6 回滚流程

- 用户进入历史记录页面，选择删除批次。
- 回滚管理器根据 `DeletionRecord` 中的原路径和哈希，从回收站或隔离区恢复文件。
- 恢复成功后，**通过该操作的 `operation_id` 批量删除 `LocalTombstone` 中所有关联记录**（`DELETE FROM Tombstones WHERE operation_id = @opId`）。
- 若恢复时哈希不匹配，警告用户并跳过。
- **定期清理机制**：除自动过期（30天，FRN 降级模式下为 3 天）和回滚清理外，启动时额外执行一次扫描：如果墓碑记录的 `file_path` 当前存在文件，且文件物理 ID 与墓碑 `file_identity_key` 不匹配，则自动删除该墓碑，保证路径复用场景的实时性。

## 8. 安全与信任设计

| 机制 | 描述 |
|------|------|
| 硬规则禁止删除 | 内置只读，启动时校验哈希。 |
| 用户白名单优先 | 任何阶段不被删除。 |
| **乐观删除 + 锁异常降级** | 删除失败自动移入人工确认，杜绝误删和过度打扰。 |
| 默认回收站 | 所有删除默认放入回收站。**删除前仅调用一次 `SHQueryRecycleBin` 检查容量，不足时强制用户确认永久删除**。**此检查在所有删除路径（含隔离区决策）之前执行**（见 7.2 节优化时序）。 |
| 隔离区空间管理 | 大文件智能绕过，跨卷复制校验，异步进度可取消。所有路径操作使用 `\\?\`。 |
| 完整 SHA-256 哈希 | 删除时记录，回滚时校验。 |
| 操作日志与审计日志 | 所有删除和配置变更均有记录。**审计日志“不可修改”通过代码层强制实现：数据访问层仅提供 `Insert` 方法，不公开任何 `Update`/`Delete` API；应用程序数据库连接仅拥有写入和查询权限，从根源杜绝篡改。** |
| 二次确认 | 任何删除操作弹窗确认，按语义标签汇总。 |
| **非管理员只读模式** | 非管理员启动时，所有分析功能可用，但删除、清理、隔离区修改等按钮全部禁用，界面明确提示需管理员权限。 |
| 自动化默认不实现 | 当前版本无全自动清理，消除权限令牌和误删风险。 |
| 模拟运行 | 所有删除计划可预览，含回收站空间预估。 |
| 决策快照冻结 | 执行删除时锁定决策为不可变 `ReadOnlyCollection`，序列化 JSON（超 4KB 压缩）存入审计记录，杜绝 AI 回调干扰。 |
| **本地墓碑（物理 ID + operation_id）** | 基于 FRN 精确标记；启动时检测 FRN 能力，不可用时降级为指纹模式（3 天强制过期 + 额外路径/大小校验，匹配率 >99.9%）；回滚时按操作 ID 批量清除；定期清理失效墓碑。 |

## 10. 技术选型与性能考量

| 领域 | 建议 |
|------|------|
| 语言 | C# + .NET 8 (64位优先) |
| UI | WinUI 3 或 WPF |
| 文件索引 | Everything SDK (IPC)，**版本 ≥ 1.4.1.1000**，**流式 yield return 枚举，排序/分组下推至 Everything 原生** |
| 路径处理 | **所有内部路径统一加 `\\?\` 前缀，Everything 适配器输出时标准化** |
| 数据库 | SQLite（WAL模式，同步=NORMAL，**内存队列批量 Upsert，500ms/200条刷新**） |
| 回收站 | `IFileOperation`；**仅在操作开始前调用一次 `SHQueryRecycleBin`**，且在所有删除路径之前执行（详见 7.2 节） |
| 隔离区 | 文件夹 + SQLite 索引；跨卷复制使用 `.cleaning.tmp` + 重命名；异步 SHA-256 校验，通过 `IProgress` 更新进度 |
| AI调用 | `HttpClient` + 熔断，**最大批次 500 文件，速率 30 RPM，并发 5** |
| 取消与进度 | 所有长耗时操作（删除、校验）必须接受 `CancellationToken` 并报告进度 |
| 删除快照 | **深拷贝待删列表为 `ReadOnlyCollection<DeleteSnapshotEntry>`**，字段：`{ file_path, file_size, sha256_hash, final_action, operation_id }`；序列化为紧凑 JSON 写入审计记录（超 4KB 时 GZip 压缩 + Base64） |
| 规则热加载 | `FileSystemWatcher` 监控规则目录，500ms 防抖后递增 `RuleCacheVersion`，异步通知重扫 |
| Everything 启动检测 | 进程存活 → 版本号（≥1.4.1.1000）→ IPC ping → FRN 能力探测 → 索引就绪 |
| 性能指标 | 启动可交互 <3s；流式处理内存峰值 <200MB（200万文件）；排序无内存压力；规则决策 5s；批量写入不阻塞 UI |

## 12. 总结

本设计文档 V3.7 是在 V3.6 基础上，针对终审阶段提出的 6 项交互与逻辑模糊点进行深度整合的最终版本。关键改进包括：

- **清理流程时序优化**：将回收站容量预检移至快照生成之前，避免无效哈希计算，大幅提升用户体验（见 7.2 节）。
- **交互闭环完善**：明确后台等待模式下托盘图标双击唤回及右键菜单行为（见 3.1 节）。
- **功能边界清晰化**：定义“显示所有文件”平面列表模式下的列头排序能力（见 3.4 节）。
- **数据降级量化说明**：补充 FRN 不可用时指纹匹配率预期（>99.9%），消除过度设计顾虑（见 5.1 节）。
- **安全机制落地**：明确审计日志“不可修改”的代码层实现约束（仅 Insert，无 Update/Delete API）（见 4.2 节、第 8 章）。

V3.7 已消除全部已知设计与开发歧义，可作为编码实现的最终基线。任何后续调整须经变更评审并同步更新本文档。
